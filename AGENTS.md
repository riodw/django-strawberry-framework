## Project
django-strawberry-framework — DRF-shaped Django integration for Strawberry GraphQL. Pre-alpha. Goal: provide FilterSet, OrderSet, AggregateSet, DjangoType, DjangoConnectionField, apply_cascade_permissions, and per-field permission hooks — all configured via Meta classes the way DRF, django-filter, and Django itself do it. The differentiator vs strawberry-graphql-django, which uses stacked decorators. See docs/README.md for the full positioning argument
## DRF first, strawberry second
Every public surface uses Meta classes. If you find yourself writing strawberry decorators on consumer-facing classes, stop — that is strawberry-graphql-django's API and the explicit reason this package exists. Strawberry is the engine; DRF is the shape. Cribbing from django-graphene-filters and DRF idioms is encouraged; cribbing from strawberry-graphql-django's decorator API is not
## Layout
django_strawberry_framework/ holds package source: __init__.py re-exports DjangoType and DjangoOptimizerExtension and pins __version__; conf.py reads the DJANGO_STRAWBERRY_FRAMEWORK settings dict; exceptions.py defines DjangoStrawberryFrameworkError, ConfigurationError, OptimizerError; registry.py owns the model→type registry; converters.py owns convert_scalar, convert_choices_to_enum, convert_relation; types.py owns DjangoType, _validate_meta, _build_annotations; optimizer.py owns DjangoOptimizerExtension; py.typed marks the package typed. examples/fakeshop/ is the example Django project using the standard apps/config layout: orchestration lives under config/ (settings.py, urls.py, schema.py, wsgi.py), and domain apps live under apps/ with explicit dotted imports. The project root is a directory, not an importable package; apps/ and config/ are the importable packages via pythonpath = examples/fakeshop. apps/products/models.py holds the product catalog chain Category > Item and Category > Property, then Entry joining Item and Property. apps/library/models.py holds a separate acceptance-test domain for Branch, Shelf, Book, Genre, Patron, MembershipCard, and Loan. apps/products/services.py exposes seed_data, delete_data, create_users, delete_users, discover_providers. apps/products/schema.py is the aspirational design fully commented out plus a placeholder Query — when implementation lands, strip the leading hash from the design block; the doubly-commented filters/orders/aggregates/perms lines stay commented until those subsystems ship. Tests live in three trees — tests/ (package tests), examples/fakeshop/tests/ (example project, no /graphql HTTP), and examples/fakeshop/test_query/ (example project, live /graphql HTTP); see the Test placement section for the rules. pytest.ini sets DJANGO_SETTINGS_MODULE to config.settings.

## docs/review/ is human-curated
docs/review/ holds the canonical record of every code review cycle. Files directly under docs/review/ — REVIEW.md, review-<version>.md, worker-<n>.md role briefs, and rev-<file>.md per-file review artifacts — are committed source of truth, edited only by humans or by an agent acting on the active review workflow. Never bulk-delete or bulk-overwrite at the docs/review/ level. The static helper and current bug-hunt snapshot write shared shadow byproducts directly under docs/shadow/ per worker-1.md, BUILD.md, and HUNT.md; bug-hunt stripped diffs live under docs/shadow/bug_hunt/. Running scripts/bug_hunt.py or scripts/review_current_from_commit.py wipes the entire docs/shadow/ tree (including the bug_hunt/ subtree) before refreshing the current snapshot, so any prior review/build helper output or diff-helper output is discarded. docs/shadow/ is safe to wipe directly, but it is outside docs/review/. The bug-hunt orchestrator's human-facing files stay under docs/bug_hunt/: docs/bug_hunt/bug_hunt.<sha>.md is the per-file checklist scripts/bug_hunt.py emits and docs/bug_hunt/dicta.md is the maintainer-edited header it prepends. Everything under docs/shadow/ and generated docs/bug_hunt/bug_hunt.*.md files are regenerable script output and may be wiped freely. When clearing tool output, target the specific subdirectory or subdirectories — for example find docs/shadow -mindepth 1 -delete, or find docs/bug_hunt -name 'bug_hunt.*.md' -delete to reset generated bug-hunt checklists — never run a recursive delete rooted at docs/review/ itself, which silently eats every rev-*.md artifact. The same prohibition applies to rm -rf docs/review/* and any glob-expanded form that includes the top-level .md files. If review artifacts go missing, restore from git via git checkout HEAD -- docs/review/ before doing anything else.
## Test placement is mandatory
Three test locations, no overlap. tests/ at the repo root holds package tests — tests whose system-under-test is django_strawberry_framework itself, even when fakeshop models are used as fixtures (test_django_types.py, test_optimizer.py, test_choice_enums.py belong here). Package tests intentionally use real example-project models from apps.products.models and apps.library.models; treat those model surfaces as part of the test contract, not disposable example-only code. tests/base/ is reserved for exactly two files: tests/base/test_init.py (version checking) and tests/base/test_conf.py (django_strawberry_framework/conf.py); both may grow but no new files are ever added there. examples/fakeshop/tests/ holds tests whose system-under-test is the fakeshop example project but that do NOT hit the GraphQL HTTP endpoint — services, admin, management commands, models, urls, and in-process schema execution via schema.execute_sync. examples/fakeshop/test_query/ holds live GraphQL-API tests that DO ping /graphql/ over HTTP via django.test.Client; test_library_api.py is the current acceptance suite for the library app and documents the schema-reload fixture required for registry isolation. Future example projects mirror the same two-folder split: example/<project>/tests/ and example/<project>/test_query/. The empty examples/fakeshop/apps/products/tests/ directory is the per-Django-app convention placeholder and stays empty — do not add files there. pytest.ini sets testpaths = tests examples/fakeshop/tests examples/fakeshop/test_query so uv run pytest discovers all three; the GitHub Actions Run Tests step calls uv run pytest unmodified, so adding/removing test files in any of the three roots flows through CI without workflow edits. Do not add __init__.py to examples/fakeshop/tests/ or examples/fakeshop/test_query/; the root tests/ tree already has __init__.py and adding them under examples/fakeshop/ collides on the top-level tests package name once examples/fakeshop is on pythonpath.
[tool.coverage.run] source = ["django_strawberry_framework"]. The example app is example code, not shipping code; bugs in it don't affect the published package, so it is intentionally outside the coverage gate. HTTP tests under examples/fakeshop/test_query/ cover the package via real Django + Strawberry execution; the example apps' own files (apps/library/, apps/products/) are intentionally outside the coverage gate. Example tests still run and exercise the package end-to-end via real Django flows — that is their value — they just don't gate fail_under.
## Test through real usage
Every line that can be exercised by running the example in a real-world flow should be tested that way. Management commands via django.core.management.call_command capturing io.StringIO, not by importing handle directly. Admin actions via django.test.Client logged in as a superuser hitting admin URLs with query params, not by calling changelist_view directly. URL views via django.test.Client. Schema via schema.execute_sync with a real GraphQL query string. Models via Model.objects.create through their managers. Mock only when the real path is genuinely impossible — for example, forcing builtins.\_\_import\_\_ to raise ImportError to cover a defensive except branch. Mock the behaviour, not the class.
## First step of every test: seed via services
apps.products.services exposes seed_data(count=N), create_users(count=N), delete_data, delete_users. Tests touching the catalog or auth start by calling them. Do not hand-roll Category, Item, Property, Entry, or User instances test by test — fixtures fragment and grow divergent shapes that way. Tests for the seed helpers themselves are the only exception. Library acceptance tests use inline Model.objects.create(...) calls instead; the library app intentionally has no services.py because deterministic small seed shapes matter more than convenience for HTTP query-count assertions. seed_data(1) deterministically produces 25 categories, 174 properties, 25 items, and 174 entries; is_private alternates by sorted index so the public/private split is reproducible across runs. create_users(1) produces 6 users covering staff, regular, and the four view_category, view_item, view_property, view_entry permission roles, all sharing the password admin. Superusers are never created by the helper; build them inline when a test needs one
## Coverage is 100% on the package
fail_under = 100 lives in pyproject.toml under tool.coverage.report. The Django CI workflow gates every push and PR on it. Coverage source is django_strawberry_framework/ only. Add tests in the same change you add code. Sweep all three test trees (tests/, examples/fakeshop/tests/, examples/fakeshop/test_query/) for orphan imports when you remove code. pragma no cover is only for branches genuinely unreachable under the test runner's environment.
## Formatting and lint
Run uv run ruff format . and uv run ruff check --fix . after every edit. Line length is 110. COM812 (flake8-commas missing-trailing-comma) is enabled — a trailing comma on a multi-arg call expands it across lines and locks the layout in. Removing the comma collapses it back. ERA001 (eradicate commented-out code) is enabled but TODO-anchored pseudo code blocks (the "Pseudo:" sections inside spec TODO comments) are exempt — they will be replaced by real code when the slice ships. Do not refactor, rewrite, or remove pseudo code just to satisfy ERA001; suppress with noqa ERA001 inline if needed, or ignore the violations until the slice lands.
## Settings surface today
django_strawberry_framework.conf.settings reads DJANGO_STRAWBERRY_FRAMEWORK from the consumer's Django settings dict; missing keys raise AttributeError. Add new keys only when the feature that needs them lands. Do not preemptively populate the settings module with future-feature placeholders
## Sharded mode
FAKESHOP_SHARDED=1 swaps DATABASES from a single dev sqlite file to db_shard_a.sqlite3 (alias default) and db_shard_b.sqlite3 (alias shard_b). The two modes are mutually exclusive — Django cannot see the other set of files in either mode. seed_shards is the management command that materializes both shards idempotently. Sharded-specific tests must live behind that env-var toggle and must not run under the default pytest invocation
