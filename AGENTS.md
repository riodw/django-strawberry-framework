## Project
django-strawberry-framework — DRF-shaped Django integration for Strawberry GraphQL. Pre-alpha. Goal: provide FilterSet, OrderSet, AggregateSet, DjangoType, DjangoConnectionField, apply_cascade_permissions, and per-field permission hooks — all configured via Meta classes the way DRF, django-filter, and Django itself do it. The differentiator vs strawberry-graphql-django, which uses stacked decorators. See docs/README.md for the full positioning argument
## DRF first, strawberry second
Every public surface uses Meta classes. If you find yourself writing strawberry decorators on consumer-facing classes, stop — that is strawberry-graphql-django's API and the explicit reason this package exists. Strawberry is the engine; DRF is the shape. Cribbing from django-graphene-filters and DRF idioms is encouraged; cribbing from strawberry-graphql-django's decorator API is not
## Layout
django_strawberry_framework/ holds package source: __init__.py re-exports DjangoType and DjangoOptimizerExtension and pins __version__; conf.py reads the DJANGO_STRAWBERRY_FRAMEWORK settings dict; exceptions.py defines DjangoStrawberryFrameworkError, ConfigurationError, OptimizerError; registry.py owns the model→type registry; converters.py owns convert_scalar, convert_choices_to_enum, convert_relation; types.py owns DjangoType, _validate_meta, _build_annotations; optimizer.py owns DjangoOptimizerExtension; py.typed marks the package typed. examples/fakeshop/ is the example Django project; models live in fakeshop/products/models.py with the chain Category > Item and Category > Property, then Entry joining Item and Property. fakeshop/products/services.py exposes seed_data, delete_data, create_users, delete_users, discover_providers. fakeshop/products/schema.py is the aspirational design fully commented out plus a placeholder Query — when implementation lands, strip the leading hash from the design block; the doubly-commented filters/orders/aggregates/perms lines stay commented until those subsystems ship. Tests live in three trees — tests/ (package tests), examples/fakeshop/tests/ (example project, no /graphql HTTP), and examples/fakeshop/test_query/ (example project, live /graphql HTTP); see the Test placement section for the rules. pytest.ini sets DJANGO_SETTINGS_MODULE to examples.fakeshop.fakeshop.settings.

## Test placement is mandatory
Three test locations, no overlap. tests/ at the repo root holds package tests — tests whose system-under-test is django_strawberry_framework itself, even when fakeshop models are used as fixtures (test_django_types.py, test_optimizer.py, test_choice_enums.py belong here). tests/base/ is reserved for exactly two files: tests/base/test_init.py (version checking) and tests/base/test_conf.py (django_strawberry_framework/conf.py); both may grow but no new files are ever added there. examples/fakeshop/tests/ holds tests whose system-under-test is the fakeshop example project but that do NOT hit the GraphQL HTTP endpoint — services, admin, management commands, models, urls, and in-process schema execution via schema.execute_sync. examples/fakeshop/test_query/ holds live GraphQL-API tests that DO ping /graphql/ over HTTP via django.test.Client (currently empty; reserved for the schema build-out). Future example projects mirror the same two-folder split: example/<project>/tests/ and example/<project>/test_query/. The empty examples/fakeshop/fakeshop/products/tests/ directory is the per-Django-app convention placeholder and stays empty — do not add files there. pytest.ini sets testpaths = tests examples/fakeshop/tests examples/fakeshop/test_query so uv run pytest discovers all three; the GitHub Actions Run Tests step calls uv run pytest unmodified, so adding/removing test files in any of the three roots flows through CI without workflow edits. Do not add __init__.py to examples/fakeshop/tests/ or examples/fakeshop/test_query/; the root tests/ tree already has __init__.py and adding them under examples/fakeshop/ collides on the top-level tests package name once examples/fakeshop is on pythonpath.

## Test through real usage
Every line that can be exercised by running the example in a real-world flow should be tested that way. Management commands via django.core.management.call_command capturing io.StringIO, not by importing handle directly. Admin actions via django.test.Client logged in as a superuser hitting admin URLs with query params, not by calling changelist_view directly. URL views via django.test.Client. Schema via schema.execute_sync with a real GraphQL query string. Models via Model.objects.create through their managers. Mock only when the real path is genuinely impossible — for example, forcing builtins.\_\_import\_\_ to raise ImportError to cover a defensive except branch. Mock the behaviour, not the class.
## First step of every test: seed via services
fakeshop.products.services exposes seed_data(count=N), create_users(count=N), delete_data, delete_users. Tests touching the catalog or auth start by calling them. Do not hand-roll Category, Item, Property, Entry, or User instances test by test — fixtures fragment and grow divergent shapes that way. Tests for the seed helpers themselves are the only exception. seed_data(1) deterministically produces 25 categories, 174 properties, 25 items, and 174 entries; is_private alternates by sorted index so the public/private split is reproducible across runs. create_users(1) produces 6 users covering staff, regular, and the four view_category, view_item, view_property, view_entry permission roles, all sharing the password admin. Superusers are never created by the helper; build them inline when a test needs one
## Coverage is 100% on the package
fail_under = 100 lives in pyproject.toml under tool.coverage.report. The Django CI workflow gates every push and PR on it. Coverage source is django_strawberry_framework/ only. Add tests in the same change you add code. Sweep all three test trees (tests/, examples/fakeshop/tests/, examples/fakeshop/test_query/) for orphan imports when you remove code. pragma no cover is only for branches genuinely unreachable under the test runner's environment.
## Formatting and lint
Run uv run ruff format . and uv run ruff check --fix . after every edit. Line length is 110. COM812 (flake8-commas missing-trailing-comma) is enabled — a trailing comma on a multi-arg call expands it across lines and locks the layout in. Removing the comma collapses it back. ERA001 (eradicate commented-out code) is enabled but TODO-anchored pseudo code blocks (the "Pseudo:" sections inside spec TODO comments) are exempt — they will be replaced by real code when the slice ships. ## Settings surface today
django_strawberry_framework.conf.settings reads DJANGO_STRAWBERRY_FRAMEWORK from the consumer's Django settings dict; missing keys raise AttributeError. Add new keys only when the feature that needs them lands. Do not preemptively populate the settings module with future-feature placeholders
## Sharded mode
FAKESHOP_SHARDED=1 swaps DATABASES from a single dev sqlite file to db_shard_a.sqlite3 (alias default) and db_shard_b.sqlite3 (alias shard_b). The two modes are mutually exclusive — Django cannot see the other set of files in either mode. seed_shards is the management command that materializes both shards idempotently. Sharded-specific tests must live behind that env-var toggle and must not run under the default pytest invocation
