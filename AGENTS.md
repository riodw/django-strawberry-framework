## Project
django-strawberry-framework — DRF-shaped Django integration for Strawberry GraphQL. Pre-alpha. Goal: provide FilterSet, OrderSet, AggregateSet, DjangoType, DjangoConnectionField, apply_cascade_permissions, and per-field permission hooks — all configured via Meta classes the way DRF, django-filter, and Django itself do it. The differentiator vs strawberry-graphql-django, which uses stacked decorators. See docs/README.md for the full positioning argument
## DRF first, strawberry second
Every public surface uses Meta classes. If you find yourself writing strawberry decorators on consumer-facing classes, stop — that is strawberry-graphql-django's API and the explicit reason this package exists. Strawberry is the engine; DRF is the shape. Cribbing from django-graphene-filters and DRF idioms is encouraged; cribbing from strawberry-graphql-django's decorator API is not
## Layout
django_strawberry_framework/ holds package source — today only conf.py. examples/fakeshop/ is the example Django project; models live in fakeshop/products/models.py with the chain Category > Item and Category > Property, then Entry joining Item and Property. fakeshop/products/services.py exposes seed_data, delete_data, create_users, delete_users, discover_providers. fakeshop/products/schema.py is the aspirational design fully commented out plus a placeholder Query — when implementation lands, strip the leading hash from the design block; the doubly-commented filters/orders/aggregates/perms lines stay commented until those subsystems ship. tests/base/ is for the package; tests/example/ is for the example app. pytest.ini sets DJANGO_SETTINGS_MODULE to examples.fakeshop.fakeshop.settings
## Test placement is mandatory
The django-graphene-filters predecessor had scattered tests; this is the explicit cleanup we did and want to keep. Tests for the package go in tests/base/. Tests for any example app behaviour — services, admin, commands, schema, urls, models, settings — go in tests/example/. Do not put tests under examples/fakeshop/.../tests/. That path exists only as a Django convention and must stay empty
## Test through real usage
Every line that can be exercised by running the example in a real-world flow should be tested that way. Management commands via django.core.management.call_command capturing io.StringIO, not by importing handle directly. Admin actions via django.test.Client logged in as a superuser hitting admin URLs with query params, not by calling changelist_view directly. URL views via django.test.Client. Schema via schema.execute_sync with a real GraphQL query string. Models via Model.objects.create through their managers. Mock only when the real path is genuinely impossible — for example, forcing builtins.\_\_import\_\_ to raise ImportError to cover a defensive except branch. Mock the behaviour, not the class
## First step of every test: seed via services
fakeshop.products.services exposes seed_data(count=N), create_users(count=N), delete_data, delete_users. Tests touching the catalog or auth start by calling them. Do not hand-roll Category, Item, Property, Entry, or User instances test by test — fixtures fragment and grow divergent shapes that way. Tests for the seed helpers themselves are the only exception. seed_data(1) deterministically produces 25 categories, 174 properties, 25 items, and 174 entries; is_private alternates by sorted index so the public/private split is reproducible across runs. create_users(1) produces 6 users covering staff, regular, and the four view_category, view_item, view_property, view_entry permission roles, all sharing the password admin. Superusers are never created by the helper; build them inline when a test needs one
## Coverage is 100%
fail_under = 100 lives in pyproject.toml under tool.coverage.report. The Django CI workflow gates every push and PR on it. Coverage source spans django_strawberry_framework/ AND examples/fakeshop/fakeshop/ — migrations, manage.py, wsgi.py, and asgi.py are omitted. Add tests in the same change you add code. Sweep tests/ for orphan imports when you remove code. pragma no cover is only for branches genuinely unreachable under the test runner's environment — currently just settings.py's FAKESHOP_SHARDED if-branch, since pytest cannot re-import settings under a different env var
## Formatting and lint
Run uv run ruff format . and uv run ruff check --fix . after every edit. Line length is 110. COM812 (flake8-commas missing-trailing-comma) is enabled — a trailing comma on a multi-arg call expands it across lines and locks the layout in. Removing the comma collapses it back
## Settings surface today
django_strawberry_framework.conf.settings reads DJANGO_STRAWBERRY_FRAMEWORK from the consumer's Django settings dict; missing keys raise AttributeError. Add new keys only when the feature that needs them lands. Do not preemptively populate the settings module with future-feature placeholders
## Sharded mode
FAKESHOP_SHARDED=1 swaps DATABASES from a single dev sqlite file to db_shard_a.sqlite3 (alias default) and db_shard_b.sqlite3 (alias shard_b). The two modes are mutually exclusive — Django cannot see the other set of files in either mode. seed_shards is the management command that materializes both shards idempotently. Sharded-specific tests must live behind that env-var toggle and must not run under the default pytest invocation
