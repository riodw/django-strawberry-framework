# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/).

## Versioning

This project follows a milestone-style cadence during pre-`1.0.0`:

- **Alpha (`0.0.x`)** — each patch ships a coherent feature group on the road to `0.1.0`. Public surface is still in flight; breaking changes can land in any patch. Strict [Semantic Versioning](https://semver.org/spec/v2.0.0.html) does **not** apply here.
- **`0.1.0` (beta release)** — feature parity with `graphene-django` (`⚛️`) and `strawberry-graphql-django` (`🍓`) is complete. Alpha → beta cut-over. Tracked by [037-beta_release_cleanup_verification_alpha_beta-0.1.0][card-beta-release-cleanup-verification-alpha-beta] in [`KANBAN.md`][kanban].
- **Beta (`0.1.x`)** — each patch ships a Layer-3 capability (`FieldSet`, `search_fields`, aggregations, choice-enum naming, fakeshop activation, migration guides) on the road to `1.0.0`. Public surface stabilizes; breaking changes are discouraged but not forbidden.
- **`1.0.0` (stable release)** — full `django-graphene-filters` depth on top of parity. Beta → stable cut-over; **API freeze**. Tracked by [046-stable_release_api_freeze_cleanup_verification_beta_stable-1.0.0][card-stable-release-api-freeze-cleanup-verification-beta-stable] in [`KANBAN.md`][kanban].
- **Stable (`1.x.y`)** — strict [Semantic Versioning](https://semver.org/spec/v2.0.0.html) applies from this point forward: breaking changes require a MAJOR bump, additive changes require a MINOR bump, and bug-fix-only releases get a PATCH bump.

See [`KANBAN.md`][kanban] for the per-card sequencing and the version scope of each patch.

## [0.0.13] - 2026-07-06

### Added
- **DRF serializer-based mutations — `SerializerMutation`.** The serializer-validated write flavor on the DRF-shaped `class Meta` surface (`Meta.serializer_class` over a DRF `Serializer` / `ModelSerializer`, not Strawberry decorators), for consumers whose write validation already lives in a serializer. [`SerializerMutation`][glossary-serializermutation] rides the shipped [`DjangoMutation`][glossary-djangomutation] write pipeline via its overridable seams and reuses the byte-identical frozen [`FieldError`][glossary-fielderror-envelope] envelope `036` froze — a `serializer.is_valid()` field error keys to its field and the serializer's non-field errors key to the same `"__all__"` sentinel, returning a null-object payload rather than raising at the GraphQL boundary. `djangorestframework` is a **soft** dependency: `import django_strawberry_framework` and `from django_strawberry_framework import *` both stay DRF-free, and `SerializerMutation` is a lazy root export resolved through the package `__getattr__`, deliberately **not** in `__all__` (a DRF-absent consumer who reaches the symbol gets a single install-hint `ImportError`).
- **Session-auth mutations — `login` / `logout` / `register` + the `current_user` query.** An opt-in authentication surface imported from the `django_strawberry_framework.auth` submodule (**no** package-root re-export — the four symbols are absent from `__all__`, so `from django_strawberry_framework import *` stays auth-free). `login_mutation()` / `logout_mutation()` / `register_mutation()` are field factories carrying a per-schema `permission_classes=` seam, and `current_user()` is the matching query helper (`me: UserType`). All ride the shared frozen [`FieldError`][glossary-fielderror-envelope] envelope; `register` is a [`DjangoMutation`][glossary-djangomutation] rider that adds `validate_password` + `set_password` over the shipped create pipeline (the plaintext password is validated then hashed — never persisted in the clear, asserted on both the sync and async paths). The family defaults to `AllowAny` — the documented inversion of the write family's deny-by-default [`DjangoModelPermission`][glossary-djangomodelpermission], since login and register must serve the anonymous caller — with every factory still accepting `permission_classes=` for a per-schema gate. The transport is Django's session framework only (no Channels until the `0.0.14` router card); the auth resolvers run through the same write-stack and `check_permission` / `authorize_or_raise` helpers by call rather than re-spelling them.

### Changed
- **`django_strawberry_framework.__version__` is now `0.0.13`.**

## [0.0.12] - 2026-06-23

### Added
- **Form-based mutations — `DjangoFormMutation` / `DjangoModelFormMutation`.** Two new write bases on the DRF-shaped `class Meta` surface (`Meta.form_class` + optional `fields` / `exclude`, not Strawberry decorators), for consumers whose write validation already lives in a Django `Form` / `ModelForm`. [`DjangoModelFormMutation`][glossary-djangomodelformmutation] (a `ModelForm`) subclasses [`DjangoMutation`][glossary-djangomutation] via the shipped `_resolve_model` seam — it supplies its model from `form_class._meta.model`, returns the post-save object in the uniform `node` / `result` slot, and inherits the [`DjangoModelPermission`][glossary-djangomodelpermission] write-auth default and the visibility-scoped `update` locate unchanged. [`DjangoFormMutation`][glossary-djangoformmutation] (a plain model-less `Form`) is its sibling — its own metaclass and declaration registry, no `DjangoType` object slot, the pinned `ok: Boolean!` + `errors: [FieldError!]!` payload, and an overridable `perform_mutate(self, form, info)` write hook (defaults to `form.save()`-if-present else no-op). Both are exported from the package root (two net-new public exports).
- **Form-field → Strawberry-input mapping (`forms/converter.py`).** A `convert_form_field` registry derives the mutation input from the form's *declared* fields (`form_class.base_fields`, narrowed by `Meta.fields` / `Meta.exclude`), reusing the read-side scalar / choice-enum / [`Upload`][glossary-upload-scalar] converters where a form field's type overlaps a Django column type — so a plain `Form` can declare fields a model does not have (a `confirm_email`, a `captcha`). `ModelChoiceField` maps to the target's id (Relay `GlobalID` or raw pk by the target's primary `DjangoType`), `ModelMultipleChoiceField` / `MultipleChoiceField` to a list, and `forms.FileField` / `forms.ImageField` to [`Upload`][glossary-upload-scalar]. A custom form-field subclass with no supported ancestor raises [`ConfigurationError`][glossary-configurationerror] naming the field.
- **The `form.errors` → frozen `FieldError` envelope pipeline.** `form.is_valid()` populates the byte-identical [`FieldError`][glossary-fielderror-envelope] envelope `036` froze — a `clean_<field>` error keys to its field, and the form's `NON_FIELD_ERRORS` bucket keys to the same `"__all__"` sentinel — and returns a null-object payload rather than raising at the GraphQL boundary; the write runs through `form.save()` (`ModelForm`) / `perform_mutate` (plain form), wrapped by the `036` `IntegrityError` → envelope mapper. Every relation id (Relay `GlobalID` or raw pk) is visibility-checked through the related primary type's `get_queryset` before the form runs, closing the raw-pk visibility gap. The `ModelForm` `update` is a partial update (the located instance's fields reconstruct the form payload, so omitted fields are preserved) and its payload object is re-fetched and optimizer-planned. Sync + async, inside the one `transaction.atomic()` boundary `036` set; the `ModelForm` re-fetch rides the `spec-035` **G2** gate (keep `select_related` / `prefetch_related`, suppress `.only(...)`).

### Changed
- **The `036` `DjangoMutationField` factory + mutation base were generalized to expose the form flavors — no model-driven regression.** The shipped [`DjangoMutationField`][glossary-djangomutationfield] was generalized along three axes (target-check, `_resolve` dispatch, and the synthesized `data:` lazy-ref derivation) so it accepts the mutation/form family and routes a form flavor to `forms/resolvers.py`; the `DjangoMutation` base grew overridable `_validate_meta` / `build_input` / `input_type_name` / `input_module_path` / `resolve_sync` / `resolve_async` seams. Every seam defaults to today's model behavior — a model-driven `DjangoMutation` validates, materializes its model-column input, and resolves exactly as before. This is an internal generalization, not a breaking change; no consumer-visible behavior of the `0.0.11` model-driven mutation changes.
- **`django_strawberry_framework.__version__` is now `0.0.12`.**

## [0.0.11] - 2026-06-19

### Added
- **`DjangoMutation` (the write-side foundation).** The package's first write surface — a [`DjangoMutation`][glossary-djangomutation] base class configured through a nested `class Meta` (`model` + `operation` ∈ `{"create", "update", "delete"}`, the DRF shape, not Strawberry decorators), exposed on the schema's `Mutation` type through the [`DjangoMutationField`][glossary-djangomutationfield] factory (the write-side sibling of [`DjangoConnectionField`][glossary-djangoconnectionfield]; assigned with no class-attribute annotation and typed from the generated payload via a `strawberry.lazy` forward-ref). Sync and async resolver paths; the write runs inside one `transaction.atomic()` (async via a single `sync_to_async(thread_sensitive=True)`). Ships under `django_strawberry_framework/mutations/` and is exported from the package root.
- **Auto-generated `Input` / `PartialInput` types.** [Input type generation][glossary-input-type-generation] derives `<Model>Input` (each editable field required only when it has no usable Django `default` / `blank` / `null`, else optional) and `<Model>PartialInput` (every field optional) from the mutation's `Meta.model` editable fields narrowed by its own `Meta.fields` / `Meta.exclude` — reusing the read-side scalar / choice-enum / specialized-scalar converters so read and write share one wire contract. Forward FK / OneToOne map to a single `<field>_id` (a `GlobalID` for a Relay-Node target, the raw pk scalar otherwise, type-checked against the relation target at decode); M2M maps to `list[<id>]` (replace-on-provide / clear-on-empty / unchanged-on-omit). The canonical full editable shape takes the stable `<Model>Input` name; narrowed shapes take a deterministic shape-derived name, and two distinct shapes colliding on one generated name raise [`ConfigurationError`][glossary-configurationerror] at finalization. A consumer overrides either generated shape through `Meta.input_class` (create) / `Meta.partial_input_class` (update), each **merged** with the generated remainder rather than wholesale-replaced. The relation-override contract from `spec-010` holds on the input side.
- **Shared `FieldError` envelope + per-mutation payload.** A public [`FieldError`][glossary-fielderror-envelope] `@strawberry.type` (`field: str`, `messages: list[str]` — graphene-django's `ErrorType` shape) surfaced through a generated `<Name>Payload` carrying the mutated object in a uniform `node` slot (Relay-Node target) / `result` slot (non-Node target) — never model-derived — plus `errors: list[FieldError]!`. `full_clean()` validation (incl. `validate_constraints()` `UniqueConstraint` checks caught before `save()`, multi-field constraints keyed to Django's `"__all__"` sentinel) populates the envelope and returns a null object rather than raising at the GraphQL boundary. The envelope is **frozen here** for reuse by the `0.0.12` form and `0.0.13` serializer / auth flavor cards.
- **Write authorization — `DjangoModelPermission` / `Meta.permission_classes`.** A first-class, DRF-shaped write-auth contract: [`Meta.permission_classes`][glossary-djangomodelpermission] (default `[DjangoModelPermission]`, the Django `add` / `change` / `delete` model perms) plus an overridable `check_permission(info, operation, data, instance=None)`, run by all three operations and kept **separate** from `get_queryset` visibility (can-view ≠ can-write). An unauthorized / anonymous caller is denied with a top-level `GraphQLError` before any write. `DjangoModelPermission` is exported from the package root.
- **Permission- and optimizer-composition for mutations.** `update` / `delete` lookups run through the target type's [`get_queryset`][glossary-get_queryset-visibility-hook] (+ any [`apply_cascade_permissions`][glossary-apply_cascade_permissions]) for visibility — a row the caller cannot see is not-found, no existence leak. The post-write row is re-fetched **by pk** and routed through [`DjangoOptimizerExtension`][glossary-djangooptimizerextension] for the response selection; under the mutation operation the `spec-035` **G2** gate keeps `select_related` / `prefetch_related` but suppresses `.only(...)` column deferral, so the re-fetched instance carries no deferred-field set ([`only()` projection][glossary-only-projection] / [FK-id elision][glossary-fk-id-elision]). The products example gains a live `Mutation` write surface that discharges the `spec-035` G2 live-test handoff.

- **Structured file/image read output — `DjangoFileType` / `DjangoImageType`.** A `FileField` column now converts on **read** to a resolver-backed [`DjangoFileType`][glossary-djangofiletype] output object (`name` non-null; `path` / `size` / `url` nullable and storage-safe), and an `ImageField` to [`DjangoImageType`][glossary-djangoimagetype] (the `DjangoFileType` fields plus nullable `width` / `height`), replacing the earlier `str` mapping. The nullable subfields delegate to a shared `_safe_file_attr` guard that degrades to `null` on the storage-shaped errors a non-filesystem backend or a vanished file raises (`ValueError` / `OSError` / `NotImplementedError`); `SuspiciousFileOperation` is deliberately not swallowed (it propagates as a path-traversal security signal). An empty / absent file resolves the whole object to `null`, so the generated read object is **nullable by default** in the SDL regardless of the column's `null` / `blank` (a `Meta.required_overrides` entry opts into a non-null `DjangoFileType!`). The mapping lives in a new `FIELD_OUTPUT_TYPE_MAP` kept off the shared [`SCALAR_MAP`][glossary-scalar-field-conversion] / filter-input path, so no output object reaches a [`FilterSet`][glossary-filterset] input and the filter / scalar-input value stays `str`. A consumer `attachment: str` annotation override bypasses generated conversion and keeps the legacy `str` shape.
- **`Upload` scalar re-export + mutation-input mapping.** Strawberry's built-in [`Upload`][glossary-upload-scalar] scalar (`NewType("Upload", bytes)`) is **re-exported** from the package root — it needs **no** `_PACKAGE_SCALAR_MAP` entry because it already resolves through Strawberry's built-in `DEFAULT_SCALAR_REGISTRY` (the contrast with the package-custom [`BigInt`][glossary-bigint-scalar]). The generated [`DjangoMutation`][glossary-djangomutation] `Input` / `PartialInput` types ([Input type generation][glossary-input-type-generation]) now map a `FileField` / `ImageField` editable column to `Upload` — required per the shipped per-field rule (a `blank=False` / `null=False` / no-default file column is required in the create `Input`, optional otherwise and in `PartialInput`), widened to `Upload | None` on `blank` / `null`. This lifts the `spec-036` staged `NotImplementedError` seam so file columns participate in the `Meta.input_class` merge override like any scalar; the shipped write pipeline assigns the uploaded file through Django's normal model-field path before `full_clean()` / `save()`.
- **Three net-new public exports.** `Upload` (re-exported from `django_strawberry_framework.scalars`), [`DjangoFileType`][glossary-djangofiletype], and [`DjangoImageType`][glossary-djangoimagetype] (from `django_strawberry_framework.types.converters`) are re-exported from the package root and added to `__all__`.

### Changed
- **Breaking wire-format change**: `FileField` / `ImageField` **read** output now converts to the structured [`DjangoFileType`][glossary-djangofiletype] / [`DjangoImageType`][glossary-djangoimagetype] objects (the object itself nullable by default in the SDL regardless of the Django column's `null` / `blank`) instead of `str` — acceptable pre-`1.0.0`, parallel to the `0.0.6` `PositiveBigIntegerField → BigInt` and `0.0.9` model-anchored `GlobalID` precedents. Opt out per field with a concrete `attachment: str` annotation override. The filter / scalar-input shape is unchanged (still `str`), and the mutation input is the [`Upload`][glossary-upload-scalar] scalar.
- **`django_strawberry_framework.__version__` is now `0.0.11`.**

## [0.0.10] - 2026-06-16

Cascading visibility permissions, plus two optimizer robustness guards (G1/G2).

### Added
- **`apply_cascade_permissions` / `aapply_cascade_permissions` (cascade visibility helper pair).** One call inside a [`DjangoType`][glossary-djangotype]'s [`get_queryset`][glossary-get_queryset-visibility-hook] cascades that type's visibility across its **single-column forward FK / OneToOne edges** — for each such edge it resolves the target type through the registry primary lookup, runs that target type's own `get_queryset`, and intersects `Q(<fk>__in=<visible>) | Q(<fk>__isnull=True)` into the caller's queryset, so a parent row whose target a target hook hides drops out. Four invariants: a module-level `ContextVar` seen-set cycle guard (partial-narrow on cycle break, never a raise, `finally`-reset for WSGI/ASGI request isolation), single-column forward scope (M2M / reverse relations / `GenericForeignKey` / `GenericRelation` / the MTI `<parent>_ptr` parent link all excluded), nullable-FK preservation, and per-edge subquery pinning to the caller's resolved DB alias (`queryset.db`). `fields=` validates loudly — a bare string and any unknown / non-cascadable name raise [`ConfigurationError`][glossary-configurationerror]. The sync helper raises [`SyncMisuseError`][glossary-syncmisuseerror] (coroutine closed) for an async target hook; `aapply_cascade_permissions` runs the same walk through `sync_to_async(thread_sensitive=True)`. Composes with the shipped [`FilterSet`][glossary-filterset] / [`OrderSet`][glossary-orderset] `check_<field>_permission` gates (cascade narrows rows first, gates judge input second — no existence leak), connections, node refetch, list fields, and nested filter branches via the optimizer `Prefetch` downgrade, at zero added query round-trips (the `__in` subqueries compile into the caller's single `SELECT`). Both symbols ship under [`django_strawberry_framework/permissions.py`][permissions] and are exported from the package root.
- **Products cascade activation.** The four [`products`][products-schema] schema `get_queryset` hooks (`Category` / `Item` / `Property` / `Entry`) now call `apply_cascade_permissions`, so visibility cascades across the `Entry → Item → Category` / `Entry → Property → Category` FK chain — exercised live by `services.create_users(1)` permission users over `/graphql/`: staff sees everything, a `view_<model>` user sees non-private rows but loses entries under hidden targets, and anonymous requests lose any entry whose item or property points at a private category.

### Changed
- **Operation-type gating of [`.only()`][glossary-only-projection] (optimizer guard G2).** The [`DjangoOptimizerExtension`][glossary-djangooptimizerextension] now applies column projection for `QUERY` operations only — for a mutation / subscription operation whose root resolver returns a queryset it suppresses all `.only(...)` column deferral at plan-build time while keeping `select_related` / `prefetch_related`, so a mutation-returned queryset never carries a selection-shaped deferred-field set (and never triggers a per-access deferred-field refetch or a deferred-instance partial `save()`). The plan-cache key is unchanged — the printed-AST key already separates `query` from `mutation` / `subscription` documents. [FK-id elision][glossary-fk-id-elision] stays enabled under non-`QUERY` operations, now with a consumer-`.only()` loaded-check: the elision stub verifies the FK column is loaded on the parent row and falls back loudly ([strictness][glossary-strictness-mode]-visible) when a consumer projection deferred it, rather than a silent per-row lazy load.

### Fixed
- **Evaluated-queryset guard (optimizer guard G1).** The [`DjangoOptimizerExtension`][glossary-djangooptimizerextension] no longer silently re-executes a root queryset the consumer already evaluated. If a root resolver evaluated the queryset (`len(qs)`, `bool(qs)`, slicing) before returning it, `_optimize` now returns it unchanged rather than re-running it through an `.only()` / `select_related` clone — closing a doubled query that was invisible to the consumer. The manager-coercion path (`return Model.objects`) still optimizes, since the `.all()` coercion yields a fresh unevaluated queryset.

## [0.0.9] - 2026-06-13

The Relay release: cursor connections, root refetch fields, and model-anchored GlobalIDs.

### Added
- **`DjangoConnectionField` (Relay connection field).** [`DjangoConnectionField`][glossary-djangoconnectionfield] wraps a Relay-Node-shaped [`DjangoType`][glossary-djangotype] into `edges` / `node` / `pageInfo` cursor pagination on Strawberry's native `relay.connection()`, injecting `filter:` / `orderBy:` arguments derived from the type's `Meta.filterset_class` / `Meta.orderset_class` sidecars — no hand-written resolver, no parallel argument declarations. The composition pipeline runs `get_queryset` visibility → `filter` → `orderBy` → default deterministic pk-ordering → optimizer-plan → cursor slice, and the field owns its own optimizer cooperation point; a package-owned guard rejects `first` + `last` together. Ships under [`django_strawberry_framework/connection.py`][connection] alongside the generic [`DjangoConnection`][glossary-djangoconnection]`[T]` return alias.
- **`Meta.connection` opt-in `totalCount`.** `{"total_count": True}` on a Relay-Node type (validated at type-creation) resolves the connection through a generated per-target `<TypeName>Connection` exposing a selection-gated [`totalCount`][glossary-metaconnection]`: Int!` — counted on the post-filter, pre-slice queryset, and carried per connection instance so two aliases with different `filter:` values report independent counts.
- **`DjangoNodeField` / `DjangoNodesField` (root Relay refetch fields).** Root `node(id:)` / `nodes(ids:)` in bare-interface and typed forms (new top-level `django_strawberry_framework/relay.py`, both exported from the package root). Ids are decoded **server-side** through the strategy-aware `decode_global_id` dispatch — the client's claim of an id's type is never trusted — and dispatch honors the `get_queryset` visibility hook: hidden, missing, and uncoercible-pk ids return `null` with no existence leak, while malformed ids raise `GraphQLError` with `extensions={"code": "GLOBALID_INVALID"}`. `nodes(ids:)` is per-type-batched, order-preserving, with positional `null` holes and duplicate-id support; a declared node field on a registry with no Relay-Node-shaped types raises [`ConfigurationError`][glossary-configurationerror] at finalization.
- **Relation-as-Connection upgrade + `Meta.relation_shapes`.** A Relay-Node type's selected many-side relations (reverse FK, forward / reverse M2M) to Relay-Node-shaped targets synthesize a `<field>Connection` sibling at finalization, reusing the [`DjangoConnectionField`][glossary-djangoconnectionfield] machinery — per-target connection classes, sidecar-derived `filter:` / `orderBy:` arguments, and target-driven `totalCount`. The net-new `Meta.relation_shapes` key (`dict[str, str]`, values `"list"` / `"connection"` / `"both"`; `"both"` is the implicit default) narrows the shape per relation; invalid keys/values/targets raise [`ConfigurationError`][glossary-configurationerror], and non-Node targets degrade silently to list-only under the implicit default.
- **`Meta.globalid_strategy` + `RELAY_GLOBALID_STRATEGY` (GlobalID encoding strategy).** Per-type [`Meta.globalid_strategy`][glossary-metaglobalid_strategy] selects the `GlobalID` encode strategy — `"model"` (the new default), `"type"` (the legacy type-name payload), `"type+model"` (transitional — emits model-anchored while decoding both), or a callable `(type_cls, model, root, info) -> str` (arity / sync-ness validated at type-creation; a non-`str` return raises [`ConfigurationError`][glossary-configurationerror] at encode). [`RELAY_GLOBALID_STRATEGY`][glossary-relay-globalid-strategy] is the schema-wide default; precedence is `Meta.globalid_strategy` → `RELAY_GLOBALID_STRATEGY` → `"model"`, frozen at schema-build time.
- **`django_strawberry_framework.testing.relay` (public test helpers).** `global_id_for(type_cls, id)` returns the encoded `GlobalID` a finalized Relay-Node type emits for a pk; `decode_global_id(gid)` re-exports the internal dispatch returning `(target_type, node_id)`. The pair is deliberately asymmetric — decode routes a model-label payload to the model's **primary** type — so round-trip identity holds for lone / primary model-label types and all `type`-strategy payloads. Imported from the module path only (intentionally NOT re-exported from `django_strawberry_framework.testing`).
- **`manage.py inspect_django_type` (diagnostic command).** `inspect_django_type <Type> [--schema <selector>]` prints a finalized `DjangoType`'s per-field GraphQL resolution table (Django field → resolved GraphQL type → nullability → converter row). It reads the resolved annotation from `origin.__annotations__` (so it reflects consumer-authored annotations and overrides, not a `convert_scalar` re-run), dispatches the positional arg by shape (dotted path vs unique bare-name registry lookup), uses `--schema` to register + finalize on a cold CLI process, and reports the interface-supplied `GlobalID!` for a Relay-Node-suppressed pk.
- **`Meta.nullable_overrides` / `Meta.required_overrides`.** Two net-new tuple-set `Meta` keys that decouple a scalar field's GraphQL nullability from its Django column (`T!`→`T` or `T`→`T!`) without an `AlterField` migration or a consumer-authored annotation. Validated at type-creation (unknown / excluded / consumer-authored / relation / Relay-suppressed-pk targets and the both-sets collision raise [`ConfigurationError`][glossary-configurationerror]); scalar-only, and the override flips a choice field's generated enum nullability for free.
- **Connection-aware optimizer planning.** The optimizer recognizes synthesized `<field>Connection` selections in a parent's selection walk and plans a windowed `Prefetch` — `RowNumber()` / `Count(1)` window functions partitioned by the parent key, the slice window derived from the connection's pagination arguments capped by `relay_max_results` — serving `edges` / cursors / `pageInfo` / `totalCount` from the annotations in **one** window-function query per relation per request. It falls back to the per-parent pipeline for ambiguous empty windows (`first: 0`, overshot `after:`) and unplanned shapes (sidecar `filter:` / `orderBy:`, divergent aliases, [`OptimizerHint`][glossary-optimizerhint]`.SKIP`, `.distinct()` targets).
- **Strictness coverage of connection paths.** An unplanned, unserved nested-connection access now fires the [strictness][glossary-strictness-mode] contract (`OptimizerError` under `"raise"`, a logged warning under `"warn"`) through the same resolver-key vocabulary list relations use, closing the silent-N+1 hole the relation-as-Connection upgrade opened.
- **Pagination-aware plan-cache keys.** Variables feeding **nested** connection pagination arguments are now hashed into the [plan-cache][glossary-plan-cache] key (their values bake into windowed prefetches); variables feeding **root** pagination arguments stay out (root slicing is post-plan), including through root-level fragments.
- **`auto`-typed field annotations.** [`auto`][glossary-auto-typed-annotations] is re-exported from the package root (`from django_strawberry_framework import auto`, no direct `strawberry` import), and a [`DjangoType`][glossary-djangotype] field annotated `name: auto` is declare-but-infer: the field is listed explicitly yet its GraphQL type is still inferred from the Django column through the normal `SCALAR_MAP` conversion — the annotation form of a bare [`Meta.fields`][glossary-metafields] entry, not a consumer override. Dogfooded by the scalars example.

### Changed
- **Breaking wire-format change**: the default Relay `GlobalID` payload for a Relay-Node-shaped [`DjangoType`][glossary-djangotype] is now the Django **model label** (`app_label.modelname:<pk>`, e.g. `products.item:42`) instead of the GraphQL type name (`ItemType:42`), so a schema author can rename a GraphQL type (or [`Meta.name`][glossary-metaname]) without invalidating cached client IDs — parallel to the `0.0.6` `PositiveBigIntegerField → BigInt` precedent. The break lands **live**: root `node(id:)` / `nodes(ids:)` decode every emitted id, so every old type-anchored cached id is undecodable under the `model` default the moment the upgrade deploys. Opt out per type via `Meta.globalid_strategy = "type"` or schema-wide via `RELAY_GLOBALID_STRATEGY = "type"` (byte-identical to the pre-`0.0.9` payload); for a deployed schema, deploy `"type+model"` **while the old type names still exist**, let clients age out the cached old ids, and only then rename types or flip to `"model"`. See [`TODAY.md`][today] for the full upgrade sequence. **Multiple `DjangoType`s per model:** a model-label payload is shared by every type over the model and always decodes to the model's **primary** type, so two Relay-Node types over one model that both use the `model` default now emit identical ids and the secondary's refetch as the primary — its distinct identity / `get_queryset` scope collapses onto the primary's (under the pre-`0.0.9` type-name default they were distinct and self-routing). Finalization emits a warning naming the collapsing secondaries; restore disjoint identity with `Meta.globalid_strategy = "type"` on the secondary.
- Migrated `extensions=[DjangoOptimizerExtension()]` to the module-level-singleton factory form (`extensions=[lambda: _optimizer]`), preserving the instance-bound plan cache and removing Strawberry 0.316.0's instance-form `DeprecationWarning`.
- Products example converted to connections-only — the four [`products`][products-schema] `Query` list resolvers are now [`DjangoConnectionField`][glossary-djangoconnectionfield] attributes (the `django-graphene-filters` cookbook mirror).

### Fixed
- **Ordering a list by a to-many relation no longer multiplies parent rows.** [`OrderSet`][glossary-orderset]`.apply_sync` / `apply_async` now order a to-many [`RelatedOrder`][glossary-relatedorder] path (reverse FK, forward / reverse M2M) by a `Min` (ASC) / `Max` (DESC) **aggregate annotation** instead of a raw `order_by("rel__col")` fan-out JOIN — forcing a `GROUP BY` so one row returns per parent. Corrects `0.0.8` behavior: a `DjangoListField` ordered by a to-many relation previously duplicated parents (a branch with three shelves appeared three times). The same fix makes the new [`DjangoConnectionField`][glossary-djangoconnectionfield] correct over a to-many `orderBy` (no duplicate / skipped nodes under the cursors; `totalCount` counts distinct parents).

### Fixed
- `FilterSet._q_for_branch` now threads the live `request` through to nested filterset instances built for `and` / `or` / `not` sub-branches. Previously the sibling instance was constructed with `request=None`, so any `method=`-shaped filter that read `self.request` inside a logical branch silently lost the context. The new shape is the right semantic — a logically-nested filter sees the same request as its top-level peer — and only affects consumers who happened to depend on the request being `None` to disambiguate nested calls (no real-world consumer is expected to take that branch). The behavioral change is observable to any consumer running a custom `Filter(method=...)` callable that reads the request.
- `_pascal_case` (filters/inputs.py internal helper used to derive operator-bag class names) now raises [`ConfigurationError`][glossary-configurationerror] for inputs that contain no word-character tokens (e.g. `"_"`, `""`, `"__"`); previously it silently returned the empty string, producing a downstream naming collision on `f"{name}FilterInputType"`. The fix moves the failure to the source so the consumer sees an actionable message naming the offending field. Only relevant for synthetic test fixtures or single-underscore module-private classes — no real-world consumer FilterSet name should be affected.
- `FilterSet.apply(...)` now translates sync execution of async `get_queryset` hooks into `SyncMisuseError` with an actionable "use apply_async" message, instead of leaking coroutine/async-loop errors from deeper in the filter path.
- Related-filter constraints are re-applied inside logical branches, so a normalized `and_` / `or_` / `not` branch cannot bypass the target `RelatedFilter(queryset=...)` or visibility-derived child queryset that the top-level branch applies.
- Partial range inputs now drop `None` / `UNSET` axes before building positional `RangeFilter` form-data keys. Supplying only one side of a range surfaces through normal form validation instead of leaking sentinels into the filter data.
- Filter input normalization now unwraps enum members structurally with `isinstance(value, enum.Enum)`, so generated and consumer enums are normalized by value rather than by a narrow class identity check.
- The optimizer plan-cache key now includes the selected operation plus reachable named fragment definitions, directive variables from `@skip` / `@include`, the root response path, and the resolver's origin Strawberry type. Plans no longer collide across same-shaped operations with different fragment bodies or across multiple `DjangoType`s for the same model.
- Repeated root field selections now merge all converted root field nodes before planning. A query that selects the same root field twice with different sub-selections no longer drops relations from the later field node.
- `OptimizerHint.prefetch(obj)` validation now computes the rebased `Prefetch` before mutating the in-flight plan. Invalid hint lookups no longer leave phantom resolver keys, connector columns, or non-cacheable flags on a plan that failed validation.
- Cached `OptimizationPlan` instances are finalized with tuple-backed directive fields before cache handoff, making accidental post-cache mutation fail loudly instead of poisoning the cached plan for later requests.
- Optimizer context read/write helpers now handle dict subclasses, frozen mappings, `__slots__` mapping contexts, and `__getitem__` implementations that raise `AttributeError`, preserving plan and strictness hand-off across more Strawberry context shapes.
- Relay `resolve_node` / `resolve_nodes` defaults now await async `get_queryset` hooks on async resolver paths and raise `SyncMisuseError` on sync resolver paths that cannot await them, rather than attempting queryset operations on a coroutine.
- `scalar_for_field(field)` centralizes Django-field-to-scalar MRO lookup, so `DjangoType` scalar conversion and generated filter-input scalar conversion resolve consumer field subclasses and consumer-registered `SCALAR_MAP` entries through the same path.
- `FieldMeta._from_field_shape(...)` is now the shared implementation for canonical Django field descriptors and resolver test-double fallback shapes, keeping cardinality, nullable, target-field, and connector-column metadata from drifting between optimizer and resolver code.

Previously the command wrote the file and exited silently, giving the user no in-terminal signal that the write succeeded. Post-ship polish on [018-schema_export_management_command-0.0.7][card-schema-export-management-command].
- `manage.py export_schema` positional `schema` argument is now a single scalar value rather than a one-element list. Internally the `nargs=1` declaration on `parser.add_argument("schema", ...)` was dropped and the `options["schema"][0]` indexing was replaced with a direct `options["schema"]` read. No consumer-visible behavior change at the shell — the command is still invoked as `manage.py export_schema config.schema [--path schema.graphql]` — and Django's `call_command(...)` accepts the positional as a separate argument either way. Removes the magic-number-indexing pattern post-ship. Post-ship polish on [018-schema_export_management_command-0.0.7][card-schema-export-management-command].
- `manage.py export_schema --path` now requires a value when the flag is given. Previously the `nargs="?"` declaration on `parser.add_argument("--path", ...)` silently accepted a bare `--path` (with no following value) and set `options["path"]` to `None`, indistinguishable from omitting the flag entirely. Dropping `nargs="?"` makes argparse raise `CommandError` at parse time when the user gives `--path` without a value, catching the obvious typo. Omitting `--path` entirely still routes to stdout as before; passing `--path schema.graphql` still writes to the named file. Post-ship polish on [018-schema_export_management_command-0.0.7][card-schema-export-management-command].
- **Breaking change**: `BigInt` registration moved from `strawberry.scalar(NewType("BigInt", int), name="BigInt", ...)` to the `StrawberryConfig.scalar_map` path. Any schema that resolves to `BigInt` — whether through a direct `BigInt` annotation (`category: BigInt`, `@strawberry.field def x(self) -> BigInt: ...`) OR through a [`DjangoType`][glossary-djangotype] field backed by `BigIntegerField` / `PositiveBigIntegerField` (resolved by the [`Specialized scalar conversions`][glossary-specialized-scalar-conversions] converter table) — must add `config=strawberry_config()` to its `strawberry.Schema(...)` call; Strawberry schema construction will fail with `Unexpected type ...BigInt` without it. The migration applies even to consumers who never import or annotate `BigInt` directly, because the converter table resolves the field type to `BigInt` for them. Matches the `PositiveBigIntegerField → BigInt` precedent in `0.0.6`. Single-line migration:

  ```diff
  - schema = strawberry.Schema(query=Query, extensions=[DjangoOptimizerExtension()])
  + from django_strawberry_framework import strawberry_config
  + schema = strawberry.Schema(query=Query, config=strawberry_config(), extensions=[DjangoOptimizerExtension()])
  ```

  The wire format, parser, serializer, and direct-annotation usage of `BigInt` are unchanged. Tracked as [047-warning_free_scalar_registration_via_strawberryconfigscalar_map-0.0.7][card-warning-free-scalar-registration-via-strawberryconfigscalar-map] in [`KANBAN.md`][kanban].

### Fixed
- `Meta.optimizer_hints` no longer raises `IndexError` when `Meta.exclude` covers every concrete + relation field on the model. The pre-fix code in `_validate_optimizer_hints` derived `model = fields[0].model` and crashed on the empty tuple before the consumer could see a typed error; `meta.model` is now threaded through explicitly, so the empty-selection shape raises the normal `optimizer_hints names unknown fields` `ConfigurationError` against the model's full field surface.
- `Meta.fields` / `Meta.exclude` mutual-exclusivity check now fires for a value inherited from a parent `Meta` class. The pre-fix check used `"fields" in meta.__dict__`, which missed an inherited value — a child `Meta` declaring only `exclude` could silently end up with both directives set and produce confusing field-selection downstream. The check now uses `getattr(meta, ..., None) is not None`, so inheritance is honored and an explicit `fields = None` (or `exclude = None`) remains "unset" — matching the `_normalize_fields_spec` rule that returns `None` for `None`.
- `unwrap_return_type` no longer crashes on a bare `typing.List` or bare `list` resolver return-type annotation. The pre-fix code did `get_args(rt)[0]` after `get_origin(rt) is list` — `typing.List` (no parameter) passes the origin check but `get_args` returns `()`, so the index crashed with `IndexError`. Bare `list` missed the origin check entirely and was returned un-peeled. Both shapes now return `typing.Any` as the "unknown element type" sentinel.
- `OptimizerHint.select_related()` plan-time rejection and `OptimizerHint.prefetch(obj)` adapt-time raises now name the offending DjangoType alongside the Django field: `OptimizerHint.select_related() on CategoryType.items: Django requires prefetch_related …` instead of the previous `OptimizerHint.select_related() on 'items': …`, and `OptimizerHint.prefetch(obj) on CategoryType.items requires a Prefetch with a lookup path` / `OptimizerHint.prefetch(obj) lookup on CategoryType.items must target the hinted relation 'items'; got 'unrelated_relation'.` The pre-fix wording let a triage-by-grep walker from a stack trace recover the field name but not the type, forcing the consumer to read the surrounding frames to find the offending `DjangoType`.
- `TypeRegistry.register(model, type_cls, primary=True)` duplicate-primary error now reads `Cannot register AdminItemType as primary for Item; ItemType is already the primary type` instead of `AdminItemType is already declared primary as ItemType`. The pre-fix wording omitted the Django model name, so a stack-trace grep had to guess which model the collision was about — load-bearing context now that one model can carry multiple `DjangoType`s.
- Class-attribute shadow error on a `DjangoType` subclass now attributes to the consumer's subclass name (`CategoryType.items shadows a Django relation field …`) instead of the Django model name (`Item.items shadows …`). The shadow lives on a class attribute of the `DjangoType` subclass, so the consumer-visible site is the `DjangoType`, not the underlying Django model — the consumer needs the `cls.__name__` attribution to find the offending override.
- `manage.py export_schema --path <file>` now wraps file-write `OSError` (e.g. missing parent directory, permission denied, target is a directory) in `CommandError`, matching the existing wrapping for `ImportError` / `AttributeError` from `import_module_symbol`. Previously the user got a raw `FileNotFoundError` / `PermissionError` traceback for what is a user-input error, while schema-resolution failures already exited cleanly — the wrap closes the consistency gap inside `handle()`. Post-ship polish on [018-schema_export_management_command-0.0.7][card-schema-export-management-command].

### Removed
- **Breaking change**: `models.DurationField` and `models.BinaryField` removed from `SCALAR_MAP`. A `DjangoType` with `Meta.fields = "__all__"` (or an explicit `fields` list including either column) on a model that uses one of these field types now raises `ConfigurationError("Unsupported Django field type …")` at class-definition time. The previous mappings (`DurationField → datetime.timedelta`, `BinaryField → bytes`) crashed at schema construction time with a far-away `TypeError` from a Strawberry frame because Strawberry has no first-party scalars for either Python type. Failing locally at `convert_scalar` keeps the error grep-stable at the consumer's offending field. The supported extension hooks are documented in the updated `SCALAR_MAP` docstring: `SCALAR_MAP[BinaryField] = strawberry.scalars.Base64` for binary columns; a consumer-defined Strawberry scalar (e.g. `SCALAR_MAP[DurationField] = MyDurationScalar`) for duration columns. Existing consumers can also fall back to `Meta.exclude = (...)` on the affected column.
- Internal `warnings.catch_warnings()` suppression block in `django_strawberry_framework/scalars.py` that silenced Strawberry's `Passing a class to strawberry.scalar() is deprecated` `DeprecationWarning`. No longer needed — the migrated registration path uses Strawberry's no-warning `strawberry.scalar(name=..., serialize=..., parse_value=...)` overload. Tracked as [047-warning_free_scalar_registration_via_strawberryconfigscalar_map-0.0.7][card-warning-free-scalar-registration-via-strawberryconfigscalar-map] in [`KANBAN.md`][kanban].

### Notes
- Package test suite maintains the 100% coverage gate. Positive pins protect each new error-attribution shape so a future cosmetic refactor cannot silently drop the type-name in a regression. The `_consumer_only_fields` helper in `optimizer/plans.py` (defensive guards against Django's private `query.deferred_loading` contract — missing-attribute, malformed-shape, and empty-`.only()`-set branches) gains direct unit coverage. Per `examples/fakeshop/test_query/README.md`, every new pin lives in the package-internal `tests/` tree because each underlying error fires at class-definition or finalize time (Meta validation, registry registration, `convert_scalar` at `DjangoType` subclass creation) or in defensive helper code that the package itself does not invoke from a live `/graphql/` request path.

## [0.0.6] - 2026-05-19
### Added
- `BigInt` public scalar export — JSON-safe, decimal-string-serialized, with a strict regex parser (`^(0|-?[1-9][0-9]*)$`) and a strict serializer that rejects `bool`, `float`, `str`, `Decimal`, and any non-`int` type with `TypeError`. See [`BigInt` scalar][glossary-bigint-scalar]. Tracked as [013-deferred_scalar_conversions-0.0.6][card-deferred-scalar-conversions] in [`KANBAN.md`][kanban].
- `JSONField → strawberry.scalars.JSON` mapping in `SCALAR_MAP`.
- `HStoreField → strawberry.scalars.JSON` mapping via a sentinel-guarded branch in `convert_scalar` (soft-registered, only when `django.contrib.postgres.fields` imports successfully).
- PostgreSQL `ArrayField` recursion through `field.base_field` via a sentinel-guarded branch in `convert_scalar`. Nested `ArrayField` and outer `choices` on `ArrayField` / `HStoreField` are rejected with `ConfigurationError`.
- `Meta.primary` boolean flag. Multiple `DjangoType` subclasses per Django model. Registry surface: `primary_for`, `types_for`, `models_with_multiple_types`. Tracked as [014-multiple_djangotypes_per_model_with_metaprimary-0.0.6][card-multiple-djangotypes-per-model-with-metaprimary] in [`KANBAN.md`][kanban].
- `TypeRegistry.unregister(type_cls)` public helper. Removes all traces of a type from the registry: types list, model index, primary slot, definition, and pending relations sourced from the type. Honours `_check_mutable`; no-op for unregistered types. Test fixtures that previously poked the registry's private maps to drop a registration now go through this helper.
- Annotation-only scalar field overrides on `DjangoType`. Writing `description: int` (or any other class-level scalar annotation that shadows a Django scalar column selected via [`Meta.fields`][glossary-metafields]) is now a stable public contract — the consumer's annotation wins over the auto-synthesized one and survives `finalize_django_types()` / `strawberry.type(...)` decoration. Mirrors the annotation-only relation-override path that has shipped since `0.0.4` (`DONE-006-0.0.4`). Tracked as [015-consumer_override_semantics_scalar_fields-0.0.6][card-consumer-override-semantics-scalar-fields] in [`KANBAN.md`][kanban].
- `DjangoTypeDefinition.consumer_annotated_scalar_fields: frozenset[str]` — introspection surface for the new override path; symmetric with the existing `consumer_annotated_relation_fields`, `consumer_assigned_relation_fields`, and `consumer_assigned_scalar_fields` sets.
- `ConfigurationError` raised at `DjangoType.__init_subclass__` time when a consumer authors an `id` annotation on a `Meta.interfaces = (relay.Node,)`-shaped type that is not a `relay.NodeID[...]`-marked annotation. Points at `strawberry.relay.NodeID[<pk_type>]` as the supported escape hatch. Replaces the downstream Strawberry-side `ValueError` ("Interface field Node.id expects type ID! but ...") that surfaced only at `strawberry.Schema(...)` construction. Narrow guard: `id: relay.NodeID[int]` is accepted in direct, stringified / PEP 563 / `from __future__ import annotations`, and mixed (directly-resolved `id` alongside other unresolved annotations on the same class) forms; non-`id` consumer scalar overrides on Relay-Node-shaped types (including custom-named primary keys like `code: str` on `models.CharField(primary_key=True)`) are accepted; inherited `id` annotations on a Relay-Node-shaped subclass also pass through at class-creation time (the guard does not walk the MRO) and are silently handled by `_build_annotations`'s pk-suppression branch — Strawberry applies the Relay-supplied `id: GlobalID!` and `resolve_id_attr()` falls back to `"pk"`, so schema construction succeeds. Detection uses `typing.get_type_hints(cls, include_extras=True)` with a fail-soft fallback for unresolved forward references that distinguishes "id itself failed to resolve" (accept only when the raw string matches the token-shaped regex `(?:^|\.)NodeID\[`, so prefixed-substring lookalikes like `"NotNodeID[int]"` are rejected) from "id is directly resolved but another annotation failed" (inspect the resolved object via `_has_node_id_marker`). The fail-soft accept window for unresolved NodeID-shaped strings is package-level guard suppression only — Strawberry's downstream resolution against `cls`'s module globals still applies.

### Changed
- Consolidated field metadata onto `DjangoTypeDefinition` as the single source of truth. Three reader sites in `types/` (`_record_pending_relation`, `resolved_relation_annotation`, `_make_relation_resolver`) now read `FieldMeta` from `DjangoTypeDefinition.field_map` instead of re-deriving relation shape via `relation_kind(field)` + raw `getattr(field, ...)`. The optimizer reads `FieldMeta` from `registry.get_definition(type_cls)` at all four former mirror-reader sites (`walker._resolve_field_map`, `walker._walk_selections`, `extension._collect_schema_reachable_types`, `extension.check_schema`); the legacy `cls._optimizer_field_map` / `cls._optimizer_hints` class-attribute mirrors are retired. Internal refactor only — no public surface or consumer-visible behavior change. Tracked as [012-fieldmeta_single_source_of_truth_consolidation_and_mirror_retirement-0.0.6][card-fieldmeta-single-source-of-truth-consolidation-and-mirror-retirement] in [`KANBAN.md`][kanban].
- `PositiveBigIntegerField` mapping switched from `int` to [`BigInt`][glossary-bigint-scalar]. Breaking wire-format change: `PositiveBigIntegerField` values are now serialized as decimal strings on the wire (not JSON integers) to survive GraphQL's signed 32-bit `Int` boundary. Consumers using the existing 32-bit `int` shape must update wire-format expectations.
- `registry.register` now returns `bool` (whether state was added; was `None`). `registry.register` and `registry.register_with_definition` gained a keyword-only `primary: bool = False` parameter. `registry.get(model)` semantics: returns the primary if declared; the single type if only one is registered; `None` if multiple types are registered with no primary. Tracked as [014-multiple_djangotypes_per_model_with_metaprimary-0.0.6][card-multiple-djangotypes-per-model-with-metaprimary] in [`KANBAN.md`][kanban].
- `registry.iter_types()` now yields once per registered type — a model with multiple types appears multiple times. Consumers iterating to drive a per-model action should explicitly dedupe by model, or use `models_with_multiple_types()` + `types_for(model)` for an explicit grouping.
- `_build_annotations` (`types/base.py`) always defers **auto-synthesized** relation annotations to `PendingRelationAnnotation` + the registry's pending list; the eager-bind shortcut is removed. Consumer-authored relation fields (annotation overrides and assigned `strawberry.field`) continue to skip synthesis entirely — the existing `if field.name in consumer_authored_fields: continue` short-circuit is preserved.
- optimizer plan cache key includes the resolver's origin Strawberry type alongside the model. Primary-return and secondary-return resolvers on the same model produce distinct cache entries.
- Annotation-only and assigned scalar field overrides bypass `convert_scalar` validations and side effects for the overridden field — unsupported-field-type rejection, grouped-choices rejection, `ArrayField` shape rejection, `null=True` widening, and choice-enum registration are skipped. The consumer's annotation is authoritative. `Meta.exclude` and annotation override are now parallel consumer recourses for unsupported scalar fields.
- `id = <StrawberryField>` assignment on a `Meta.interfaces = (relay.Node,)`-shaped `DjangoType` now raises `ConfigurationError` at `__init_subclass__` time. Previously consumers could write `@strawberry.field def id(self) -> relay.GlobalID: ...` (or `id = strawberry.field(description="…")`) and the resulting schema would build because the assigned-field type matched `Node.id: ID!`; this card uniformly rejects assigned `id` overrides on Relay-Node-shaped types for consistency with the annotation-side guard. The supported alternatives are `@classmethod resolve_id` (custom id resolver), `id: relay.NodeID[<pk_type>]` (custom id annotation), and a **resolver-backed sibling field** for the field-level GraphQL metadata use case (declare a separate field with a resolver — e.g., `@strawberry.field(description="…") def display_id(self) -> strawberry.ID: return str(self.pk)` — carrying the metadata AND a value source; the Relay-supplied `id` stays undecorated). Field-level metadata on the Relay-supplied `id` field is not configurable in `0.0.6`; the resolver-backed sibling-field is the documented alternative. **Note**: a metadata-only sibling like `display_id: ID = strawberry.field(description="…")` without a resolver would build but fail at query time because Strawberry's default resolver looks up `display_id` as an attribute on the returned Django model instance and does not find it.
