## Key glossary references

Skim these [`docs/GLOSSARY.md`][glossary] entries first — they anchor the vocabulary used throughout the spec:

- [`DjangoOptimizerExtension`][glossary-djangooptimizerextension] — the subject. G1 lands inside its `_optimize` middleware path ([`extension.py`][extension]) and G2 inside the [`walker.py`][walker] plan-build it drives; G3's deferred narrowing would live in the [`selections.py`][selections] / `walker.py` fragment seam. The entry's shipped-behavior list (root-gated optimization, `Manager` coercion, `only` projection, the [`get_queryset` visibility hook][glossary-get-queryset-visibility-hook] → `Prefetch` downgrade) is the surface these guards refine; Slice 4 appends the G1 + G2 "what the optimizer will not touch" notes (and a G3-deferral pointer).
- [`only()` projection][glossary-only-projection] — the surface G2 gates. The entry pins that scalar selections become `.only(...)` projections with connector columns preserved; G2 suppresses that projection (only `only_fields`, never `select_related` / `prefetch_related`) for non-`QUERY` operations so a mutation-returned queryset never carries a selection-shaped deferred-field set.
- [Plan cache][glossary-plan-cache] — the cache whose key both G2 and G3 must not perturb. The key's first component is the printed operation AST ([`extension.py::_print_operation_with_reachable_fragments`][extension]); G2's cache-safety rests on that print including the `query` / `mutation` / `subscription` keyword, and G3's on narrowing being a pure function of `(document, target_model, origin)` — all three already cache-key components.
- [Queryset diffing][glossary-queryset-diffing] — the B8 consumer-state reconciliation G1 extends. B8 reconciles framework plans against optimization state the consumer already applied (`select_related`, `Prefetch`, `only`); G1 extends the same "respect what the consumer already did" posture to **execution** state (a queryset the consumer already evaluated is not re-executed by an optimizer clone).
- [Strictness mode][glossary-strictness-mode] — the N+1 detector the **deferred** G3 would interact with. The G3 design would make a sibling-concrete-type interface / union plan correct (rather than detected at runtime by strictness) and pin that strictness no longer false-fires for a narrowed sibling fragment — but G3 ships nothing in this card, so that is a carry-forward pin ([Decision 6](#decision-6--g3--registry-only-fragment-type-condition-narrowing)). (Strictness is otherwise unchanged by G1 + G2 — except [Decision 5](#decision-5--g2--fk-id-elision-stays-enabled-under-non-query-operations)'s loaded-check, which keeps a deferred FK access strictness-visible instead of a silent lazy load.)
- [FK-id elision][glossary-fk-id-elision] — the optimization whose interaction with G2 is the card's open decision. Elision reads the FK column off the parent row when a `{ relation { id } }` selection touches only the target pk; [Decision 5](#decision-5--g2--fk-id-elision-stays-enabled-under-non-query-operations) keeps it enabled but adds a loaded-check, because the FK column is **not** always present: a consumer-returned `.only(...)` queryset survives B8 diffing and can defer the FK column even when the optimizer suppresses its own `.only()`, so the elision stub must verify the column is loaded and fall back loudly otherwise.
- [Relation handling][glossary-relation-handling] / [Connection-aware optimizer planning][glossary-connection-aware-optimizer-planning] — the relation-planning contracts the deferred G3 narrowing must not regress; the connection planning ([`DONE-033-0.0.9`][kanban]) last rewrote the fragment-inlining seam, which the follow-up G3 card inherits (its tests can then cover connection-wrapped fragments).
- [`DjangoType`][glossary-djangotype] / [`Meta.primary`][glossary-metaprimary] / [`Meta.interfaces`][glossary-metainterfaces] / [`Meta.relation_shapes`][glossary-metarelation_shapes] / [Relay Node integration][glossary-relay-node-integration] — G3 resolves a fragment's `type_condition` against the current planning type **only**: its own GraphQL name ([`types/definition.py::DjangoTypeDefinition.graphql_type_name`][definition]) and the names of the interfaces it actually implements (the **union** of two complementary sources: `definition.interfaces` — the normalized **declared** `Meta.interfaces` tuple, stored verbatim by [`types/base.py::_validate_interfaces`][types-base], which injects nothing — *and* the interfaces reached by **direct class inheritance**, found by walking `origin.__mro__` for `@strawberry.interface` / `relay.Node` bases. Neither source subsumes the other: a type that is Relay-Node-shaped via `class Foo(DjangoType, relay.Node)` has `relay.Node` only in its MRO, **never** in `definition.interfaces`, per [`_is_relay_shaped`][types-base] (`any(issubclass(i, relay.Node) for i in interfaces) or issubclass(cls, relay.Node)`)). The model's registered primary type name is **not** in the accept set: a `... on PrimaryType` fragment matches only when the primary type ([`Meta.primary`][glossary-metaprimary]) is itself the planning type (its own name then carries it), never when a *secondary* type roots the walk — a `type_condition` matches the runtime GraphQL type, not the shared Django model. Registry / definition metadata only, never a graphql-core schema lookup.
- [`Schema audit`][glossary-schema-audit] — `check_schema` already descends through interface implementors; the deferred G3 narrowing would be the runtime-walk analogue (resolving the concrete-type fragment match through the same registry the audit uses).
- [`ConfigurationError`][glossary-configurationerror] — no new validation surface in this card; named only to confirm none of the three guards adds one.
- [Multi-database cooperation][glossary-multi-database-cooperation] — the guards are alias-agnostic; G1 / G2 / G3 change neither `.using()` preservation nor router-hint behavior.
- [`DjangoConnectionField`][glossary-djangoconnectionfield] — the connection field shares the `apply_to` plan-build-and-apply tail with `_optimize` but builds its own pre-slice queryset, so it is **out of scope** for G1 (its queryset is framework-built and never consumer-evaluated). (The deferred G3 narrowing would apply inside the nested-connection walk; not in this card.)
- [`Meta.optimizer_hints`][glossary-metaoptimizer_hints] / [`OptimizerHint`][glossary-optimizerhint] — unchanged; hints shape relation planning, orthogonal to the execution-state / operation-type / fragment-narrowing concerns of this card.
- [`DjangoMutation`][glossary-djangomutation] / [Auth mutations][glossary-auth-mutations] — the `0.0.11` write-side cohort that makes mutation root resolvers returning querysets a mainstream path; G2 is sequencing-critical *because* of it (ship the gate before the consumers arrive).
- [`apply_cascade_permissions`][glossary-apply_cascade_permissions] — the joint-`0.0.10`-cut sibling ([`DONE-034-0.0.10`][kanban]); independent of these guards (the cascade composes through `get_queryset`, which the walker's downgrade rule honors unchanged), named here only for the shared version boundary.

## Problem statement

The optimizer applies a selection-shaped ORM plan to whatever root queryset a resolver returns, regardless of three properties of that queryset and its operation that upstream `strawberry_django` guards and this package did not. A 2026-06-11 comparative audit of [`django_strawberry_framework/optimizer/`][extension] against [`strawberry_django/optimizer.py`][upstream-optimizer] found three robustness gaps by direct grep and read (not inferred); each exists upstream with a known mechanism and a specific file:line anchor. **G1 closed in commit `d1dea2fd`; G2 shipped in the `DONE-035-0.0.10` build; G3 deferred.**

**G1 — the evaluated-queryset gap (now closed).** [`DjangoOptimizerExtension._optimize`][extension] applied its plan to any root queryset. If the consumer's root resolver already *evaluated* that queryset — a `len(qs)` guard, a `bool(qs)` branch, slicing for a log line — the optimizer's `.only()` / `.select_related()` clone silently re-executed the SQL: a doubled query invisible to the consumer. Upstream guards this twice (the resolve hook only optimizes when `ret._result_cache is None`; `optimize()` re-checks `is_optimized(qs) or qs._result_cache is not None`). The package had zero `_result_cache` references — the execution-state check was missing. **Commit `d1dea2fd` closed it: the execution-state half (`getattr(result, "_result_cache", None) is not None` → pass through) now lives in `_optimize` with tests ([Current state](#current-state)); the flag half stays unported (redundant under the O3 root gate).**

**G2 — the `.only()`-on-mutations gap.** The walker projects `only_fields` onto every root queryset identically, whatever the operation. Upstream disables `.only()` for non-`QUERY` operations (`enable_only and info.operation.operation == OperationType.QUERY`). The risk: a mutation resolver returning a queryset gets a selection-set-shaped `.only()`, so post-mutation consumer code touching any unprojected field triggers one deferred-field refetch query per access — and `Model.save()` on a deferred instance writes only loaded fields (Django's documented deferred-save semantics), a surprising interaction with signal handlers and downstream writes. Before this card the package had zero `OperationType` references; G2 added the operation gate in [`walker.py::_enable_only_for_operation`][walker]. This is **sequencing-critical**: the `0.0.11` mutations cohort makes mutation root resolvers returning querysets a mainstream consumer path, so the gate must land before that release bakes deferred-refetch storms and deferred-`save()` surprises into the first write-side surface.

**G3 — the fragment-type-condition gap.** The walker treats a fragment's `type_condition` purely as a fragment *marker* — [`selections.py::included_field_selections`][selections] inlines every fragment body unconditionally, and `type_condition` is read only to build the inline-fragment shell and to duck-type `is_fragment`, never *matched* against the planning type. Two **latent** failure modes — the guard is verified *absent* (no `type_condition` is matched against a planning type anywhere in the walker), **not** verified to reproduce against a shipped feature; see the reachability note in [Decision 7](#decision-7--g3--narrow-do-not-multi-plan). Both manifest only when the optimizer walks a queryset behind an **interface- or union-typed output position with ≥2 concrete `DjangoType` members** (GraphQL validation forbids `... on SiblingType` under a concretely-typed position, so a sibling fragment is only valid under an abstract one): (a) fields from sibling concrete types miss the current `field_map` and fall through the unknown-name `continue` guard in [`walker.py::_walk_selections`][walker], so every sibling-type relation selection is an N+1 the plan can never cover (strictness detects it at runtime — detection, not prevention); (b) a same-named relation existing on two union / interface members gets planned for the wrong branch — a spurious `select_related` join / over-projection (over-fetch, never wrong data). No shipped fakeshop feature exposes such a position today (the trigger is a consumer-authored multi-implementor interface field, which *is* buildable now); the feature that would exercise G3 end-to-end, `polymorphic_interface_connections`, is an unscheduled [`BACKLOG.md`][backlog] card ([Decision 7](#decision-7--g3--narrow-do-not-multi-plan) records the ship-now-as-forward-hardening choice). Upstream narrows by resolving each model's possible concrete types and re-walking hints per concrete type under a synthesized `ResolveInfo`; this card adopts the *narrowing outcome* through the registry, without the per-type re-walk.

None of the three is a feature; each is a parity-with-`strawberry_django` robustness item the audit pinned at a specific upstream line, and each closes a correctness or safety hole without changing the consumer-visible API surface.

## Borrowing posture

Per the [`START.md`][start] "do both libraries provide it? → foundational" test, all three guards are **Required `strawberry-graphql-django` parity** — the card's own parity tag (🍓 Required). `strawberry_django`'s optimizer guards execution state, gates `.only()` by operation type, and narrows fragment planning to concrete types; this package's optimizer does none of the three. **This card achieves the first two (G1 shipped, G2 here); the third (G3) is deferred** because its narrowing has no reachable trigger until the abstract-return entry contract exists ([Decision 6](#decision-6--g3--registry-only-fragment-type-condition-narrowing) / [Decision 7](#decision-7--g3--narrow-do-not-multi-plan)) — the parity intent is recorded, the runtime is carried forward. The borrowing is at the **outcome** level: the package adopts each guard's *behavior* with its own lighter mechanisms (the O3 root gate, the registry, the printed-AST cache key) rather than porting upstream's heavier machinery, because the package already holds four advantages over upstream the audit confirmed (the global LRU plan cache B1, FK-id elision B2, strictness N+1 detection B3, class-creation-time `FieldMeta` precompute B7) and must not regress them.

### From `strawberry-graphql-django` — borrow the guard outcomes, not the mechanisms

Each guard is ported at the *behavior* level with the package's own seam:

- **G1** adopts the execution-state check (`_result_cache is not None` → pass through) but not the optimized-flag bookkeeping: the O3 root gate already guarantees single application, so a clone never re-enters the optimizer at a nested resolver the way upstream's nested-capable optimizer can.
- **G2** adopts the operation-type gate but lands it at plan-build time (the package's plan is cached by printed AST, which already separates operations — so the gate bakes into the right cache entry with zero key change), where upstream re-checks it at apply time inside a non-cached path.
- **G3** adopts the narrowing *result* (sibling-concrete-type branches don't plan) but resolves the concrete-type match through the package's own registry / definition metadata, planning the requested type once — never the per-concrete-type re-walk with a synthesized `ResolveInfo` that the package's single-plan-per-(document, model, origin) cache contract would fight.

### Explicitly do not borrow

- **The `is_optimized` flag + `_clone` monkeypatch** — monkeypatching `QuerySet._clone` is exactly the kind of upstream-internals coupling the package avoids; the O3 root gate makes it unnecessary ([Decision 3](#decision-3--g1--evaluated-queryset-guard-_result_cache-early-return-in-_optimize)).
- **The per-concrete-type re-walk** — multiplying the plan by concrete type contradicts the package's single-plan-per-key cache and its B7 precompute; the registry narrowing achieves the correctness outcome at a fraction of the walk cost ([Decision 7](#decision-7--g3--narrow-do-not-multi-plan)).
- **Prefetch merging** — the package's consumer-wins drop is a deliberate permission-boundary safety stance, not an oversight ([Out of scope](#out-of-scope-explicitly-tracked-elsewhere)).

**Status: shipped in commit `d1dea2fd` (code + tests; the GLOSSARY note remains in Slice 4).** This Decision is the contract record for the shipped guard, not pending work — the implementation matches it line for line (its docstring cites "G1, `spec-035` Decision 3").

In [`extension.py::DjangoOptimizerExtension._optimize`][extension], after the [`utils/querysets.py::normalize_query_source`][querysets] coercion and its `is_queryset` gate and before the `apply_to` plan-build-and-apply tail, return the result unchanged when `getattr(result, "_result_cache", None) is not None`. Read with `getattr` (defensive access, the package posture). No `is_optimized` flag, no `CONFIG_KEY`, no `QuerySet._clone` monkeypatch.

Placement is load-bearing in two directions:

- **After the manager coercion.** [`normalize_query_source`][querysets] coerces a returned `Manager` (`Model.objects`) to a fresh `.all()` queryset whose `_result_cache` is `None` — so the guard must sit *after* the coercion, or a `Model.objects`-returning resolver would wrongly pass through unoptimized. (The card's manager-coercion line cite is stale; the live DRY home is the symbol [`normalize_query_source`][querysets], called from `_optimize` — recorded in [Risks](#risks-and-open-questions).)
- **Before `diff_plan_for_queryset` / `plan.apply`.** Once past the guard, `apply_to` clones the queryset to attach `select_related` / `only`; a `.only()` clone of an already-evaluated queryset re-executes the SQL. The guard's whole point is to never reach that clone for an evaluated queryset.

Scope is the `_optimize` middleware path only. [`DjangoConnectionField`][glossary-djangoconnectionfield] calls the same `apply_to` tail but on a pre-slice queryset the connection pipeline built and never handed to a consumer resolver — it is never consumer-evaluated, so it needs no guard. The async path (`_async_optimize` awaits the result then calls `_optimize`) inherits the guard unchanged.

Justification: this is upstream's execution-state check, minus the flag bookkeeping the package's O3 root gate makes redundant. Upstream guards twice (`_result_cache is None` at the resolve hook AND `is_optimized(qs) or qs._result_cache is not None` inside `optimize()`) because its optimizer can run at nested resolvers and must stay idempotent across `_clone` calls; the package's optimizer runs only at the operation root (`info.path.prev is None`), so a single execution-state check at the one entry is complete. It extends the [`spec-004`][spec-004] B8 "respect what the consumer already did" posture from optimization state (consumer `.only()` / `select_related` wins) to execution state (consumer-evaluated queryset is left alone).

Alternatives considered (and rejected):

- **Port the full upstream two-part guard (flag + `_clone` monkeypatch).** Rejected: monkeypatching `QuerySet._clone` couples the package to a Django private and exists upstream only to make a nested-capable optimizer idempotent — a problem the O3 root gate already solves. Carrying machinery for a scenario the architecture forbids is dead weight.
- **Guard inside `apply_to` (shared with the connection field) instead of `_optimize`.** Rejected: the connection field's queryset is framework-built and never evaluated, so guarding the shared tail would add a per-connection `getattr` check that can never fire — and would muddy the contract that `apply_to` optimizes whatever pre-built queryset it is handed. The risk is specific to consumer-returned querysets, which only reach `_optimize`.
- **Detect evaluation by `bool(qs._result_cache)` / `len`.** Rejected: `_result_cache` is `None` until evaluated and a (possibly empty) list after — `is not None` is the exact, allocation-free signal upstream uses; truthiness would mis-handle an evaluated-but-empty queryset.

### Decision 4 — G2 — operation-type gating of `.only()`: suppress `only_fields` for non-`QUERY` operations at plan-build time

In [`walker.py::plan_optimizations`][walker], when `info` is present and `info.operation.operation is not OperationType.QUERY`, suppress `only_fields` across the whole plan tree (root plan and every nested child plan) while leaving `select_related` / `prefetch_related` / `fk_id_elisions` intact. The gate is at **plan-build time** in the walker entry point, not apply time in [`plans.py`][plans].

Mechanism: derive `enable_only` once, **defensively** — suppress only when an operation is present and is non-`QUERY`; a `None` `info` *or* a `None` `info.operation` (a partial test-double) falls to the `QUERY`/enabled default rather than raising (the package's `getattr` posture, and the 100% gate needs each arm hit: no-`info` → enabled, `QUERY` → enabled, non-`QUERY` → suppressed). During real execution `info.operation` is always present ([`extension.py::_build_cache_key`][extension] reads it directly), so the defensive arm guards direct / test callers. Thread the gate as an explicit projection flag through **every** writer that populates `only_fields` or applies `.only(...)` — not just the scalar-leaf appends. A "block scalar appends and rely on the connector helper no-op" mechanism is insufficient: the walker writes or applies column projection in four places, and each must consult the gate:

- [`walker.py::_walk_selections`][walker] — Relay custom-pk + scalar-column appends to `plan.only_fields`;
- [`walker.py::_record_relation_access`][walker] — FK connector-column appends on **every** relation traversal; this writer makes `only_fields` non-empty *independently of* scalar appends, so a relation-selecting mutation would otherwise still carry a projection even with scalar leaves blocked;
- [`walker.py::_ensure_connector_only_fields`][walker] — prefetch connector-column appends (the helper that no-ops on empty `only_fields`, but only once nothing else has populated it);
- [`walker.py::_project_scalar_only_window`][walker] — a direct `.only(...)` call on scalar-only nested connection windows; this writer applies projection **without touching** `OptimizationPlan.only_fields` at all, so an empty-`only_fields` check can never catch it.

Carrying the gate explicitly (a plan-level `enable_only` flag every projection writer checks before appending or applying) is preferred over a post-hoc "clear `only_fields` before `plan.finalize()`" sweep precisely because `_project_scalar_only_window`'s direct `.only(...)` never lands in `only_fields` for such a sweep to clear. When the gate is closed (non-`QUERY`), no writer appends to `only_fields` and `_project_scalar_only_window` applies no `.only(...)`; `select_related` / `prefetch_related` / `fk_id_elisions` are untouched. The suppression bakes into the cached plan.

Cache-safety (zero key change): the plan-cache key's first component is the printed operation AST ([`extension.py::_print_operation_with_reachable_fragments`][extension]), and `print_ast(operation)` includes the `query` / `mutation` / `subscription` keyword — so a query document and a textually-identical-selection mutation document can never collide on one cache entry. The two plans (query-with-`only_fields`, mutation-without) coexist under distinct keys; gating at build time means each key caches the correct plan with no new key dimension.

Plan-wide suppression (root + nested): upstream's `enable_only` is a global gate on whether `.only()` is applied at all for the operation, not a root-only one. A mutation returning a queryset whose response selects nested relations would otherwise carry deferred-field sets on prefetched children too, with the same deferred-refetch / deferred-`save()` exposure on those child rows. Suppressing plan-wide matches upstream's behavior and removes the exposure everywhere it can occur.

Justification: G2 is sequencing-critical (the `0.0.11` mutations cohort makes mutation root querysets mainstream); landing it at plan-build time is the cache-correct placement (the printed-AST key already separates operations, so no key change is needed and the suppression is cached, not recomputed per request). `select_related` / `prefetch_related` stay on because they never carry the deferred-field hazard — they shape *which related rows load*, not *which columns of a row* are deferred.

Alternatives considered (and rejected):

- **Apply-time gate in [`plans.py::OptimizationPlan.apply`][plans].** Rejected: a plan built with `only_fields` then conditionally not applying them at apply time means the cache stores a `only_fields`-carrying plan that two operations (query and mutation) would want to apply differently — but the cache already separates them by printed-AST key, so building the right plan per key (build-time) is both simpler and avoids an apply-time branch on `info.operation`. The card pins build-time as preferred for exactly this cacheability reason.
- **Block scalar appends only, relying on `_ensure_connector_only_fields`'s empty-`only_fields` no-op to suppress the rest.** Rejected: this was the original draft's mechanism and it leaks. [`_record_relation_access`][walker] appends FK connector columns on relation traversal, making `only_fields` non-empty *independently of* scalar leaves — so a mutation selecting a relation would still get a non-empty projection and the connector helper would not no-op. And [`_project_scalar_only_window`][walker] applies `.only(...)` directly without populating `only_fields`, so no empty-set check reaches it. The gate must be threaded through all four projection writers, not just the scalar path.
- **Root-only suppression.** Rejected: leaves nested prefetched children carrying deferred-field sets under a mutation, reintroducing the deferred-refetch hazard one level down; upstream gates `.only()` operation-wide.
- **Suppress `select_related` / `prefetch_related` too under non-`QUERY`.** Rejected: those carry no deferred-field hazard and dropping them would reintroduce N+1s on a mutation's response selection — the hazard is specific to column deferral.

### Decision 5 — G2 — FK-id elision stays enabled under non-`QUERY` operations

The card's open decision, resolved with a correctness guard: **FK-id elision stays enabled under non-`QUERY` operations *only when the source FK column is guaranteed loaded* — by either the optimizer plan or the consumer's own projection.** [FK-id elision][glossary-fk-id-elision] reads the FK `attname` column off the parent row for a `{ relation { id } }` selection ([`types/resolvers.py::_build_fk_id_stub`][types-resolvers], which does `getattr(root, field_meta.attname)`). G2 suppressing the *optimizer's* `only_fields` makes the full row load **when the optimizer owns the projection** — but it does not when the **consumer** returns an already-projected queryset.

The consumer-`.only()` hazard (the safety hole the first draft missed): a consumer can return `Item.objects.only("name")`, and the B8 consumer-wins rule in [`plans.py::diff_plan_for_queryset`][plans] **preserves that consumer projection and drops the optimizer's `only_fields`**. So for a mutation (or query) resolver returning `Item.objects.only("name")` with a `{ category { id } }` selection, the plan can still carry an FK-id elision for `category` while `category_id` is **deferred by the consumer projection**. `_build_fk_id_stub`'s `getattr(root, "category_id")` then triggers a **per-row deferred-column fetch** — and worse, because the relation was recorded as *planned*, [Strictness mode][glossary-strictness-mode] does not surface it as an unplanned N+1. This is independent of operation type (it bites under `QUERY` too), so G2's `only`-suppression argument does not cover it.

Implementation rule (the follow-up the first draft owed): the elision branch must verify the FK `attname` is actually loaded before reading it, because the branch did **not** also record a `select_related` fallback to restore. The spec pins the resolver-time form: [`_build_fk_id_stub`][types-resolvers] checks whether `field_meta.attname` is present (the deferred set on `root` / absence from `root.__dict__`) and, when it is **not** loaded, falls back **loudly** — either to the normal related-object resolve (so strictness sees the access and can fire) or to an explicit single-row fetch with a logged warning — never a silent per-row lazy load that strictness mistakes for a planned relation. (Storing enough plan metadata to restore a `select_related` join instead is the heavier alternative; the resolver-time guard is preferred because it is local to the one stub and needs no plan-shape change.)

Justification: elision's correctness precondition is "the FK column is loaded on the parent row." G2 guarantees that *for optimizer-owned projections* but consumer-wins diffing can still defer it, so the precondition must be **checked**, not assumed. Keeping elision on (with the guard) preserves the B2 advantage and avoids a needless join on the common fully-loaded path; the guard only changes the rare consumer-`.only()`-defers-the-FK path, turning a silent lazy-load into a visible, strictness-honest fallback.

Alternatives considered (and rejected): **disable elision entirely under non-`QUERY` ops.** Rejected: it does not address the real hazard (which is consumer projection, not operation type — it bites under `QUERY` too) and trades a correct single-query elision for an unnecessary join on every fully-loaded mutation row. **Drop all elisions after diffing whenever the consumer applied `.only()`.** Rejected as insufficient on its own: the elision branch recorded no `select_related` fallback, so merely dropping the elision still leaves the relation needing a resolve path — the resolver-time loaded-check is what makes the fallback honest. Pinned by `test_fk_id_elision_enabled_under_mutation` (fully-loaded row → elides, no join) **and** `test_fk_id_elision_falls_back_when_consumer_only_defers_fk` (consumer `.only()` defers the FK column → loud fallback, strictness sees the access), under both `QUERY` and non-`QUERY` operations.

### Decision 6 — G3 — registry-only fragment type-condition narrowing

> **Status: DEFERRED — carry-forward requirements, no runtime code in spec-035.** A production-reachability review (Revision 3) established that G3's narrowing has **no reachable trigger today**: an interface / union root field never enters the walker. [`extension.py::_resolve_model_from_return_type`][extension] unwraps `info.return_type`, resolves the abstract type's `origin` (the interface / union class, which is **not** a registered `DjangoType`), and [`registry.model_for_type(origin)`][registry] returns `None` — so [`_optimize`][extension] passes the queryset through *before* the walker (and any fragment classifier) runs. Shipping the classifier now would be synthetic-only. This Decision is therefore **retained verbatim as the design contract for the follow-up *abstract-return optimizer entry* card** (the [`BACKLOG.md`][backlog] `polymorphic_interface_connections` work, or a dedicated card), **not** as work in spec-035. That card must first build the **abstract-return production-entry contract** before this narrowing has anything to narrow — see the carry-forward requirements at the end of this Decision and in [Decision 7](#decision-7--g3--narrow-do-not-multi-plan). Everything below describes the *intended* mechanism for that card.

[`selections.py::included_field_selections`][selections] gains a **tri-state classifier plus a recursion-mode parameter** — **not** a boolean predicate, because three outcomes per fragment are required (see the mechanism note below); [`walker.py::_walk_selections`][walker] supplies the classifier at the `_merge_aliased_selections(included_field_selections(...))` inlining call, where the planning `type_cls` / `definition` are already in hand from [`_resolve_field_map`][walker]. The classifier maps a fragment's `type_condition` to one of three results against the current planning type and nothing else — a `type_condition` matches the runtime GraphQL type or an abstract type that runtime type belongs to, **never** the shared Django model. The rule:

- A fragment with `type_condition is None` (an anonymous inline fragment, `... { f }`) **always inlines** — it carries no type condition and applies to the enclosing type.
- **`INLINE`** — a fragment whose non-`None` `type_condition` names the current planning type inlines fully. The accept set is exactly: the planning type's own GraphQL name ([`types/definition.py::DjangoTypeDefinition.graphql_type_name`][definition]) and the names of the interfaces it actually implements. Implemented-interface names are the **union of two complementary sources**: (a) `definition.interfaces` — the normalized **declared** `Meta.interfaces` tuple, stored verbatim by [`types/base.py::_validate_interfaces`][types-base] (it injects nothing); and (b) the interfaces reached by **direct class inheritance**, found by walking `origin.__mro__` for `@strawberry.interface` / `relay.Node` bases. **Both arms are required and neither subsumes the other**: per [`_is_relay_shaped`][types-base] (`any(issubclass(i, relay.Node) for i in interfaces) or issubclass(cls, relay.Node)`), a type that is Relay-Node-shaped via `class Foo(DjangoType, relay.Node)` has `relay.Node` **only** in its MRO and **never** in `definition.interfaces` — so an MRO-only collection misses declared interfaces and a `definition.interfaces`-only collection misses inherited ones.
- **`SKIP`** — a fragment whose condition names a **known sibling concrete type** (a registry-known object type that is neither the planning type nor one of its implemented interfaces) has its subtree **skipped whole**. A sibling concrete type's fields cannot apply to the planning type (failure mode (a)), and a same-named relation on the sibling must not be planned for this branch (failure mode (b)), so whole-subtree skip is the correct granularity.
- **`RECURSE_FRAGMENTS_ONLY`** — a fragment whose condition names an **unknown composite type** (a union name, or an interface the planning type cannot be confirmed to implement from either source) is **not** skipped whole (that would under-plan a nested `... on <ConcreteType>` fragment that *does* match the planning type). The walk recurses into the unknown fragment's nested fragments, re-classifying each against the planning type, while declining the unknown fragment's **own direct fields** — which the recursion-mode flag (`fragments_only=True`) enforces by dropping non-fragment selections at that level.
- The model's registered **primary** type name is deliberately absent from the accept set. When the primary type itself roots the walk, its own `graphql_type_name` already carries a `... on PrimaryType` fragment (the first bullet); when a *secondary* type roots the walk, that same fragment is a sibling concrete condition and is skipped. There is no separate "model primary name" rule — that rule would inline a primary-type fragment for a secondary resolver and reintroduce the over-planning G3 removes.
- Names resolve through the registry / definition metadata only — **no graphql-core schema lookups** in the walk, preserving the B7 zero-per-request-introspection invariant.

Mechanism (the seam, concretely): [`included_field_selections`][selections] today is a clean boolean recursion (include-or-not per selection; inline every fragment body unconditionally via its self-recursive `included_field_selections(fragment.selections)` call). G3 adds two parameters: an optional `classifier` (returning `INLINE` / `SKIP` / `RECURSE_FRAGMENTS_ONLY` for a fragment's `type_condition`) and a `fragments_only: bool` recursion flag (when `True`, non-fragment selections at this level are dropped). The `RECURSE_FRAGMENTS_ONLY` arm makes the self-recursive call with `fragments_only=True`; `INLINE` recurses as today; `SKIP` returns nothing for that fragment. The `fragments_only` flag **must propagate through the existing self-recursion** so a union nested inside a union still drops its own direct fields. The classifier is threaded **only** through the walker's call site; the extension's cache-key walk and the connection node-selection extraction ([`selections.py::named_children`][selections] / [`node_children_with_runtime_prefix`][selections]) pass **no** classifier (the default is `INLINE`-all with `fragments_only=False` — today's unconditional behavior, byte-for-byte) — they don't plan relations, so the over-fetch / silent-N+1 hazards don't apply there. A nested connection's node selection re-enters `_walk_selections` against the node's model, so a fragment inside `edges { node { ...Frag } }` is narrowed against the node's planning type at that recursion level; the connection-wrapped-fragment test pins this.

This closes both **latent** failure modes (the guard is verified *absent* today, not verified to reproduce against a shipped feature — see [Decision 7](#decision-7--g3--narrow-do-not-multi-plan) and [Risks](#risks-and-open-questions)): (a) sibling-concrete-type fields no longer reach the unknown-name `continue` guard in [`walker.py::_walk_selections`][walker] because the sibling fragment is skipped before inlining — so the silent-N+1 branch the plan could never cover is gone; (b) a same-named relation on two members is planned only for the matching branch — the spurious `select_related` join / over-projection disappears.

Cache-safety (zero key change): the narrowing is a pure function of `(document, target_model, origin)` — the fragment bodies come from the document, the planning type from the origin, the model from the model — all three already plan-cache key components ([`extension.py::_build_cache_key`][extension]). Narrowed plans cache correctly with no new key dimension.

Justification: `type_condition` is already carried through the substrate (the inline-fragment shell, the `is_fragment` duck-type); G3 is the first code to *match* it against the planning type. Resolving the match through the registry (the type's `graphql_type_name` and the union of its declared `definition.interfaces` + MRO-inherited interface bases) reuses the exact metadata [`Schema audit`][glossary-schema-audit] already descends and keeps the walk free of per-request schema introspection. Confining the accept set to the planning type's own name and the interfaces it implements — and excluding the shared model's primary type name — keeps the narrowing faithful to GraphQL type-condition semantics (a condition matches the runtime type or an abstract type it belongs to, not the Django model behind it). Threading the classifier only through the walker (not the shared primitive's other callers) contains the change to the one path that plans relations.

Alternatives considered (and rejected):

- **graphql-core schema lookup of possible types per fragment.** Rejected: violates the B7 invariant (zero per-request Django / schema introspection); the registry already answers "does this planning type satisfy this type condition" from finalized metadata.
- **Accept the model's registered primary type name (the original draft's third accept rule).** Rejected: a `type_condition` matches the runtime GraphQL type, not the Django model. Accepting the primary name would inline a `... on PrimaryType` fragment while planning a *secondary* type over the same model, planning fields / relations the secondary may not expose and crossing distinct `get_queryset` / `relation_shapes` / field-override contracts — the exact over-planning G3 removes. The plan cache already keys on the origin Strawberry type, so there is no cache reason to blur primary and secondary. When the primary type itself roots the walk, its own `graphql_type_name` accepts the fragment anyway, so dropping the rule loses no valid match.
- **Collect interface names from a single source (either `definition.interfaces` alone or `origin.__mro__` alone).** Rejected: the two sources are **complementary, not redundant**, so either alone is incomplete. `definition.interfaces` is the normalized **declared** `Meta.interfaces` tuple ([`types/base.py::_validate_interfaces`][types-base] stores it verbatim and injects nothing); interfaces implemented by **direct class inheritance** (`class Foo(DjangoType, relay.Node)`) appear **only** in `origin.__mro__`, never in `definition.interfaces` (per [`_is_relay_shaped`][types-base]'s `... or issubclass(cls, relay.Node)` arm). A `definition.interfaces`-only collection silently misses every inherited interface (re-introducing the silent-N+1 for a fragment conditioned on an inherited interface); an MRO-only collection misses declared ones. The accept set must be the **union** of both arms.
- **Skip every non-matching condition whole, including unknown composite / union conditions.** Rejected: a union or unrecognized-abstract condition can wrap a nested `... on <ConcreteType>` fragment that *does* match the planning type; skipping the whole subtree under-plans that valid nested fragment. The unknown-composite fallback recurses into nested fragments (re-classifying each) while declining the unknown fragment's own direct fields — conservative in both directions (no under-plan of a valid nested match, no over-plan of unconfirmable direct fields).
- **Skip only the unknown-name fields, not the whole fragment subtree (for sibling concrete types).** Rejected: a sibling-type fragment can name a relation that happens to exist on the planning type too (failure mode (b)); skipping field-by-field on the unknown-name guard misses the same-named-relation over-fetch. For a *known sibling concrete type* the non-matching fragment subtree is skipped whole — that is the correct granularity (distinct from the unknown-composite case above, which recurses).

#### Carry-forward requirements for the follow-up card (added in Revision 3)

These are obligations the abstract-return optimizer entry card must satisfy; they are *not* implemented in spec-035.

- **(R1) Abstract-return production-entry contract — the precondition for everything else.** G3's narrowing only matters once the optimizer actually walks an abstract (interface / union) root field, which it does **not** today ([`registry.model_for_type`][registry] returns `None` for the abstract origin, so [`_optimize`][extension] passes through). The follow-up card must define how an interface / union return resolves: the **target model(s)** (one model, or a per-concrete-type set), the **origin / plan-cache identity** to use when the origin is abstract (the cache key today is `(document, target_model, origin)` — an abstract origin needs a defined identity or a per-concrete-type fan-out), the **possible concrete types** enumeration (registry-only — no per-request graphql-core introspection, per the B7 invariant), and the **strictness** behavior for an abstract walk. The narrowing classifier below is meaningless until this entry contract exists.
- **(R2) Both walker inliner consumers must use the classifier.** The original framing named only [`walker.py::_walk_selections`][walker] as the classifier call site, but the walker has a **second** consumer of the fragment inliner: [`walker.py::_selected_scalar_names`][walker] (the FK-id-elision-safety analyzer for `{ relation { id } }` selections) also calls `included_field_selections(...)`. If the classifier is threaded only through `_walk_selections`, relation child selections are still flattened unconditionally during elision analysis, so elision decisions could be made from sibling / unknown-composite fragments under different semantics than the main walk. The follow-up card must **either** thread the same classifier into `_selected_scalar_names`, **or** prove and document that it only ever receives concretely-typed relation child selections (where sibling fragments are GraphQL-invalid) and so needs none.
- **(R3) Concrete name-resolution + ambiguity contract.** "Known sibling concrete type" needs a **non-Relay** registry lookup: the existing [`registry.definition_for_graphql_name`][registry] is Relay-Node-specific (it filters `implements_relay_node`) and raises on miss / ambiguity, so G3 cannot reuse it. Define the primitive as a scan of [`registry.iter_definitions()`][registry] matching `definition.graphql_type_name` across **all** registered `DjangoType`s. Define **ambiguity behavior** explicitly: duplicate GraphQL names must fail closed (classify as `RECURSE_FRAGMENTS_ONLY`, or raise a loud implementation error in tests) — never an implicit first-match. Interface GraphQL names (for both the declared `definition.interfaces` arm and the MRO-inherited arm) must come from the **Strawberry definition metadata** (honoring `Meta.name` / `@strawberry.interface(name=...)`), not the Python class `__name__`, so renamed interfaces do not drift.

### Decision 7 — G3 — narrow, do not multi-plan

> **Status: DEFERRED — carry-forward requirement, no runtime code in spec-035** (see [Decision 6](#decision-6--g3--registry-only-fragment-type-condition-narrowing)). The narrow-not-multi-plan posture below is a constraint the follow-up abstract-return optimizer entry card must honor; it ships nothing in this card.

The narrowing plans the *requested* type once and excludes non-matching fragment subtrees; it does **not** resolve a model's possible concrete types and re-walk hints per concrete type under a synthesized `ResolveInfo` (upstream's `get_possible_concrete_types` + per-type re-walk). The package narrows; it does not multi-plan.

Justification: the package's plan cache stores one plan per `(document, target_model, origin)` key; a per-concrete-type re-walk would either multiply cache entries or build a union plan that re-introduces the over-projection G3 removes. The registry narrowing achieves the correctness outcome (sibling branches don't plan; same-named relations plan only for the matching branch) at one extra set-membership check per fragment, preserving B7 precompute and the single-plan-per-key contract. Upstream multi-plans because its optimizer lacks the package's class-creation-time metadata and global plan cache — the package doesn't need to.

Alternatives considered (and rejected): **adopt upstream's per-concrete-type re-walk for completeness.** Rejected: it fights the package's cache contract and B7 advantage for a correctness outcome the narrowing already achieves; the card is explicit ("we narrow, we do not multi-plan").

**Reachability and the deferral decision.** Unlike G2 — which has a concrete, dated trigger (the `0.0.11` mutations cohort makes mutation-root querysets a mainstream consumer path, so the gate must precede it) — G3's narrowing has **no reachable trigger at all today**, and not merely "no shipped feature": the optimizer cannot even enter the walker for an abstract output position. A `... on Sibling` fragment is GraphQL-valid only under an interface- or union-typed position, and for such a return [`extension.py::_resolve_model_from_return_type`][extension] resolves the abstract `origin` (the interface / union class) and [`registry.model_for_type(origin)`][registry] returns `None`, so [`_optimize`][extension] passes the queryset through before the walker runs. (An earlier draft of this Decision claimed a consumer-authored multi-implementor interface field "would be mis-walked today" — that was **wrong**: it would be passed through unoptimized, not mis-walked. The bug only appears once an abstract-return *entry contract* exists to make the optimizer walk such a field — requirement R1 in [Decision 6](#decision-6--g3--registry-only-fragment-type-condition-narrowing).) The shipped surface has no abstract optimizer-planned position at all: connection / list / root fields are concretely typed; the Relay `node` / `nodes` refetch fields' optimizer cooperation is deferred per [`docs/GLOSSARY.md`][glossary]; the package generates no unions; `polymorphic_interface_connections` is an unscheduled [`BACKLOG.md`][backlog] card.

## Edge cases and constraints

- **G1 — guard sits after manager coercion.** A `Model.objects`-returning resolver is coerced by [`normalize_query_source`][querysets] to a fresh `.all()` whose `_result_cache` is `None`; the guard must follow the coercion so that path still optimizes (pinned by `test_optimizer_still_optimizes_manager_after_evaluated_queryset_guard`).
- **G1 — sliced-but-unevaluated queryset still optimizes.** `qs[:5]` returns a new queryset with limits but `_result_cache is None` until iterated; the guard does not fire, the optimizer plans it, and the slice composes (a `len(qs)` / `bool(qs)` / iteration is what sets `_result_cache`). The guard targets *evaluated* state, not *sliced* shape.
- **G1 — `.values()` / `.values_list()` returns are unchanged.** They resolve to non-model rows; the return-type resolution path already handles them as today, and the guard only adds a pass-through for the evaluated case — no new behavior on values querysets.
- **G1 — async path inherits the guard.** `_async_optimize` awaits the result then calls `_optimize`; an evaluated awaited queryset passes through identically (pinned by an async mirror test).
- **G2 — every projection writer checks the gate, not just scalar appends.** Blocking scalar leaves alone is insufficient: [`_record_relation_access`][walker] appends FK connector columns on relation traversal (non-empty `only_fields` independent of scalars, so [`_ensure_connector_only_fields`][walker]'s empty-set no-op would not fire) and [`_project_scalar_only_window`][walker] calls `.only(...)` directly (never touching `only_fields`). Under a non-`QUERY` operation the `enable_only` gate is consulted in all four projection writers ([Decision 4](#decision-4--g2--operation-type-gating-of-only-suppress-only_fields-for-non-query-operations-at-plan-build-time)), so no FK / prefetch connector column is appended and no scalar-window `.only(...)` is applied — a relation-selecting mutation and a scalar-only nested connection window both end with no column deferral on the *applied* queryset.
- **G2 / Decision 5 — consumer-provided `.only(...)` can defer the FK column (both `QUERY` and non-`QUERY`).** A consumer-returned `Item.objects.only("name")` survives B8 consumer-wins diffing ([`diff_plan_for_queryset`][plans] keeps the consumer projection, drops the optimizer's `only_fields`). With a `{ category { id } }` selection the plan may still carry an FK-id elision for `category` while `category_id` is deferred — `_build_fk_id_stub`'s `getattr(root, "category_id")` then lazy-loads per row, and because the relation is recorded *planned*, strictness stays silent. This is **not** operation-type-specific (it bites under `QUERY` too). [Decision 5](#decision-5--g2--fk-id-elision-stays-enabled-under-non-query-operations) handles it: the elision stub verifies the FK `attname` is loaded and falls back **loudly** (visible to strictness) when it is not — pinned by `test_fk_id_elision_falls_back_when_consumer_only_defers_fk`.
- **G2 — subscription operations are gated identically.** `info.operation.operation is not OperationType.QUERY` covers both `MUTATION` and `SUBSCRIPTION`; the gate is "QUERY-only projects `only`," so subscriptions drop `only_fields` like mutations (pinned).
- **G2 — a mutation returning a non-queryset never reaches the gate.** The common mutation shape returns a model instance or a payload type, which `normalize_query_source` reports as non-queryset and `_optimize` passes through; the gate matters only for the (rarer) mutation-resolver-returns-queryset shape, which is exactly the `0.0.11` path the gate protects.
- **G2 — a mutation returning a `DjangoConnectionField` is gated by construction.** A connection field builds its own pre-slice queryset and routes it through [`plan_optimizations`][walker], so `enable_only` is derived there from the same `info` and a non-`QUERY` operation suppresses the connection's projection automatically — there is no separate gate site to thread. This is a property of the build-time placement ([Decision 4](#decision-4--g2--operation-type-gating-of-only-suppress-only_fields-for-non-query-operations-at-plan-build-time)), so the path is covered by construction rather than a dedicated test (mutations returning connections are rare); recorded here as a decision, not an untested gap.
- **G2 — query and mutation plans coexist.** A query and a mutation with byte-identical selection sets get distinct printed-AST cache keys (the operation keyword differs), so the `only`-carrying query plan and the `only`-suppressed mutation plan never overwrite each other (pinned).
- **G2 — a missing `info` *or* `info.operation` defaults to enabled.** The derivation is defensive (the package's `getattr` posture): `enable_only` is suppressed **only** when an operation is present and is non-`QUERY`; a `None` `info`, or an `info` whose `operation` is absent (a partial test-double), falls to the `QUERY`/enabled default rather than raising `AttributeError`. During real execution `info.operation` is always present ([`extension.py::_build_cache_key`][extension] already reads it directly), so the defensive arm exists for direct / test callers of [`plan_optimizations`][walker]. All three arms are covered (no-`info` → enabled; `QUERY` → enabled; non-`QUERY` → suppressed) so the 100% gate has a hit on each.
**G3 edge cases below are DEFERRED carry-forward design** (not behavior in spec-035) — they describe the follow-up abstract-return optimizer entry card's narrowing once its production-entry contract ([Decision 6](#decision-6--g3--registry-only-fragment-type-condition-narrowing) R1) exists:

- **G3 — anonymous inline fragment always inlines.** `... { f }` carries `type_condition is None`; it applies to the enclosing type and is never narrowed (the existing `test_anonymous_inline_fragment_*` live coverage must stay green).
- **G3 — interface-typed planning (both collection arms required).** A fragment `... on <Interface>` inlines when the interface name is in the planning type's implemented-interface set; a `... on <SiblingConcreteType>` does not. The set is the **union** of two complementary sources, because a declared interface and an inherited one live in different places: `Meta.interfaces = (relay.Node,)` puts `relay.Node` in `definition.interfaces`, but `class GenreType(DjangoType, relay.Node)` (direct inheritance) puts it **only** in `origin.__mro__` and **not** in `definition.interfaces` (per [`_is_relay_shaped`][types-base]). So the collection must read both — `definition.interfaces` for declared, an `origin.__mro__` walk for inherited — or it silently misses one kind.
- **G3 — secondary-type planning and a `... on PrimaryType` fragment.** When a secondary [`DjangoType`][glossary-metaprimary] roots the walk (via `source_type`), a fragment naming the model's **primary** type is a sibling-concrete-type condition and is **skipped** — a `type_condition` matches the runtime GraphQL type, not the shared Django model, so `... on PrimaryType` does not apply to a `SecondaryType` resolver even though both map to one model. When the **primary** type itself roots the walk, the same fragment inlines via the planning type's own `graphql_type_name`. Pinned by a secondary-return regression so the type-condition semantics are documented, not mistaken for a model-identity match.
- **G3 — unknown composite / union `type_condition`.** A condition the walk cannot classify as the planning type, an implemented interface, or a known sibling concrete type — a union name, or an interface reached only by a path the registry does not record — is **not** skipped whole (that would under-plan a nested `... on <ConcreteType>` fragment that *does* match the planning type). The walk recurses into the unknown fragment's nested fragments, re-classifying each against the planning type, while declining the unknown fragment's own direct fields; the fallback is test-pinned.
- **G3 — connection-wrapped fragments narrow at the node walk.** A fragment inside `edges { node { ...Frag } }` is narrowed against the node's planning type when `_walk_selections` re-enters for the node model, not at extraction time; the connection-wrapped-sibling test pins that the extraction helpers (which pass no predicate) don't defeat the narrowing.
- **G3 — strictness stays branch-sensitive.** Planned resolver keys remain keyed on `(type_cls, field, runtime_path)`; a correctly-narrowed sibling fragment's relation is neither planned nor executed (the resolver never runs for that concrete type), so [Strictness mode][glossary-strictness-mode] does not fire — the old silent-N+1 signature is gone, and no false positive replaces it (pinned, no regression in `tests/optimizer/` strictness coverage).
- **G3 — no per-request introspection.** The narrowing reads `graphql_type_name` and the implemented-interface names (`definition.interfaces` / the `origin.__mro__`) from finalized definition metadata; it issues no graphql-core schema lookup in the walk (B7 invariant preserved).
- **Cross-guard — plan-cache key is untouched.** G1 returns before plan build (no key involved); G2 is a pure function of components already in the key (the deferred G3 would be too). The B1 cache-hit path gains zero allocations.
- **Cross-guard — multi-database behavior unchanged.** Neither G1 nor G2 (nor the deferred G3) reads or writes a DB alias; `.using()` preservation and router hints ([Multi-database cooperation][glossary-multi-database-cooperation]) are untouched.

- `test_sibling_type_fragment_excluded_from_plan` — on an interface / union query, sibling-concrete-type fragment bodies are excluded from the plan (no spurious `select_related` / `only` entries).
- `test_interface_implementor_fragment_planned` — a fragment matching an interface the planning type implements **by direct class inheritance** (collected from `definition.interfaces` / the `origin.__mro__`, the case with no live fakeshop shape) still inlines and plans. (The own-GraphQL-name matching-type case is covered by the live test above, against `GenreType`.)
- `test_same_named_relation_on_two_members_regression` — a relation named identically on two union / interface members is planned only for the matching branch (no over-projection on the other).
- `test_primary_fragment_skipped_under_secondary_accepted_under_primary` — the [Major-2 / Decision 6](#decision-6--g3--registry-only-fragment-type-condition-narrowing) regression: a `... on PrimaryType` fragment is **skipped** when a secondary type roots the walk (sibling-concrete condition), and **inlines** when the primary type itself roots it (own `graphql_type_name`) — the model-identity match is not made.
- `test_unknown_union_condition_recurses_without_direct_fields` — an unknown composite / union `type_condition` recurses into a nested `... on <ConcreteType>` fragment that matches the planning type (so it still plans) while the union fragment's own direct fields are declined — the conservative fallback, neither under- nor over-planning.
- `test_anonymous_inline_fragment_still_inlines` — `... { f }` (`type_condition is None`) inlines unchanged.
- `test_strictness_no_false_fire_for_narrowed_sibling_fragment` — strictness `"warn"` / `"raise"` no longer fires for a correctly-narrowed sibling fragment the resolver never executes; B3 keys stay branch-sensitive.
- `test_connection_wrapped_sibling_fragment_narrowed` — a sibling-type fragment inside `edges { node { ...Frag } }` is narrowed at the node walk (extraction helpers don't defeat it).
- `test_selected_scalar_names_uses_same_classifier` (R2) — the FK-id-elision-safety analyzer [`walker.py::_selected_scalar_names`][walker] makes its elision decision under the **same** classifier as the main walk (or, if R2 proves it only ever sees concretely-typed relation child selections, a test pinning that invariant) — so elision is never decided from a sibling / unknown-composite fragment the main walk would have skipped.
- `test_non_relay_sibling_name_lookup_and_ambiguity` (R3) — the non-Relay registry lookup resolves a sibling concrete type by `graphql_type_name` across `iter_definitions()`, and a duplicate GraphQL name fails closed (`RECURSE_FRAGMENTS_ONLY` or a loud error), never an implicit first-match.

## Risks and open questions

Each item names a preferred answer for the current cut and a fallback if implementation reveals the preferred answer is wrong.

- **G2 FK-id elision under non-`QUERY` operations (the card's open decision).** Preferred answer ([Decision 5](#decision-5--g2--fk-id-elision-stays-enabled-under-non-query-operations)): keep elision enabled — with `only` suppressed the full row loads, the FK column is present, elision is correct and avoids a join. Fallback: if a real consumer surfaces a deferred-elision interaction under mutations (none is known, and the elision-correctness precondition is strictly better satisfied with `only` suppressed), gate elision alongside `.only()` — a one-line addition to the same operation-type branch, test-pinned either way.
- **G2 nested-plan `only_fields` suppression scope.** Preferred answer: suppress plan-wide (root + nested child plans), matching upstream's operation-wide `enable_only`. Fallback: if a consumer relies on nested child-row projection under a mutation (and accepts the deferred-refetch hazard on those children), root-only suppression is a contained narrowing — but it reintroduces the exact hazard one level down, so plan-wide is the safe default.
- **G3 connection-wrapped fragment narrowing.** Preferred answer: narrowing happens at each `_walk_selections` entry (the node model's planning type), so connection-wrapped fragments narrow without touching the extraction helpers. Fallback: if a test shows the extraction helpers ([`named_children`][selections] / [`node_children_with_runtime_prefix`][selections]) flatten a fragment before the node walk re-resolves type, thread the same classifier into those helpers — a contained extension of the Decision 6 mechanism.
- **G3 `... on PrimaryType` fragment when planning a secondary type.** Resolved ([Decision 6](#decision-6--g3--registry-only-fragment-type-condition-narrowing)): the primary type name is **not** in the accept set — a `... on PrimaryType` fragment is skipped when a *secondary* type roots the walk (a type condition matches the runtime GraphQL type, not the shared Django model) and inlines only when the primary type itself roots the walk via its own `graphql_type_name`. Pinned by the secondary-return regression. Fallback: none needed — the strict reading is the GraphQL-correct one and removes the over-planning the permissive model-identity match would have reintroduced (planning fields / relations the secondary may not expose, crossing distinct `get_queryset` / field-override contracts).
- **G3 interface-name collection source and unknown abstract conditions.** Resolved ([Decision 6](#decision-6--g3--registry-only-fragment-type-condition-narrowing)): implemented-interface names are the **union** of two complementary sources — `definition.interfaces` (the normalized **declared** `Meta.interfaces`, stored verbatim by [`types/base.py::_validate_interfaces`][types-base]; it injects nothing) and the interfaces reached by **direct class inheritance**, found by walking `origin.__mro__`. These are *not* primary-plus-fallback: per [`_is_relay_shaped`][types-base], a type Relay-Node-shaped via `class Foo(DjangoType, relay.Node)` has `relay.Node` only in its MRO and never in `definition.interfaces`, so reading either source alone silently drops one kind of interface (the declared, or the inherited). Both arms are required. An unknown composite / union condition recurses into nested fragments (re-classifying each) while declining its own direct fields rather than skipping the subtree whole. Fallback: if neither source records an interface a fragment names, the condition is treated as an unknown composite (recurse-without-direct-fields), never a silent whole-subtree skip.
- **Card-citation correction: the manager-coercion seam.** The card's line cite for the manager coercion the G1 guard must follow is stale; the live home is the symbol [`utils/querysets.py::normalize_query_source`][querysets], called from [`extension.py::_optimize`][extension] (the cite predates the DRY consolidation). Preferred answer: ground the guard on `normalize_query_source` (the live seam); the placement contract ("after coercion, before `diff_plan_for_queryset`") is unchanged. Fallback: none needed — the seam moved, not the contract.
- **Card-citation correction: the fragment-inlining seam.** The card's line cites for the fragment inlining and the `type_condition` marker are stale; the live primitive is [`selections.py::included_field_selections`][selections] (inlined from [`walker.py::_walk_selections`][walker]) with the unknown-name `continue` guard in the same function and the marker uses in [`selections.py`][selections]. Preferred answer: ground G3 on the live `selections.py` primitive. Fallback: none needed — the [`DONE-033-0.0.9`][kanban] connection work moved the inlining into the shared `selections.py` substrate; the gap (unconditional inlining) is unchanged.
- **Upstream parity is a behavior contract, not a line contract.** The 2026-06-11 audit recorded specific [`strawberry_django/optimizer.py`][upstream-optimizer] locations as evidence; this spec treats the *behavior* each names — the resolve-hook `_result_cache is None` guard, the `enable_only and operation == QUERY` gate, the `get_possible_concrete_types` per-type re-walk — as the parity contract, referenced by the stable upstream permalink rather than a checkout line number (which drifts with every upstream release). The behavior descriptions in the [parity checkpoint](#reference-package-parity-checkpoint) and the [Borrowing posture](#borrowing-posture) are the contract; the audit's line evidence is archival. Fallback: none needed.
- **No new module / no settings key.** The guards are edits to three existing optimizer modules; no `permissions.py`-style new module and no `DJANGO_STRAWBERRY_FRAMEWORK` entry. Preferred answer: keep it that way. Fallback: none anticipated.
