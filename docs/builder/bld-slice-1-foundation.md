## Plan (Worker 1)

### Scope summary (orientation, non-binding)

Slice 1 lands the four foundation modules of the filter subpackage on disk — `__init__.py`, `base.py`, `sets.py`, and `inputs.py` — plus three package-internal test files (`tests/filters/__init__.py`, `tests/filters/test_base.py`, `tests/filters/test_sets.py`). `factories.py` exists as a stub-only file from the maintainer's pre-build TODO skeleton; Slice 1 leaves its body unchanged. The slice owns:

- Layers 1, 2, 3, 4 of the spec's six-layer pipeline (Decision 3): `RelatedFilter` + `LazyRelatedClassMixin` in `base.py`; `FilterSetMetaclass` + `FilterSet` (subclassing `django_filters.filterset.BaseFilterSet`) in `sets.py`; cycle-safe `get_filters()` expansion with `_expanded_filters` cache and `_is_expanding_filters` guard.
- The five parity-floor `graphene-django` primitive ports from Decision 4: `Filter`, `TypedFilter`, `ArrayFilter` + `ArrayFilterMethod`, `RangeFilter` + `RangeField` + `validate_range`, `ListFilter` + `ListFilterMethod`, `GlobalIDFilter` + `GlobalIDMultipleChoiceFilter`. All in `base.py`.
- The `FILTER_DEFAULTS` class-attribute + `filter_for_field` / `filter_for_lookup` overrides on `FilterSet` (Decision 4 — the owner-aware Relay-vs-scalar conditional with a fallback to `registry.primary_for(target_model)` when `_owner_definition` is still `None`).
- The `apply_sync` / `apply_async` / `apply` (thin dispatcher) classmethod trio on `FilterSet` plus the five named internal helpers (`_normalize_input`, `_derive_related_visibility_querysets_sync` / `_async`, `_run_permission_checks`, `_validate_form_or_raise`, `_apply_related_constraints`) per Decision 8 / [Decision 11](docs/spec-021-filters-0_0_8.md) + M1 of rev5 (decomposition discipline).
- The `_owner_definition: DjangoTypeDefinition | None` slot on `FilterSet` (per H4 of rev4) — declared with a default of `None`; the binding ITSELF lands in Slice 3 finalizer phase 2.5. Slice 1 only owns the slot's declaration and the `filter_for_field` / `filter_for_lookup` fallback when the slot is still `None`.
- Skeleton seams for `inputs.py`: the module docstring + `LOOKUP_PREFIXES` constant + `LOOKUP_NAME_MAP` constant + `FieldSpec` dataclass + the `_materialized_names` / `_field_specs` module-level dicts. The full converter pair (`convert_filter_to_input_annotation`, `normalize_input_value`) and the `_build_input_fields` / `_build_logic_fields` builders land in Slice 2; the `materialize_input_class` / `clear_filter_input_namespace` helpers land in Slice 3 with the finalizer wiring. Slice 1 ships only the symbols Slice 1 itself needs (the `apply_sync` pipeline consults `LOOKUP_NAME_MAP` via `_normalize_input`'s name-mapping table) AND the symbols other modules import as forward references so Slice 2 / 3 can extend without circular-import drama.
- `_helper_referenced_filtersets: set[type[FilterSet]]` module-level set in `django_strawberry_framework/filters/__init__.py`. The `filter_input_type(...)` consumer helper that records into it lands in Slice 2 (or wherever `inputs.py` grows the helper itself); Slice 1 only owns the set's existence so `registry.clear()` (touched in Slice 3) and `inputs.py` (Slice 2) can both reference it under their own slice's mandate without retrofit.

- **Duplication risk avoided.**
  - The Decision-4 Relay-vs-scalar conditional could naively be encoded TWICE — once at the factory level (Slice 2's `convert_filter_to_input_annotation` choosing the input shape) and once at the runtime level (Slice 1's `FilterSet.filter_for_field` / `filter_for_lookup` overrides picking the actual `django-filter` instance). Decision 4's H1-of-rev3 correction explicitly forbids this: the runtime override is the source of truth, and the factory must derive shape FROM the resolved filter instance. Slice 1 codifies this by placing the conditional ONLY in `FilterSet.filter_for_field` / `filter_for_lookup`, and the spec's Slice-2 plan explicitly forbids parallel logic in the factory. Slice 1's responsibility is to land the runtime-side conditional in the single source-of-truth shape.
  - The cookbook's `BaseRelatedFilter` + `RelatedFilter(BaseRelatedFilter, ModelChoiceFilter)` two-class split could be ported verbatim, giving the package a `BaseRelatedFilter` class consumers might mistake for a public surface. Slice 1 collapses to a single `RelatedFilter` class that mixes in `ModelChoiceFilter` directly, keeping the consumer-facing single-symbol shape Decision 2 promises. The cookbook's two-class shape was an artifact of Graphene's inheritance ergonomics; the package does not need it.
  - The `_apply_get_queryset_sync` / `_apply_get_queryset_async` helpers already exist in `relay.py`. The naive shape of `FilterSet._derive_related_visibility_querysets_sync` / `_async` would re-implement the coroutine-vs-queryset dispatch. Slice 1 explicitly delegates to the existing helpers (the TODO anchor in `relay.py` lines 244-251 invites this) — no parallel sync/async branching machinery in `sets.py`.
  - The sync-misuse error string could exist in two places (the raise inside `_apply_get_queryset_sync` and the catch-and-rethrow inside `FilterSet.apply`). Slice 1 lifts a `_SYNC_MISUSE_SENTINEL` module constant so both sites reference one value. The constant is `"get_queryset returned a coroutine in a sync resolver context"` (the substring matched against `exc.args[0]`); the rethrow site uses the constant as a substring-test, not an equality-test, so existing `ConfigurationError` messages continue to work even if their prose drifts slightly.
  - The Decision-3 `_get_fields` override for `Meta.fields = "__all__"` (M3 of rev4 — package INCLUDES PK / FK, EXCLUDES M2M) could land in Slice 1 or Slice 2. The spec's Slice-1 nested sub-bullets explicitly name `Meta.fields` handling on `FilterSet`, so Slice 1 owns the override (in `sets.py`). The `tests/filters/test_sets.py` tests for it land in Slice 1 too. Tests for the OUTPUT shape (whether the generated input type actually has an `id` field for `"__all__"` AND lacks an M2M field) land in Slice 2's `tests/filters/test_factories.py` because the factory is the OUTPUT producer — Slice 1 tests only the `_get_fields` override's behavior (what fields it returns).

### Implementation notes

- **`_run_permission_checks` calls per-field gates via a bare instance (`object.__new__(cls)`)** instead of the planned `getattr(cls, method_name)(request)` shape. The plan's shape mis-matches the cookbook contract: consumer `check_<field>_permission(self, request)` methods are regular instance methods, so `getattr(cls, name)(request)` raises `TypeError: missing 1 required positional argument: 'self'`. The bare-instance form mirrors the cookbook's `object.__new__(target_class)` precedent (`django_graphene_filters/filterset.py::check_permissions #"object.__new__(target_class)"`); a `_invoke_permission_method(bare, field_path, request)` staticmethod centralizes the call and de-duplicates the same-class vs nested-relation lookup.
- **`FilterSet.get_fields` (no underscore prefix), not `_get_fields`.** The plan named `_get_fields` per a misreading of the cookbook; `django_filters.filterset.BaseFilterSet`'s actual hook is `get_fields` (classmethod). The override now matches the upstream name so `super().get_filters()`'s internal call routes through the package's narrowing. Tests were updated to call `get_fields()` (without underscore). Recorded as small drift; the narrowing behavior is identical to the plan's intent.
- **`_resolve_relation_target_type` uses `registry.primary_for(...) or registry.get(...)`.** The plan listed `registry.primary_for(target_model)` as the fallback; in practice, a `DjangoType` declared without `Meta.primary = True` only lands in `_types`, not `_primaries`, so `primary_for(...)` returns `None` for the common single-type-no-primary case. Falling through to `registry.get(...)` covers the single-type case the spec's `registry.get()` docstring describes. Until Slice 3 binds `_owner_definition.related_target_for(...)`, this fallback is the only reachable branch through Slice-1 tests.
- **`GlobalIDFilter` `field_class` inherits the upstream default**: I did NOT set `field_class = forms.CharField` explicitly; `Filter.field_class = forms.Field` is the upstream default and the decode step in `GlobalIDFilter.filter()` runs before the form-field's `clean` would touch the value. The TODO anchor in the class body names Slice 3 as the wire-up site for the type-name validation.
- **`TypedFilter` is empty (no `__init__` override)** so passing `input_type=` does NOT raise — it falls into the `**kwargs` slot on `django_filters.Filter.__init__` (stored as `self.extra`). The mock queryset shape from the plan would have raised; the MRO-walk spies on the parent class's `filter` for one assertion and restores it in `finally`.
