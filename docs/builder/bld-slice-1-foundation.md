## Plan (Worker 1)

### DRY analysis

The Slice 1 work is intentionally a class-substitution mirror of the shipped Filtering subsystem; the DRY question is which pieces port verbatim, which get re-imported through the neutral home, and which are genuinely new. Every cite below uses the symbol-qualified `path::Symbol` form per `AGENTS.md` #"Source references in docs and code comments" ‚Äî these are standing-doc references, not per-cycle scratchpad refs.

- **Existing patterns reused (verbatim sibling-import, no copy).**
  - `django_strawberry_framework/sets_mixins.py::LazyRelatedClassMixin` ‚Äî the cookbook's Layer-2 resolution mixin. `orders/base.py::RelatedOrder` inherits from it via `from ..sets_mixins import LazyRelatedClassMixin` (NOT from `filters/base.py`, per Spec Revision 4 H1 and the build-plan's "Shared mixins live in the neutral module" build-wide flag). The shipped `django_strawberry_framework/filters/base.py::RelatedFilter` already uses this exact import shape (`from ..sets_mixins import LazyRelatedClassMixin`), so the order side mirrors that import line one-for-one.
  - `django_strawberry_framework/sets_mixins.py::ClassBasedTypeNameMixin` ‚Äî supplies the `{cls.__name__}InputType` naming rule shared across the set family. `orders/sets.py::OrderSet` inherits this mixin (same shape `django_strawberry_framework/filters/sets.py::FilterSet` uses at the `class FilterSet(ClassBasedTypeNameMixin, filterset.BaseFilterSet, metaclass=FilterSetMetaclass)` declaration head). `OrderSet` does NOT override `_field_type_suffix` ‚Äî the cookbook's upstream `django_graphene_filters/orderset.py::AdvancedOrderSet` keeps the default `"InputType"` for both root and per-field suffixes (verbatim from upstream lines 57-58). This is the divergence from `FilterSet`, which overrides `_field_type_suffix = "FilterInputType"`. The default keeps `BookOrder` ‚Üí `BookOrderInputType` (root) and any future per-field-bag use ‚Üí `BookOrderShelfInputType` (consistent with the cookbook).
  - `django_strawberry_framework/exceptions.py::ConfigurationError` ‚Äî raised by `_input_type_name_for` when a field path collapses to empty (via the `ClassBasedTypeNameMixin.type_name_for` guard) and by `materialize_input_class` (Slice 3 wiring will land the latter call site, but the Slice 1 namespace plumbing already imports the exception so the contract is pinned).
  - `django_strawberry_framework/filters/inputs.py::INPUTS_MODULE_PATH` shape ‚Äî the order side's `INPUTS_MODULE_PATH` is its own module string `"django_strawberry_framework.orders.inputs"`, but the symbol name and the use-pattern (single hoisted constant referenced by every `strawberry.lazy(...)` call site AND by `materialize_input_class`'s `sys.modules[INPUTS_MODULE_PATH]` lookup) is copied verbatim from `django_strawberry_framework/filters/inputs.py::INPUTS_MODULE_PATH`. Per Spec Decision 2, the constant is hoisted at the top of `orders/inputs.py`.
  - `django_strawberry_framework/filters/inputs.py::_input_type_name_for` shape ‚Äî the order side's `_input_type_name_for` delegates to `orderset_class.type_name_for()` (the inherited `ClassBasedTypeNameMixin` method); same one-line implementation as the filter side. Reusing the mixin keeps both subsystems pinned to one naming derivation.
  - `django_strawberry_framework/filters/inputs.py::_materialized_names` shape ‚Äî Slice 1 ships `_materialized_names: dict[str, type]` at module scope of `orders/inputs.py` per Spec Decision 9 M1 of rev3 (the value is the materialized **input class**, NOT the source `OrderSet`; source-class collision tracking lives in Slice 2's `OrderArgumentsFactory._type_orderset_registry`). Same key-shape and lifetime contract as the filter side's ledger ‚Äî different module, disjoint namespace per Spec Decision 9 line 703.
  - `django_strawberry_framework/filters/inputs.py::materialize_input_class` shape ‚Äî Slice 1 ships the order-side `materialize_input_class(name: str, input_cls: type) -> None` body modelled on the filter side's idempotent `(name, cls)` check + `ConfigurationError` on collision + `sys.modules[INPUTS_MODULE_PATH]` setattr. The function is **shipped in Slice 1 but called only by Slice 3's `_bind_ordersets()` materialize subpass**; Slice 1 lands the namespace plumbing so Slice 2's `OrderArgumentsFactory` and Slice 3's finalizer wiring can both depend on a stable, tested helper.
  - `django_graphene_filters/orders.py::BaseRelatedOrder` (cookbook source ‚Äî `~/projects/django-graphene-filters/django_graphene_filters/orders.py` lines 8-29) ‚Äî ported to `orders/base.py::RelatedOrder` per Spec Decision 2 with `BaseRelatedOrder` + `RelatedOrder` collapsed into a single consumer-facing class (mirroring the filter side's Decision-2 single-symbol collapse at `django_strawberry_framework/filters/base.py::RelatedFilter`). `__init__`, `bind_orderset`, `.orderset` property + setter ‚Äî all body-equivalent to upstream lines 11-28.
  - `django_graphene_filters/orderset.py::OrderSetMetaclass` (cookbook source ‚Äî `~/projects/django-graphene-filters/django_graphene_filters/orderset.py` lines 14-43) ‚Äî ported to `orders/sets.py::OrderSetMetaclass`. The upstream metaclass's inherited-then-overridden `related_orders` OrderedDict shape is preserved verbatim (cookbook lines 30-40); the only adaptation is replacing the cookbook's `orders.BaseRelatedOrder` import with the local `from .base import RelatedOrder` (Spec Decision 2 collapses `BaseRelatedOrder` and `RelatedOrder` into the one shipped class).
  - `django_graphene_filters/orderset.py::AdvancedOrderSet` (cookbook lines 46-58 for the class head; the body is Slice 2's domain) ‚Äî ported to `orders/sets.py::OrderSet`. Slice 1 ships the class head + the `_owner_definition` slot + `_expanded_fields` / `_is_expanding_fields` cache slots + the `Meta` validation + the cookbook's `get_fields` (Layer 4 expansion, cookbook lines 265-285) only. The cookbook's `__init__`-driven application (lines 60-84), the `check_permissions` body (lines 86-113), the `get_flat_orders` body (lines 115-170), and the `apply_distinct` branches (lines 172-263) ALL move to Slice 2 or live as no-ops in Slice 1 because `OrderArgumentsFactory` (the BFS consumer of `get_fields()`) lands in Slice 2. Slice 1 ships only the class slots and the `Meta`-driven primitive shape so Slice 2's BFS has a target to consume.

- **New helpers justified (none for Slice 1).** Slice 1 is the foundation pass ‚Äî every helper named in the Slice 1 contract is either a port from the cookbook or a sibling-import from the neutral `sets_mixins.py`. The genuinely new modules (`orders/factories.py::OrderArgumentsFactory`, `orders/inputs.py::convert_order_field_to_input_annotation`, `orders/inputs.py::normalize_input_value`) all belong to Slice 2 per the spec's Slice 2 checklist. Slice 1 ships zero net-new shared helpers across the package.

- **Duplication risk avoided (three risks named explicitly).**
  - **Risk: copy-paste `LazyRelatedClassMixin` into `orders/base.py`.** Naive implementation would replicate the resolution algorithm in two places, silently bifurcating behaviour the first time a maintainer fixes one and forgets the other. **Avoidance:** Slice 1 imports `LazyRelatedClassMixin` from `..sets_mixins` (the explicit build-plan rule); Worker 3 should flag any `class LazyRelatedClassMixin` definition inside `orders/base.py` as a High finding. Spec Revision 4 H1 already retired the alternative; the rejected-alternative prose at Spec Decision 2's third alternative is the historical breadcrumb.
  - **Risk: re-deriving `<Name>OrderInputType` inline at every call site.** The filter side hoisted `_input_type_name_for` to one helper specifically to dodge this exact bifurcation (filter spec-027 M3 of rev1). **Avoidance:** Slice 1 ships `orders/inputs.py::_input_type_name_for(orderset_class)` as a one-line `orderset_class.type_name_for()` delegate at module scope, with the same call surface as the filter side's helper. Slice 2's `OrderArgumentsFactory` and Slice 3's `order_input_type` consumer helper both call it.
  - **Risk: `_helper_referenced_ordersets` migrating to `orders/inputs.py` for "convenience" with the namespace plumbing.** Per Spec Decision 2 M9 of rev1 AND N-new-2 of rev2, the ledger lives in `orders/__init__.py` co-located with its only writer (`order_input_type`, which is the public helper that hosts the writer). Putting it in `inputs.py` would force the import-cycle dance the filter side already designed against. **Avoidance:** Slice 1 does NOT ship `_helper_referenced_ordersets` in `inputs.py`; the ledger is **planned for** `orders/__init__.py` but the writer (`order_input_type`) does not exist until Slice 2 (per Spec line 850-853 ‚Äî `order_input_type` is part of the Slice 2 consumer-helper surface; Slice 1 ships only the re-exports the helper will need, namely the namespace plumbing in `inputs.py`). For Slice 1, `orders/__init__.py` carries only the `__all__` skeleton + re-exports of `RelatedOrder`, `OrderSet`, `OrderSetMetaclass`, `Ordering`. The ledger is added to `__init__.py` in Slice 2 alongside `order_input_type`. Worker 3 should NOT flag the ledger's absence in Slice 1.
