Spec reference: `docs/spec-036-mutations-0_0_11.md` (Slice 1 checklist lines 48-52; Decision 6 lines 325-344; Decision 7 lines 346-364; Decision 11 lines 418-427 for primary-type / relation-id strategy; Decision 12 lines 429-438 for the phase-2.5 materialize discipline; Edge cases lines 504, 509; Test plan lines 519-520; staged-seam discipline line 496)
Status: final-accepted

## Plan (Worker 1)

This slice ships **`django_strawberry_framework/mutations/inputs.py`** (new) + **`django_strawberry_framework/mutations/__init__.py`** (new) and adds the `FieldError` export to the package root. It is the generation substrate: pure functions + dataclasses that, given a Django model + an operation kind + an effective field set + the resolved primary `DjangoType`, produce `<Model>Input` / `<Model>PartialInput` `@strawberry.input` classes, the public `FieldError` `@strawberry.type`, and a generated `<Name>Payload` `@strawberry.type`. **No metaclass, no resolver, no finalizer wiring lands here** — those are Slice 2 (`sets.py` + the phase-2.5 bind) and Slice 3 (`resolvers.py` + `fields.py`). This slice's functions are *callable in isolation and unit-tested in isolation*; Slice 2 calls them from the bind.

Critical scoping note for Worker 2: the spec's Slice-1 checklist references "materialized as module globals so identical shapes dedupe" and "finalize-time `ConfigurationError`". The **materialization call site and the finalize-time collision raise are Slice 2's bind** (Decision 12, finalizer phase 2.5). Slice 1 supplies (a) the generators, (b) the shape-identity / name-derivation logic, and (c) a `materialize_*` + `clear_*_namespace` pair mirroring `orders/inputs.py` so Slice 2 only has to *call* them. Build the collision-detecting `materialize_mutation_input_class` here (it is the shared-substrate `materialize_generated_input_class` wrapper — see DRY analysis); but do **not** wire it into the finalizer or `registry.clear()` in this slice (those TODO anchors at `registry.py #"co-clear mutation generated-input globals"` and `finalizer.py #"bind ``DjangoMutation`` declarations"` are Slice 2's). Ticking the checklist boxes that depend on the *finalize-time* raise is therefore shared with Slice 2 — see the per-box deferral notes in `### Spec slice checklist (verbatim)`.

### DRY analysis

- **Existing patterns reused (cite file:line).**
  - **The whole generated-input substrate** lives in `django_strawberry_framework/utils/inputs.py` and is the single source of truth for materialize / build / camel-name / clear mechanics. Reuse verbatim, do not re-derive:
    - `utils/inputs.py::build_strawberry_input_class` (`utils/inputs.py` lines 66-100) — constructs a `@strawberry.input`-decorated dataclass from `(python_attr, annotation, field_kwargs)` triples, preserving `strawberry.field(name=..., default=..., description=...)` metadata. Use this to build **both** `<Model>Input` and `<Model>PartialInput`. The `default=` slot carries `strawberry.UNSET` for optional fields; the `name=` kwarg carries the camelCase GraphQL alias (e.g. `category_id` → `categoryId`).
    - `utils/inputs.py::materialize_generated_input_class` (lines 103-139) — pins a class as a module global of a given module path under a ledger, idempotent on `(name, cls)`, raising `ConfigurationError` (naming both qualified class names + the family label) on a *distinct-class* collision under one name. This **is** the AR-M6 finalize-time collision raise (the spec's "two distinct shapes colliding on one name raise `ConfigurationError`"). Slice 1 wraps it as `materialize_mutation_input_class(name, cls)` with `module_path="django_strawberry_framework.mutations.inputs"`, `family_label="DjangoMutation"`, `ledger=_materialized_names` — byte-mirroring `orders/inputs.py::materialize_input_class` (lines 323-339).
    - `utils/inputs.py::graphql_camel_name` (lines 53-63) — `category_id` → `categoryId`. Use for every input/payload field's GraphQL `name=` alias. (Note: `materialize_*` uses raw class names; only field aliases camel-case.)
    - `utils/inputs.py::clear_generated_input_namespace` (lines 215-275) — the `registry.clear()` reset path. Slice 1 wraps it as `clear_mutation_input_namespace()` mirroring `orders/inputs.py::clear_order_input_namespace` (lines 342-391). **But:** the order/filter version resets per-set `_lifecycle` binding-state via `set_root._lifecycle.binding_attrs` and a `factory_cls` — mutations have **no** `GeneratedInputArgumentsFactory` subclass (no BFS over related sets; input fields come from one model's columns, not a related-set graph) and **no** `SetLifecycleAttrs` in Slice 1. So Slice 1's `clear_mutation_input_namespace` clears only the two module-level ledgers it owns (`_materialized_names`, `_field_specs` if used) directly — it does **not** call `clear_generated_input_namespace` (which assumes a factory + a set-base `_lifecycle`). Reuse the *pattern* (best-effort, ledger-only) not the helper. Record this divergence under `### Implementation discretion items`.
  - **The read-side scalar / relation converters** in `django_strawberry_framework/types/converters.py` (the symmetric-wire-contract requirement, Decision 6 "Map scalars through the **same** … converters the read side uses"):
    - `types/converters.py::scalar_for_field` (lines 119-139) — Django-field-class → Python/Strawberry scalar via MRO walk; raises `ConfigurationError` for an unsupported field. This is **already** the shared lookup `filters/inputs.py::_scalar_from_model_field` (lines 253-269) delegates to. Use it for the scalar input field type and for the **raw-pk target type** of a non-Relay FK (`scalar_for_field(target_model._meta.pk)`).
    - `types/converters.py::convert_scalar` (lines 142-265) — full scalar conversion incl. choice-enum substitution + null widening via the `force_nullable` tri-state. The input generator needs the **base scalar/enum** for a column but its **own** required/optional rule (Decision 6's `default`/`blank`/`null` rule, NOT `field.null`-only widening). So: call `convert_scalar(field, type_name, force_nullable=False)` to get the **non-nullable base type** (the enum substitution + array/hstore handling for free), then the generator decides `T` vs `T | None` from the create-required rule by widening to `… | None` itself when optional. Confirm with Worker 2 that `force_nullable=False` is the right primitive (it suppresses `field.null` widening so the generator owns nullability) — this is the documented use of the tri-state (`converters.py::convert_scalar` docstring #"force nullable" / #"force required").
    - `types/converters.py::convert_choices_to_enum` (lines 301-378) is reached transitively through `convert_scalar` when `field.choices` is set — no separate call needed; choice columns become the same generated `<TypeName><Field>Enum` the read side emits (registry-cached by `(model, field_name)`, so read and write share the one enum instance).
  - **`relay.GlobalID` as the FK input type** for a Relay-Node target: the filter side already uses `relay.GlobalID` as the wire input for a GlobalID-shaped field (`filters/inputs.py` lines 446-486). The mutation FK input mirrors this — `relay.GlobalID` when the target's **primary** `DjangoType` is Relay-Node-shaped, else the target pk's raw scalar.
  - **`registry.get(model)` / `registry.primary_for(model)`** (`registry.py::TypeRegistry.get` lines 221-240; `primary_for` lines 268-275) — resolve the model's primary type to (a) pick the payload object type and (b) fix the relation-id strategy (Decision 11). In Slice 1 the generator takes the resolved primary type **as an argument** (the bind in Slice 2 does the lookup); Slice 1 does not import the finalizer. For the *relation target's* id type the generator resolves the **related** model's primary (`registry.get(related_model)`) to decide GlobalID-vs-raw-pk — this read is internal to the generator and safe (called from Slice 2's phase-2.5 bind, after the registry is fully populated).
  - **`implements_relay_node(type_cls)`** (`types/relay.py::implements_relay_node` line 52) — the predicate for "is this primary type Relay-Node-shaped" (drives `node`-vs-`result` payload slot and GlobalID-vs-raw-pk relation id). Reuse; do not re-derive a Relay check.
  - **`utils/strings.py::pascal_case` / `snake_case`** (lines 55, 22) — name derivation. `pascal_case(model._meta.object_name)` is already the model's class name; use `model.__name__` directly for the canonical `<Model>Input`. Use `pascal_case` for the shape-derived suffix tokens.
  - **`ConfigurationError`** (`django_strawberry_framework/exceptions.py`) — every fail-loud raise (unsupported field reaching the generator, duplicate generated name, the `Upload`-seam `NotImplementedError` is the one exception that is `NotImplementedError`, see staged seam below).
- **New helpers justified (single responsibility each).**
  - `mutations/inputs.py::editable_input_fields(model, *, fields=None, exclude=None) -> list[<django field>]` — **single responsibility: the Decision-6 / Medium-4 editable-field selection.** Returns the model's editable, settable columns: excludes the auto pk, every `editable=False` column (which covers `auto_now` / `auto_now_add`), and reverse relations; includes forward FK / OneToOne (their `<field>_id` column is on the model's own table) and M2M; then narrows by the mutation's own `fields` / `exclude`. This is **not** `orders/inputs.py::_get_concrete_field_names_for_order` (which *includes* read-only timestamps and *excludes* M2M — the opposite of what writes need), so it is a genuinely new selector, but it borrows that helper's `model._meta.get_fields()` + `hasattr(f, "column")` idiom and its documented M2M caveat. Single call site: the two input generators.
  - `mutations/inputs.py::input_field_required(field) -> bool` — **single responsibility: the Major-1 create-required rule.** `True` only when the field has no usable default: no Django `default` (`field.has_default()` is `False`), `null=False`, and (for text/char fields, i.e. `field.blank` is meaningful) `blank=False`. Used only by the create-input generator; the partial-input generator forces every field optional.
  - `mutations/inputs.py::relation_input_annotation(field, *, related_primary_type) -> (annotation, python_attr, graphql_name)` — **single responsibility: FK/O2O → `<field>_id` and M2M → `list[<id>]` shaping (Decision 6).** `<field>_id` typed `relay.GlobalID` when `related_primary_type` is Relay-Node, else `scalar_for_field(related_model._meta.pk)`; M2M typed `list[<that id type>]`. The python attr is `<field>_id` (FK/O2O) so the resolver maps it back with no per-field mapping declaration (Decision 6 "custom inputs follow the generated field-naming scheme", AR-M2).
  - `mutations/inputs.py::build_mutation_input(model, *, operation_kind, primary_type, fields=None, exclude=None) -> type` — **single responsibility: assemble the triples and call `build_strawberry_input_class`.** `operation_kind ∈ {"create", "partial"}` ("partial" forces all-optional). Resolves the generated **name** via the shape-identity logic below, then builds. Returns the `@strawberry.input` class (unmaterialized — Slice 2 materializes).
  - `mutations/inputs.py::mutation_input_type_name(model, operation_kind, effective_field_names) -> str` — **single responsibility: AR-H1 / AR-M6 shape-derived naming.** Returns the stable `<Model>Input` / `<Model>PartialInput` when `effective_field_names` is the canonical full editable set; else a deterministic shape-derived name (model name + an operation/field-set-derived suffix, e.g. a sorted-field-name hash or a sorted-token join — Worker 2's discretion on the exact suffix shape, see discretion items). Identity is `(model, operation_kind, frozenset(effective_field_names))` (Decision 6). Two narrowings to the *same* effective shape return the *same* name → dedupe via the materialize ledger.
  - `mutations/inputs.py::FieldError` — the public `@strawberry.type` (`field: str`, `messages: list[str]`). Plain class definition; the one symbol this slice exports.
  - `mutations/inputs.py::build_payload_type(mutation_name, *, object_type, object_slot) -> type` — **single responsibility: the `<Name>Payload` `@strawberry.type` (Decision 7).** `object_slot ∈ {"node", "result"}` decided by `implements_relay_node(primary_type)` (`node` for Relay-Node, `result` otherwise — AR-H5, NEVER model-derived). Fields: `<object_slot>: object_type | None` (nullable) + `errors: list[FieldError]` (the spec/glossary write `list[FieldError]!` non-null-list — that is `list[FieldError]` in Strawberry annotation; the list itself is non-null, elements non-null). The object type is a **forward reference** via `strawberry.lazy` / `Annotated` so it resolves at schema build (the read-side `DjangoType` is a real class by phase 2.5, so a direct `object_type | None` annotation is also viable — Worker 2's discretion, see discretion items; the spec only mandates the *field* timing for the `DjangoMutationField` resolver in Slice 3, not the payload's own object slot).
  - `mutations/inputs.py::INPUTS_MODULE_PATH = "django_strawberry_framework.mutations.inputs"` — the single module-path constant the `strawberry.lazy(...)` references and `materialize_mutation_input_class` share (mirror `orders/inputs.py::INPUTS_MODULE_PATH` line 60).
  - `mutations/inputs.py::materialize_mutation_input_class(name, cls)` + `mutations/inputs.py::clear_mutation_input_namespace()` + the `_materialized_names: dict[str, type]` ledger — mirror `orders/inputs.py` (lines 128, 323, 342). **Defined here, called by Slice 2.**
- **Duplication risk avoided.**
  - **Risk: re-implementing materialize / camel-name / collision logic inline.** Avoided by wrapping `utils/inputs.py::materialize_generated_input_class` + `graphql_camel_name` + `build_strawberry_input_class` exactly as `orders/inputs.py` does — zero new copies of the materialize ledger or the BFS-collision check.
  - **Risk: forking the scalar→type mapping** so read and write resolve a column to different scalars. Avoided by routing every scalar input field through `convert_scalar(..., force_nullable=False)` (same converter table, same choice-enum cache) and every FK target-pk through `scalar_for_field` — the identical lookup `filters/inputs.py::_scalar_from_model_field` delegates to. The wire contract is symmetric by construction.
  - **Risk: a second editable-field selector drifting from `orders`' concrete-field selector.** They are deliberately different (writes exclude read-only timestamps, include M2M; orders include timestamps, exclude M2M) — the new `editable_input_fields` is documented as the write-side counterpart so a reader does not assume they should converge. Not a duplication; a justified divergence.
  - **Risk: hand-rolling a Relay-node check** instead of `implements_relay_node`. Avoided — reuse the predicate.

### Test additions / updates

All package-internal, in **`tests/mutations/test_inputs.py`** (new file; `tests/mutations/__init__.py` must exist — Worker 2 creates the package). Per AGENTS.md these are package tests (system-under-test is the generator itself, run against real example-project models as fixtures — products `Item` / `Category` are the realistic FK + `UniqueConstraint` fixtures; a package-internal model fixture is needed for **M2M** and for **non-Relay** target shapes since products has no M2M and every products primary type is Relay-Node). Worker 2 may use an existing tests/ model fixture or a minimal local one; note the choice for Worker 3. These are NOT live `/graphql/` tests — the generator is class-definition-time machinery not reachable from a query in Slice 1 (the live surface is Slice 4). Pin shapes:

- **Required/optional create shape (Major-1).** `build_mutation_input(Item, operation_kind="create", primary_type=ItemType)` → `name` required (`str`, no `| None`, no `UNSET`), `category_id` required (`relay.GlobalID`), `description` optional (`str | None`, default `UNSET` — `blank=True, default=""`), `is_private` optional (`bool | None`, default `UNSET` — `default=False`). Assert via the built class's `__annotations__` / strawberry definition fields.
- **Partial all-optional shape.** `operation_kind="partial"` → every field optional + `UNSET` default, including `name` / `category_id`.
- **FK/O2O → `<field>_id`; Relay-vs-non-Relay id type.** FK on a Relay-Node primary target → `category_id: relay.GlobalID`; the same FK pointed at a non-Relay primary fixture → `category_id: <pk scalar>` (e.g. `int` for an `AutoField` pk, `BigInt`/`uuid.UUID` as the fixture dictates). Pin both.
- **M2M → `list[<id>]`.** A package fixture model with an M2M field → `<m2m>: list[relay.GlobalID]` (Relay target) / `list[<pk scalar>]` (non-Relay). (Products has no M2M, so this is package-fixture-only per Edge cases line 505.)
- **pk / auto-field / editable=False exclusion.** `id`, `created_date` (`auto_now_add`, `editable=False`), `updated_date` (`auto_now`, `editable=False`), and reverse relations (e.g. `Category.items`) are **absent** from the generated input.
- **Canonical full shape → stable `<Model>Input` / `<Model>PartialInput` name (AR-H1).** Two create generators over `Item` with the full editable set → identical `mutation_input_type_name` → dedupe to one materialized class via `materialize_mutation_input_class` (second materialize is a no-op, no raise).
- **Narrowed shape → distinct shape-derived name; two distinct shapes colliding → `ConfigurationError` (AR-H1 / AR-M6).** A `Meta.exclude={"description"}` narrowing → a deterministic non-`ItemInput` name; two *distinct* shapes forced onto one name → `materialize_mutation_input_class` raises `ConfigurationError` naming both. (This pins the **collision raise mechanism** Slice 1 owns; the *finalize-time* trigger is exercised end-to-end in Slice 2's `test_sets.py`.)
- **`node` / `result` payload slot (AR-H5).** `build_payload_type("CreateItem", object_type=ItemType, object_slot="node")` → a `@strawberry.type` with `node: ItemType | None` + `errors: list[FieldError]`; with a non-Relay primary fixture → `result: <Type> | None`. Assert **no model-derived field name** (no `property`-named field for a `Property` payload — pin `build_payload_type` for `Property` emits `node`, never `property`).
- **`FieldError` shape.** `FieldError` is a `@strawberry.type` with `field: str` (non-null) + `messages: list[str]` (non-null list of non-null str). Pin via the strawberry definition.
- **Custom `input_class` follows the generated scheme (AR-M2) — generator seam.** A consumer override map is honored (the generated field is skipped for an overridden column) — pin the `build_mutation_input` skip-on-override seam directly (the `Meta.input_class` validation that field names must match the scheme is Slice 2; here, pin that the generator *can* be told to skip a field).
- **`Upload` staged seam.** A model with a `FileField` / `ImageField` reaching `build_mutation_input` raises `NotImplementedError` (NOT silently `str`) — pins the fail-loud 037 handoff. Use a package fixture model with a `FileField`.
- **Public export.** `from django_strawberry_framework import FieldError` resolves to the `mutations/inputs.py::FieldError` class; `"FieldError" in django_strawberry_framework.__all__`. (The `tests/base/test_init.py` `__all__` pin update is the cross-slice public-surface change flagged in the build plan — Worker 2 extends `tests/base/test_init.py` to include `FieldError` in the expected `__all__`, per build-plan context flag "Four net-new public symbols".)

**Temp/scratch tests:** none anticipated. The generator is deterministic and unit-testable directly; no scratch HTTP scaffolding needed in Slice 1. Worker 3 may add a focused temp test under `docs/builder/temp-tests/slice-1/` if it wants to probe the shape-derived-name determinism, but the permanent pins above cover the contract.

### Implementation discretion items

These are choices Worker 1 has assessed and decided belong to Worker 2 (equivalent-shape or naming-detail decisions, no architectural content):
