## Plan (Worker 1)

This slice is the plan-side foundation only: it teaches the walker to *recognize and plan* nested synthesized relation-connection selections as windowed prefetches, records the recognition metadata at finalization, consolidates the `edges { node }` helpers into the walker, hoists the deterministic-order rule, and makes `_publish_plan_to_context` union the correctness sentinels. It ships **no consumer** of the window — Slice 2 (`connection.py` fast path) consumes it. Slices 1+2 are one mechanism in two commits and must merge together (build-plan "Joint-cut mechanism gate"); Slice 1's planner is therefore harmless-but-unconsumed on its own commit only within the joint cut. The slice contributes a window prefetch under a `to_attr` that nothing reads until Slice 2, so on its own it adds a (currently un-consumed) window query for a nested connection — this is exactly why the two commits land together.

### DRY analysis

**Existing patterns reused (cite file::Symbol):**

- **Child-queryset construction.** `_plan_connection_relation` builds its windowed child queryset exactly as the list path already does: `optimizer/walker.py::_build_prefetch_child_queryset` (recursive `_walk_selections` over `sel.selections`, `_ensure_connector_only_fields`, `_merge_child_plan_metadata`, `child_plan.apply(...)`, `cacheable` propagation) and `optimizer/walker.py::_build_child_queryset` (the `related_model._default_manager.all()` + `target_type.get_queryset(queryset, info)` visibility bake). Worker 2 must **reuse these two helpers**, not re-derive the child queryset; the only addition over the list path is applying the window after `child_plan.apply(...)`. The `cacheable=False`-on-custom-`get_queryset` flip is already inside `_build_prefetch_child_queryset` (#"if not child_plan.cacheable") and `_plan_prefetch_relation` (#"plan.cacheable = False") — reuse, do not re-implement (DoD item 4 "visibility … flipping cacheable = False"; Edge case "Visibility-filtered targets").
- **Accessor vs field-name vocabulary.** `optimizer/utils.relations.instance_accessor(field)` is the lookup-segment source the list path already uses (`_plan_prefetch_relation` #"lookup_path = f\"{prefix}{instance_accessor(django_field)}\""). The window `Prefetch` lookup MUST use the accessor (`book_set`), while the `to_attr` and `relation_connections` value use the **field name** (`book`) — the divergence Decision 4/5 calls out. Reuse `instance_accessor` for the lookup; the field name comes from the `relation_connections` slot value.
- **Resolver-key vocabulary.** `optimizer/plans.py::resolver_key(parent_type, field_name, runtime_path)` + `runtime_paths`/`resolver_identities` construction in `_walk_selections` (#"resolver_identities = tuple(") and `_record_relation_access` (#"append_unique_many(plan.planned_resolver_keys, resolver_identities)"). `_plan_connection_relation` reuses this exact construction so strictness (Slice 4) sees the connection field under the same key shape list relations use (DoD item 4 "resolver keys recorded"; spec step (h)).
- **`edges { node }` unwrap.** The six helpers (`_named_children`, `_node_children_with_runtime_prefix`, `_with_runtime_prefix`, `_converted_selection_included`, `_is_converted_fragment`, `_response_key`) already exist in `optimizer/extension.py` (lines 320-389). Decision 9 moves them to `walker.py` so one implementation serves both the root seam and `_plan_connection_relation` — see "New helpers" below.
- **Deterministic total order.** `connection.py::_ends_in_unique_column` (lines 426-463) + the pk-append rule in `connection.py::_finalize_queryset` (#"qs = qs.order_by(*effective, target_model._meta.pk.attname)"). Decision 11 hoists this to `plans.py` so plan-time and resolve-time order share one source (the cursor-parity invariant).
- **Connector-column dispatch.** `optimizer/walker.py::_ensure_connector_only_fields` already branches per relation kind (`one_to_many`/`reverse_one_to_one` → `field.attname`/`reverse_connector_attname`; forward FK → `target_field.attname`/`target_field_attname`; M2M → `related_model._meta.pk.attname`). `plans.py::window_partition_for_prefetch(field)` is the **parent-side partition** sibling of this child-pk connector helper — same `FieldMeta`/field shape inputs, different output (the partition expression, not the `.only()` column). They are deliberately distinct (Decision 4 "not the existing child-pk connector helper"): do not merge them, but model the kind-dispatch structure on `_ensure_connector_only_fields`.
- **Slice arithmetic.** `strawberry.relay.utils.SliceMetadata.from_arguments(info, *, before, after, first, last, max_results, prefix)` → `(start, end, expected)`. Verified this pass against locked Strawberry `0.316.0`: it reads `info.schema.config.relay_max_results` itself when `max_results=None`, so the plan-time cap and the resolve-time cap come from the same config object **by construction** (Edge case "`relay_max_results` agreement"); it raises `ValueError` (negative/over-max `first`/`last`) and `TypeError` (malformed cursor). No hand-ported arithmetic (Decision 4 rejected alternative).
- **Union-read helper.** `optimizer/_context.py::get_context_value` is already importable; `_publish_plan_to_context` reuses it to read the existing stash before unioning (the anchor names it; do not add a new reader).

**New helpers justified (single responsibility each):**

1. `optimizer/plans.py::apply_window_pagination(queryset, *, partition_by, order_by, offset, limit, reverse=False)` — single responsibility: annotate `_dst_row_number` (`Window(RowNumber(), partition_by=partition_by, order_by=order_by)`) and `_dst_total_count` (`Window(Count(1), partition_by=partition_by)`), then filter to the requested row-number range (forward: `_dst_row_number__gt=offset` when `offset`, `_dst_row_number__lte=offset+limit` when `limit` is finite/`!= sys.maxsize`; reverse: annotate `_dst_row_number_reversed` with the reversed order and filter `__lte=limit`). Mechanism port of `strawberry-django-main/strawberry_django/pagination.py::apply_window_pagination` (lines 209-282), annotations namespaced `_dst_*` not upstream's `_strawberry_*` (Decision: "Explicitly do not borrow … `_strawberry_*`"). Call sites: `_plan_connection_relation` only (this card); future slices may reuse. Note signature divergence from upstream: this helper takes `partition_by` + `order_by` explicitly (the spec `plans.py` anchor pins this) rather than upstream's `related_field_id` + compiler-derived order — keeps `plans.py` free of queryset-compiler coupling and lets `window_partition_for_prefetch` own the partition derivation.
2. `optimizer/plans.py::window_partition_for_prefetch(field) -> str` — single responsibility: return the **parent-side partition expression** Django's prefetch attach uses, by relation kind. Reverse FK / reverse-one-to-one → child-table FK attname (`field.field.attname` / `FieldMeta.reverse_connector_attname`); reverse M2M → through the child's forward M2M field name; forward M2M → through the target's reverse query name (NOT always the accessor when `related_name` is absent — resolve via `field.remote_field` / the through model). Unsupported kind → raise a package-internal planning error so the caller leaves the selection unplanned (Decision 4 "raise a package-internal planning error and fall back unplanned rather than guessing" — reuse the existing `optimizer/exceptions` family; do NOT invent a new exception type unless none fits — Worker 2 discretion item below). Mirrors the kind-dispatch of `walker._ensure_connector_only_fields`.
3. `optimizer/plans.py` deterministic-order helper(s) — `_ends_in_unique_column` moved verbatim from `connection.py` plus a small public wrapper that performs the pk-append decision (the rule currently inline in `_finalize_queryset`). Single responsibility: produce the deterministic total ordering tuple a connection queryset must carry. `connection.py::_finalize_queryset` imports it back; `_plan_connection_relation` calls it for the window's `order_by`. This is the cursor-parity invariant — one source, two callers.
4. `optimizer/walker.py::_plan_connection_relation(...)` — single responsibility: plan one recognized nested connection as a windowed `Prefetch` (or leave it unplanned for a fallback shape). The orchestrator; delegates child-queryset build, partition, order, window, and resolver-key recording to the reused/new helpers above.
5. `optimizer/extension.py` (Slice 3 grows `_build_cache_key`, NOT this slice) — no new helper here in Slice 1 beyond the union edit; called out so Worker 2 does not pre-build Slice 3's pagination-variable collector.

The six consolidated `edges { node }` helpers are a **move**, not new helpers (Decision 9).

**Duplication risk avoided:**

- **`to_attr` string literal.** `_dst_<field>_connection` is constructed at three sites this card touches (walker plan, Slice-2 finalizer call-site precompute, Slice-2 resolver probe). The shadow scan flags `prefetch` repeated 3× in `walker.py` already. To prevent a fourth scattered f-string, the plan: derive the `to_attr` from one helper (e.g. a module-level `_relation_connection_to_attr(field_name) -> str` in `walker.py`, or a shared constant `_RELATION_CONNECTION_TO_ATTR_PREFIX = "_dst_"`+suffix). Worker 2 picks the exact shape (discretion item), but the literal must not be duplicated across walker/finalizer/connection. The `_dst_` prefix is the package-reserved namespace (consistent with `_dst_optimizer_*` context keys, `_dst_synthesized_relation_connection` marker).
- **Two `_response_key` implementations.** `walker.py::_response_key` (line 855) and `extension.py::_response_key` (line 387) are byte-identical duplicates today. Decision 9's consolidation MUST collapse to the one in `walker.py` (extension imports it back) — not leave two. The shadow Imports section will show `extension` importing from `walker` after the move; the integration pass cross-checks one-way dependency direction.
- **Re-deriving the child queryset.** Naive risk: `_plan_connection_relation` re-implements visibility-bake + connector-columns + child-plan recursion instead of reusing `_build_prefetch_child_queryset`. Avoided by reusing that helper and applying the window as a post-step (see Implementation step 6). If the helper's signature does not admit the window post-step cleanly, extract the shared body rather than copy it (Worker 2 note).
- **Deterministic order drifting between plan-time and resolve-time.** The entire fast-path↔fallback wire-parity promise (Decision 4 cursor-parity invariant) depends on the hoisted helper being the *only* implementation. `test_deterministic_order_helper_hoist_parity` (Slice 1) and `test_fast_path_cursor_round_trips_to_fallback_after` (Slice 2) guard it. Risk: Worker 2 copies the rule into `plans.py` and leaves the original in `connection.py` — MUST be a move + import-back, not a copy.
- **Partition vs connector confusion.** `window_partition_for_prefetch` (parent-side partition) and `_ensure_connector_only_fields` (child-side `.only()` column) look similar and must stay distinct (Decision 4). Pinned by `test_window_partition_for_reverse_fk_forward_m2m_reverse_m2m` (parent key) + `test_child_plan_projections_include_connector_and_ordering_columns` (child `.only()`).
- **Argument-aware merge as a second merge path.** Risk: Worker 2 forks `_merge_aliased_selections` into a connection-only copy. Avoided: extend the *existing* merge to preserve per-response-key argument payloads (or delay merging connection selections), so list and connection selections still flow through one merge.

### High:

#### `last`-only window never applies its row-count filter — over-fetches every child row

`optimizer/walker.py::_connection_window_slice` returns `limit = slice_meta.expected`. For a `last`-only window (`{ booksConnection(last: 2) }`), Strawberry's `SliceMetadata.from_arguments` sets `end = sys.maxsize` and therefore `expected is None` (verified against locked Strawberry `0.316.0` `strawberry/relay/utils.py::SliceMetadata.from_arguments`). So `_plan_connection_relation` calls `apply_window_pagination(..., limit=None, reverse=True)`, and the reverse branch's guard `if limit is not None and limit != sys.maxsize:` is **False** — the `_dst_row_number_reversed__lte=limit` filter is never added. The reversed window fetches **every** child row per parent instead of the last `N`. Temp test confirmed: a planned `booksConnection(last: 2)` queryset carries the `_dst_row_number_reversed` annotation but NO `<=` row filter (while a concrete `limit=2` correctly produces the `<=` filter).

This is the upstream-mechanism bug Decision 4 / the Edge case "`last`-only backward pagination" exists to prevent. Upstream (`strawberry-django-main/strawberry_django/optimizer.py::_optimize_prefetch_queryset`, the call site) computes `mark_reversed = slice_metadata.expected is None` and then passes `limit = field_kwargs.get("last")` (the actual `last` value) precisely **because** `expected` is `None` in the reversed case — it does NOT pass `expected`. Worker 2 mirrored upstream's `reverse` detection (`isinstance(last, int) and not isinstance(first, int) and before is None`, equivalent to `expected is None`) but passed `expected` as the limit, which is exactly the value upstream avoids in that branch.

Why it matters: wrong data. `last: N` resolves to an unbounded windowed prefetch; Slice 2's fast path would then derive `edges` / cursors / `hasPreviousPage` / `hasNextPage` from a full unbounded row set, producing wrong pages. Decision 4 names this the reversed-row-number branch and the cursor-parity invariant depends on it.

Recommended change: in `_connection_window_slice`, return the actual `last` value as the limit when `reverse` is set (the upstream `field_kwargs.get("last")` semantics), e.g. `limit = last if reverse else slice_meta.expected`. Keep `reverse` as is. Then `apply_window_pagination(reverse=True, limit=last)` applies `_dst_row_number_reversed__lte=last`.

Test expectation: extend `test_window_last_only_uses_reversed_row_number` (it currently only asserts the annotation is *present*, which is why the bug slipped through) to assert the `__lte` row filter is actually applied — e.g. `assert "_DST_ROW_NUMBER_REVERSED" in sql and "<=" in sql` over `str(prefetch.queryset.query)`, or assert the filtered queryset is bounded to `last` rows against a multi-child fixture.

### Medium:

#### Late-detected fallback connections leak the child plan's resolver keys / FK-id elisions into the parent

`_plan_connection_relation` builds the child queryset via `_build_prefetch_child_queryset` (walker.py:1146) **before** three of its fallback early-returns: the DISTINCT guard (1158), the malformed-slice guard (1169), and the unsupported-partition guard (1178). But `_build_prefetch_child_queryset` calls `_merge_child_plan_metadata(parent_plan, child_plan)`, which writes the child plan's `fk_id_elisions` and `planned_resolver_keys` into the **parent** plan (walker.py:631-634), and may flip `parent_plan.cacheable = False`. So a fallback connection whose node selection contains a nested relation leaks that relation's resolver key into the parent plan even though no window `Prefetch` is emitted for the connection.

Temp test confirmed: `booksConnection { edges { node { shelf { id } } } }` over a `.distinct()`-ing `Book.get_queryset` (a DISTINCT fallback) and the same shape with a malformed `after:` (a malformed-slice fallback) both leave the parent plan carrying `planned_resolver_keys = ('BookType.shelf@booksConnection.edges.node.shelf',)` — a `planned_resolver_keys` entry for a path under a connection the spec requires to be left fully unplanned.

Why it matters: (1) it violates the literal Decision 6 / DoD-item-4 contract for fallback shapes ("no window prefetch, **no `planned_resolver_keys` entry**"); (2) it is a Slice-4 strictness false-negative hazard — the leaked key marks `booksConnection.edges.node.shelf` "planned" in the parent plan, but the parent plan serves no prefetch covering it (the connection falls back to per-parent resolution), so a genuinely unplanned-by-the-parent nested access could read as planned. The list-relation path (`_plan_prefetch_relation`) does NOT have this hazard because it has no post-`_build_prefetch_child_queryset` early-returns — this is a defect introduced by this slice's three late guards, not a pre-existing pattern. The existing Slice-1 fallback tests miss it only because they use a scalar (`title`) node child, which generates no resolver keys.

Recommended change: order the side-effect-free guards before the child build. The window-slice resolution (`_connection_window_slice`, pure) and the partition resolution (`window_partition_for_prefetch`, pure) can both run before `_build_prefetch_child_queryset`, eliminating the leak for the malformed-slice and unsupported-partition cases entirely. The DISTINCT guard genuinely needs the built queryset; for it, build the child queryset against a throwaway sub-plan and merge into the parent only on the success path (or roll back the merged keys / cacheable flip on the distinct early-return). Recommend the reorder + throwaway-plan approach so a fallback connection mutates `plan` only when it actually plans a window.

Test expectation: add a fallback test (DISTINCT and/or malformed-slice) whose node selection carries a nested **relation** (not just a scalar), asserting `plan.planned_resolver_keys == ()` and `plan.fk_id_elisions == ()` — promote the temp test that caught this.
