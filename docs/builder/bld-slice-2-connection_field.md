## Plan (Worker 1)

### DRY analysis

**Existing patterns reused (cite file::Symbol):**

- **`DjangoListField` factory + its four constructor guards** — `list_field.py::DjangoListField` (the guard block at `list_field.py::DjangoListField #"if not inspect.isclass(target_type)"` through the `definition.origin is target_type` own-class check and the `callable(resolver)` check). `DjangoConnectionField` runs the **same four guards in the same order** (`inspect.isclass` → `issubclass(DjangoType)` → own-class `definition.origin is target_type` → callable `resolver`) plus a fifth Relay-Node-shaped guard. The guard order is load-bearing (each check assumes the prior passed). Do NOT copy the four guard bodies literally — the spec leaves the exact spelling to Worker 2, but the cleanest DRY move is to **extract the shared four-guard sequence** into one small helper both factories call (e.g. `_validate_djangotype_target(target_type, resolver, *, field="DjangoConnectionField")`), since the messages differ only in the field name. See "New helpers justified" and Implementation discretion items — Worker 2 picks extraction vs. a tightly-mirrored copy, but a near-verbatim copy of four `raise ConfigurationError(...)` blocks across two modules is a DRY defect Worker 3 will flag.
- **`DjangoListField` consumer-resolver post-processing** — `list_field.py::_post_process_consumer_sync` / `list_field.py::_post_process_consumer_async` (`Manager` → `.all()` coercion → `isinstance(QuerySet)` → `_apply_get_queryset_*` → list/generator passthrough) and `list_field.py::_is_async_callable` (the `iscoroutinefunction(fn)` + `iscoroutinefunction(fn.__call__)` two-check async detection). The connection field's consumer-`resolver=` contract (Decision 7) is a **superset** of this: same `Manager`/`QuerySet`/iterable branching, but the connection field ADDS the sidecar-input-against-non-queryset → `GraphQLError` rule and the `totalCount`-over-non-queryset → `GraphQLError` rule (the M1 carry-forward). The base `Manager`-coercion + `get_queryset`-application logic must be reused, not re-implemented; the connection field wraps it with the sidecar/filter/order/default-order/optimizer steps that come after visibility.
- **Relay-foundation sync/async `get_queryset` dispatch** — `types/relay.py::_apply_get_queryset_sync` / `types/relay.py::_apply_get_queryset_async` (the visibility step + the `SyncMisuseError` raise on a coroutine in a sync context) and `types/relay.py::_initial_queryset` (the default `model._default_manager.all()` base queryset, **unordered** for a model without `Meta.ordering` — this is exactly why Decision 7's default-ordering step exists). The connection resolver's sync path calls `_apply_get_queryset_sync`; the async path awaits `_apply_get_queryset_async`. `SyncMisuseError` (`types/relay.py::SyncMisuseError`) is inherited unchanged — the connection field raises nothing new for the sync-meets-async case.
- **`FilterSet` / `OrderSet` apply pairs** — `filters/sets.py::FilterSet.apply_sync` / `filters/sets.py::FilterSet.apply_async` and `orders/sets.py::OrderSet.apply_sync` / `orders/sets.py::OrderSet.apply_async`, all with the identical `(cls, input_value, queryset, info)` classmethod signature. The pipeline calls `filterset_cls.apply_sync(filter, qs, info)` (step 3) and `orderset_cls.apply_sync(order_by, qs, info)` (step 4) on the sync path, awaiting the `_async` variants on the async path. Active-input gating, `check_*_permission` propagation, and `RelatedFilter`/`RelatedOrder` visibility scoping all ride inside these calls — the connection field adds nothing to them.
- **`filter_input_type` / `order_input_type` helpers + their orphan ledgers** — `filters/__init__.py::filter_input_type` (returns the `Annotated["<Name>FilterInputType", strawberry.lazy(...)]` forward-ref AND adds the FilterSet to `filters/__init__.py #"_helper_referenced_filtersets"`) and `orders/__init__.py::order_input_type` (the `OrderSet` mirror, adding to `orders/__init__.py #"_helper_referenced_ordersets"`). The factory calls these two helpers to (a) get the exact lazy `Annotated` shape a hand-written resolver uses — so a connection field and a hand-written resolver on the same type resolve to the SAME `<Type>FilterInputType` / `<Type>OrderInputType` (stable class-derived names, Apollo-cache friendly) — and (b) register against the orphan ledgers so `finalize_django_types()` orphan validation stays honest. The `order_input_type` return is wrapped `list[order_input_type(OS)] | None` per the spec (`order_input_type` docstring: consumers wrap as `list[...]` for the `orderBy: [MyOrderInputType!]` shape). **Calling these helpers IS the ledger registration** — no separate `.add(...)` call is needed or wanted.
- **Relay foundation `_initial_queryset`** — `types/relay.py::_initial_queryset` (already cited above for the default base queryset / unordered property).
- **`_optimize` plan logic** — `optimizer/extension.py::DjangoOptimizerExtension._optimize` (steps 4-6: `_get_or_build_plan` → `_publish_plan_to_context` → `diff_plan_for_queryset` → `plan.apply`). Slice 2 **extracts** the plan-build-and-apply tail of `_optimize` into a reusable helper that takes `target_type` / `target_model` **directly** (NOT inferred from `info.return_type`, which for a connection field is the connection type, not the node type). The current `_optimize` does steps 1-3 itself: Manager-coerce, QuerySet-gate, then `_resolve_model_from_return_type(info)` to get `(origin, model)`. The extraction line is **after** step 3 — the helper accepts an already-resolved `(target_type=origin, target_model=model)` plus the queryset, then runs the `info.field_nodes` guard, `convert_selections` + `_root_child_selections`, `_get_or_build_plan(... , target_model, info, origin=target_type)`, `_publish_plan_to_context`, `diff_plan_for_queryset`, `plan.apply`. `_optimize` is rewritten to resolve `(origin, model)` from the return type (as today) and then delegate to the SAME helper, so the middleware path stays behavior-identical for non-connection fields.

**New helpers / module justified (single responsibility each):**

- **`DjangoConnectionField(target_type, *, resolver=None, description=None, deprecation_reason=None, directives=())`** in `connection.py` — single responsibility: the PascalCase factory that validates the target, synthesizes the sidecar-carrying resolver, and returns `relay.connection(_connection_type_for(target_type), resolver=<synthesized>, …)`. Meta-only derivation; no `filters=` / `order=` / `total_count=` kwargs (Decision 5).
- **The synthesized resolver wrapper** (an internal builder, e.g. `_build_connection_resolver(target_type)`) in `connection.py` — single responsibility: produce the field resolver whose `__signature__` / `__annotations__` carry `filter` / `order_by` params (per the type's declared sidecars) and whose body runs the composition pipeline. This is the one place the pipeline lives.
- **`apply_connection_optimization(target_type, queryset, info)`** (name per Decision 11 line 477) — the helper extracted from `_optimize` in `optimizer/extension.py`. Single responsibility: build + apply the optimizer plan to a queryset given the node type / model directly. Called by the connection resolver (step 6) AND by `_optimize` (so the middleware path is behavior-identical). See Implementation discretion item on the instance-vs-module-level shape (the plan cache is instance-bound).
- **A four-guard target validator** (optional, e.g. `_validate_djangotype_target(...)`) shared between `DjangoListField` and `DjangoConnectionField` — single responsibility: run the four DjangoType-target guards with a field-name-parameterized message. Worker 2 may extract this or mirror it tightly; the architectural call (extract vs. mirror) is left to Worker 2 in Implementation discretion items, with the constraint that a near-verbatim copy of four `raise` blocks is a DRY finding.

**Duplication risk avoided:**

- **Re-implementing the `Manager`/`QuerySet`/iterable consumer-resolver contract.** The naive shape would copy `_post_process_consumer_sync` / `_async` into `connection.py`. The plan reuses the `Manager`-coercion + `_apply_get_queryset_*` core and ONLY adds the connection-specific sidecar-input / `totalCount` rejection rules around it. (Note: the connection pipeline is genuinely richer than `DjangoListField`'s — it inserts filter/order/default-order/optimizer between visibility and return — so the wrapper cannot be a literal call to `_post_process_consumer_*`; the DRY target is reusing the Manager-coercion and `get_queryset` application, not the whole function.)
- **Duplicating `_optimize`'s plan logic.** Decision 11 (lines 477, 485) is explicit: extract a shared helper rather than copy `_optimize`. The plan extracts the build-and-apply tail and has both `_optimize` and the connection resolver call it, so the connection-aware walker work in `033` improves both call sites at once.
- **A second `filter_input_type` / `order_input_type` shape or a fresh per-field input type.** Decision 6 (line 406) forbids generating per-connection-field input types — that would break Apollo cache reuse and the stable-name contract. The factory calls the EXISTING helpers, which return the same lazy `Annotated` a hand-written resolver uses.
- **A second `first` + `last` guard or a re-parse of `Meta.connection`.** Both live in Slice 1's `connection.py` (`_guard_first_and_last`, `_connection_type_for` reading `definition.connection`); Slice 2 only CALLS `_connection_type_for(target_type)` — it does not re-derive the connection type or re-validate the opt-in.
- **A second Relay-Node predicate in the factory.** The fifth (Relay-Node) guard should reuse `types/relay.py::implements_relay_node(target_type)` (`issubclass(type_cls, relay.Node)`) rather than inventing a new spelling. By the factory's third guard the target is a registered concrete `DjangoType`; `implements_relay_node` is the single-sited predicate. (Distinct from `_validate_connection`'s interfaces-tuple check in Slice 1 — that runs at `_validate_meta` time before `cls` is usable; here the class IS usable, so the MRO-based `implements_relay_node` is correct.)

### Implementation steps

Line numbers are pin-at-write-time navigational hints. Verify against current source before editing — Slice 1 has shifted `connection.py`, and `optimizer/extension.py` is at HEAD.

1. **`connection.py` — `DjangoConnectionField` factory + the five guards.** Add `DjangoConnectionField(target_type, *, resolver=None, description=None, deprecation_reason=None, directives=())` (`# noqa: N802` for the PascalCase factory name, mirroring `list_field.py::DjangoListField`). Run the four `DjangoListField`-style guards (cite `list_field.py::DjangoListField #"if not inspect.isclass(target_type)"`): (1) `inspect.isclass`, (2) `issubclass(DjangoType)`, (3) own-class `definition.origin is target_type` (the strict invariant — NOT `hasattr`, per the `list_field.py::DjangoListField` comment block), (4) `callable(resolver)` when `resolver is not None`. Then the **fifth Relay-Node guard**: if `not implements_relay_node(target_type)` raise `ConfigurationError("a connection field requires a Relay-Node-shaped DjangoType; add `relay.Node` to `Meta.interfaces`")` (message per spec Error-shapes line 288). Messages name `DjangoConnectionField` and the offending value, mirroring the sibling factory's tone. Return `relay.connection(_connection_type_for(target_type), resolver=<synthesized>, description=description, deprecation_reason=…, directives=…)` — verify which of `description` / `deprecation_reason` / `directives` `relay.connection()` accepts against the locked Strawberry `0.316.0` (the Risks fork below) and pass through what it accepts. Do NOT export the symbol — Slice 4 owns `__init__.py` (Decision 14 / build-plan public-export flag).

2. **`connection.py` — synthesized-`__signature__` argument injection (PREFERRED route, Decision 6).** Build the field resolver (`_build_connection_resolver(target_type)` or inline in the factory) whose body is the pipeline (step 4) and whose `__signature__` + `__annotations__` carry the sidecar params:
   - Read `definition = target_type.__django_strawberry_definition__`; `definition.filterset_class` and `definition.orderset_class` (the slots Slice 0.0.8 added; cite `types/definition.py #"filterset_class: type | None = None"`).
   - When `definition.filterset_class` is set: add a `filter` parameter annotated `filter_input_type(definition.filterset_class) | None = None` (calling `filters/__init__.py::filter_input_type`, which registers the orphan ledger).
   - When `definition.orderset_class` is set: add an `order_by` parameter annotated `list[order_input_type(definition.orderset_class)] | None = None` (calling `orders/__init__.py::order_input_type`; the `list[...]` wrap produces `orderBy: [<T>OrderInputType!]` per the `order_input_type` docstring).
   - Set `resolver.__signature__` (an `inspect.Signature` with the synthesized params, including `root` / `info` as Strawberry expects) and a matching `resolver.__annotations__` so Strawberry's native resolver-argument derivation emits the `filter:` / `orderBy:` GraphQL arguments — the SAME path the hand-written filter/order resolvers use. The `search:` argument is NOT generated (search is `0.1.2`).
   - Worker 2 must verify against the locked Strawberry `0.316.0` that `relay.connection()`'s `ConnectionExtension` cleanly **merges** the resolver-signature args with the auto-added `before`/`after`/`first`/`last` pagination args and **forwards** the non-pagination `**kwargs` (`filter` / `order_by`) to the inner resolver (the Current-state note at spec line 122 verified `ConnectionExtension.resolve` forwards remaining `**kwargs`). **This is the Slice 2 argument-injection fork** — see Implementation discretion items and Risks: if signature-derivation does not compile/work, fall back to a custom `FieldExtension.apply(...)` appending `StrawberryArgument`s; record which route landed.

3. **`connection.py` — register the sidecars against the orphan ledgers.** This is satisfied **by calling** `filter_input_type` / `order_input_type` in step 2 (each helper does `_helper_referenced_filtersets.add(...)` / `_helper_referenced_ordersets.add(...)` itself). Do NOT add a separate `.add(...)` — that would double-register and diverge from the hand-written-resolver path. Worker 2 confirms the ledger entries land by the `test_connection_field_registers_sidecars_against_orphan_ledgers` test.

4. **`connection.py` — the composition pipeline inside the resolver (Decision 7), sync + async (Decision 10).** The resolver pops `filter` / `order_by` from kwargs (forwarded by `ConnectionExtension`) and applies, in this EXACT order, then returns the queryset for `ConnectionExtension` to slice:
   1. **base queryset** — `_initial_queryset(target_type)` when `resolver is None`; otherwise the consumer `resolver=` return run through the contract below.
   2. **visibility** — `_apply_get_queryset_sync(target_type, qs, info)` (sync) / `await _apply_get_queryset_async(target_type, qs, info)` (async). A sync context meeting an async `get_queryset` raises `SyncMisuseError` (inherited from `types/relay.py`, unchanged).
   3. **filter** — when `filter` is given AND `definition.filterset_class` is set: `definition.filterset_class.apply_sync(filter, qs, info)` (sync) / `await …apply_async(…)` (async).
   4. **orderBy** — when `order_by` is given AND `definition.orderset_class` is set: `definition.orderset_class.apply_sync(order_by, qs, info)` (sync) / `await …apply_async(…)` (async).
   5. **default deterministic ordering** — `if not qs.ordered: qs = qs.order_by(target_model._meta.pk.attname)`. A supplied `orderBy` (step 4) or a model `Meta.ordering` already marks `qs.ordered` True and is preserved. (`target_model = definition.model`.)
   6. **optimizer plan** — `qs = apply_connection_optimization(target_type, qs, info)` (step 5 of the implementation, the helper from step 6 below).
   7. return `qs` — `ConnectionExtension` / `resolve_connection` slices it; `totalCount` (if selected) counts the queryset entering this return, post-filter pre-slice (Slice 1's `_attach_count_*` does the count).
   - **Consumer `resolver=` contract** (Decision 7, extends `list_field.py::_post_process_consumer_sync` / `_async`): a `Manager` → `.all()` coercion → steps 2-7; a `QuerySet` → steps 2-7; a **non-queryset iterable** (list / generator) → paginated only when NO `filter:` / `orderBy:` input is supplied (steps 2-6 apply only to querysets), and **supplying sidecar input against a non-queryset raises a clear `GraphQLError`** (the advertised Meta-driven behavior cannot apply to a non-queryset). Reuse the `Manager`-coercion + `_apply_get_queryset_*` core from `list_field.py` rather than re-implementing it.
   - **M1 carry-forward — `totalCount`-over-non-queryset raises (Decision 7, the spec edit Slice 1 made).** On a `total_count`-opted-in type, selecting `totalCount` against a consumer-resolver return that is NOT a `QuerySet` must raise a clear package `GraphQLError` (NOT the engine's `Cannot return null for non-nullable field …totalCount` violation). This is symmetric with the sidecar-input rule. The cleanest home is in the count helper or the resolver's non-queryset branch: when the return is a non-queryset iterable and `_total_count_requested(info)` is True on a `total_count`-opted-in type, raise the package `GraphQLError` rather than letting the count be skipped (which would return `null` into `Int!`). **L2 (carry-forward) lands here too:** once this raise makes the count path QuerySet-only, change Slice 1's `connection.py::_build_total_count_connection #"def total_count"` resolver return type from `-> int | None` to `-> int` (the field renders `Int!`, so `int` is the honest type; the `__annotations__ = {"total_count": int}` already wins for the SDL). Decide where the raise lives so the `_attach_count_*` helpers and the resolver branch agree — Worker 2's call, recorded under discretion.
   - **Sync vs async dispatch** mirrors `list_field.py::DjangoListField`: the `resolver is None` default branch dispatches per-call on `in_async_context()`; a consumer-`resolver=` branch commits per-construction via `_is_async_callable(user_resolver)` (reuse `list_field.py::_is_async_callable`). The `totalCount` async count uses `.acount()` (already in Slice 1's `_attach_count_async`).

5. **`connection.py` — L1 cleanup (carry-forward).** Slice 1's generated `<TypeName>Connection.resolve_connection` (`connection.py::_build_total_count_connection #"_guard_first_and_last(first, last)"`) calls `_guard_first_and_last` explicitly AND super-chains into `DjangoConnection.resolve_connection`, which guards again (idempotent double-run). Since Slice 2 re-touches `connection.py`, resolve L1: either drop the explicit call in the generated override (rely on the inherited `DjangoConnection` guard via the super chain) or add a one-line comment that the double-run is intentional defensiveness. Dropping the explicit call is the cleaner DRY shape (the guard literal stays single-sited in `_guard_first_and_last`, and the super chain already enforces it). Worker 2's call — recorded under discretion.

6. **`optimizer/extension.py` — extract `apply_connection_optimization` from `_optimize` (Decision 11).** Extract the plan-build-and-apply tail of `optimizer/extension.py::DjangoOptimizerExtension._optimize` (the logic from `if not info.field_nodes` through `return plan.apply(result)`, i.e. `optimizer/extension.py::DjangoOptimizerExtension._optimize #"if not info.field_nodes"` onward) into a reusable helper that accepts `target_type` (origin) / `target_model` directly. The helper must NOT call `_resolve_model_from_return_type(info)` (the return type for a connection field is the connection type). Rewrite `_optimize` to: do steps 1-3 as today (Manager-coerce, QuerySet-gate, `_resolve_model_from_return_type` → `(origin, model)`), then **delegate to the same helper** with `target_type=origin, target_model=model`. The connection resolver (step 4 above, pipeline step 6) calls the helper with the node type / model directly.
   - **Plan-cache reuse fork (discretion):** `_get_or_build_plan` and `_publish_plan_to_context` are instance methods on `DjangoOptimizerExtension` (they mutate the instance-bound plan cache, `optimizer/extension.py::DjangoOptimizerExtension #"self._plan_cache"`). Record which route landed under Notes for Worker 1.
