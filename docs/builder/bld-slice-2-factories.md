- **Duplication risk avoided.**
  - **Risk 1: Re-deriving the Relay-vs-scalar branch in the factory.** The naive port of cookbook `_build_input_fields` would re-implement the FK/PK→GlobalID branch by reading `cls.FILTER_DEFAULTS` directly. The spec's Decision 4 line 579-584 explicitly REJECTS this pattern: "the input shape is downstream of the filter shape." Mitigation: the converter MUST receive the already-resolved `filter_instance` (from `filterset_cls.get_filters()`) and inspect ITS class via `isinstance(filter_instance, GlobalIDFilter)` etc. The factory never reads `FILTER_DEFAULTS`; it reads the resolved filters. The plan's implementation steps reflect this: `_build_input_fields` receives `all_filters = filterset_cls.get_filters()` and passes individual filter instances into the converter.
  - **Risk 2: Re-walking `LOOKUP_NAME_MAP` in both the factory and the runtime.** Slice 1's `_normalize_input` already walks the table for key mapping; Slice 2's `_build_input_fields` walks the same table for `strawberry.field(name=<graphql_name>)` emission and `normalize_input_value` walks it for the runtime value step. Three walks across two surfaces is acceptable because each walk has a different output (Strawberry field metadata vs form-data dict vs converted value), and all three consult ONE table. The risk is that someone adds a fourth call site without consulting `LOOKUP_NAME_MAP`. Mitigation: each walk is contained in one named helper (`_form_key_for_python_attr`, `_build_input_fields`, `normalize_input_value`); the static-inspection helper at integration time flags any new bare string literal matching a `LOOKUP_NAME_MAP` key as a DRY candidate.
  - **Risk 3: Parallel `FieldSpec` construction sites.** `_build_input_fields` populates `_field_specs[(filterset_cls, python_attr)]`; `normalize_input_value` reads them. The risk is that a future caller (e.g., `materialize_input_class` in Slice 3) re-derives a `FieldSpec` instead of looking it up. Mitigation: `_field_specs` is read-only outside `_build_input_fields`; Slice 2's tests pin the (write site, read site) pair and a stable accessor `_field_spec_for(filterset_cls, python_attr)` is NOT introduced (premature — only two call sites today; extract later if a third needs it). Documented in `Implementation discretion items`.
  - **Risk 4: Duplicate `_LOGIC_KEYS` constant.** Slice 1 declared `_LOGIC_KEYS` in `sets.py:53-57` for the input-normalizer; Slice 2's `_build_logic_fields` needs the same `("and_", "and") / ("or_", "or") / ("not_", "not")` pairs. The risk is declaring a parallel constant in `inputs.py`. Mitigation: `_build_logic_fields` imports `_LOGIC_KEYS` from `..sets` (forward import — no cycle because `inputs.py` is the leaf module from `sets.py`'s perspective; the import lands at module top with a `TYPE_CHECKING` guard if needed at integration time, but `_LOGIC_KEYS` is a plain tuple constant, not a class, so a regular import is safe). Alternative — move `_LOGIC_KEYS` to `inputs.py` as the input-vocabulary owner — is the cleaner long-term shape; documented in `Implementation discretion items` for Worker 2 to choose.
  - **Risk 5: Spy test fragility from Slice 1 carry-forward.** Per worker-1.md memory, `test_apply_sync_passes_constrained_queryset_to_filterset_instance` relies on a downstream `GraphQLError` from form validation to terminate `apply_sync` after the spy fires. If Slice 2 strips related-branch keys from `data` before form construction (the cookbook DOES do this — see `_apply_related_queryset_constraints` and the related-key normalization in `_normalize_input`), the GraphQLError no longer raises and `pytest.raises(GraphQLError)` fails. The Slice-2 plan addresses this — see Implementation step 7. Avoiding a parallel-pattern duplication: Slice 2 does NOT introduce a second spy test for the H3-rev8 ordering; it RESHAPES the existing test so the contract is pinned by the captured kwarg directly (no `pytest.raises` wrapper), preserving the single-test-per-contract pattern.

### Implementation steps

1. **Add `build_input_class` to `django_strawberry_framework/filters/inputs.py`.**
   - Insert after the existing `_materialized_names` declaration (currently line 107).
   - Signature: `def build_input_class(name: str, field_specs: list[tuple[str, type, dict[str, Any] | None]]) -> type:`
   - Body: use `dataclasses.make_dataclass(name, [(attr, ann, dataclasses.field(default=default, ...)) for attr, ann, kwargs in field_specs], ...)` THEN apply `@strawberry.input` via `strawberry.input(...)` callable form (NOT decorator) so the class object is returned. Per spec line 482, the annotations may include `Annotated[forward_ref, strawberry.lazy(...)]` directly. Field-kwargs dict carries optional `description=` / `name=` / `default=None`.
   - Implementation discretion: `make_dataclass + strawberry.input(cls)` vs `strawberry.input(make_dataclass(...))` vs explicit `class` body via `types.new_class`; planner suggests `strawberry.input(make_dataclass(...))` to match the cookbook's `type(name, (graphene.InputObjectType,), fields)` flat shape — Worker 2 picks whichever is most legible.

2. **Add `convert_filter_to_input_annotation` and `normalize_input_value` to `inputs.py`.**
   - Insert after `build_input_class`.
   - `convert_filter_to_input_annotation(filter_instance, model_field, owner_definition)` covers the spec Decision-4 M1 table (lines 588-597). Branch order (most-specific to least-specific):
     1. `isinstance(filter_instance, GlobalIDMultipleChoiceFilter)` → `list[str]` (per H2 of rev4: the inner annotation is `Annotated["...", strawberry.lazy(...)]` ONLY for filterset cross-references; for the GlobalID string list, the inner is just `str`).
     2. `isinstance(filter_instance, GlobalIDFilter)` → `str`.
     3. `isinstance(filter_instance, RangeFilter)` → a Strawberry input dataclass with `start: T | None` and `end: T | None` (consult `model_field.to_python` for `T`; spec line 594).
     4. `isinstance(filter_instance, ListFilter)` → `list[<scalar>]`.
     5. `isinstance(filter_instance, (ChoiceFilter, TypedChoiceFilter))` → derive the Strawberry enum via the converter pipeline at `django_strawberry_framework/types/converters.py::SCALAR_MAP` (spec line 591); raise `ConfigurationError` if the source is not a `Choices`-derived enum.
     6. Catch-all scalar branch — use `model_field.to_python` to pick `str` / `int` / `bool` / `decimal.Decimal` / `datetime.date` / `datetime.datetime` / `datetime.time` / `uuid.UUID` per spec line 590.
     7. `Filter(method=<callable>)` custom branch — inspect `filter_instance.field.__class__`; raise `ConfigurationError` naming both filter and method when the form-field shape is unknown (spec line 595).
     8. Nullable wrapping: if `filter_instance.extra.get("required", False)` is `False`, wrap in `T | None` (spec line 596).
   - `normalize_input_value(filter_instance, raw_value)` mirror table (spec line 598):
     - `GlobalIDFilter` → if `relay.GlobalID` instance, take `.node_id`; otherwise pass `str` through.
     - `GlobalIDMultipleChoiceFilter` → list-map the above per element.
     - `ChoiceFilter` → if Strawberry-enum-member, return `member.value`; else pass through.
     - `RangeFilter` → return positional `{<field>_0: start, <field>_1: end}` keys per H3 of rev5 (spec lines 594, 1022 — `RangeWidget.value_from_datadict` reads `name_0` / `name_1`). NOTE: this changes the FORM-KEY for `RangeFilter` inputs; integration with `FilterSet._normalize_input` requires letting the normalizer emit two keys for a single dataclass attr. Implementation discretion: `_normalize_input` returns a `dict[str, Any]` today (sets.py:381-391); for `RangeFilter` the value at the `range`-keyed entry must EXPAND into two positional keys. Worker 2 picks: (a) `normalize_input_value` returns a `dict[str, Any]` patch the caller merges; (b) `normalize_input_value` returns a single value for scalar primitives and a sentinel `_PositionalPair(start, end)` that `_normalize_input` recognizes and explodes; (c) `_normalize_input` itself calls `normalize_input_value` and then post-processes the result. Planner recommends (a) — keeps the per-primitive logic in `normalize_input_value` and the merge mechanics in `_normalize_input`.
   - Both helpers consult `_field_specs[(filterset_cls, python_attr)]` for `django_source_path` rebuild on per-field-name paths (the `galaxy__name` case, spec line 506-526).

3. **Add `_build_logic_fields(type_name: str)` to `inputs.py`.**

### Implementation notes

- **`build_input_class` shape — `type(name, (), namespace)` over `dataclasses.make_dataclass`.** `make_dataclass` replaces any `strawberry.field(default=..., name=...)` default with a plain `dataclasses.Field` and strips the Strawberry metadata. The `name=` alias would be silently lost. The `type(name, (), {"__annotations__": ..., "<attr>": strawberry.field(name=...)})` shape preserves both the annotation and the `strawberry.field` metadata through the `@strawberry.input` decoration. Implementation discretion item 1 named this as Worker-2 choice.
- **`_LOGIC_KEYS` placement — duplicated in `inputs.py` (mirroring `sets.py`).** Implementation discretion item 3 offered two choices: move to `inputs.py` (planner-recommended) or leave in `sets.py`. I duplicated as a separate module-level constant in `inputs.py` because Slice 1 already imports `LOOKUP_NAME_MAP` from `inputs.py` (no cycle); moving from `sets.py` would have required editing `sets.py`'s import / use site, an unnecessary churn. The two constants are byte-for-byte identical; integration-pass DRY review will likely fold them.
- **`normalize_input_value` return shape — option (a) `dict[str, Any]` patch for Range.** Implementation discretion item 4 (planner-recommended). The `_normalize_input` caller checks `isinstance(normalized, dict)` and calls `data.update(normalized)`; scalar cases assign single keys. Keeps the per-primitive logic in `normalize_input_value` and the merge mechanics in `_normalize_input`.
- **`FilterArgumentsFactory.arguments` return shape — input class itself.** Implementation discretion item 5 (planner-recommended) per spec line 477. The `filter_input_type(...)` consumer helper is the boundary that wraps it in the `Annotated[...]` lazy reference for resolver signatures.
- **`_build_input_fields` / `_build_logic_fields` placement — module-level free functions in `inputs.py`.** Implementation discretion item 2 (planner-recommended). The factory calls them via `from .inputs import _build_input_fields, _build_logic_fields, build_input_class`. No `self`-dependence on the factory state needed at Slice-2 scope.
- **`anytree`-free tree algorithm — flat grouped dict.** Implementation discretion item 7. Rather than port the cookbook's `filterset_to_trees` / `try_add_sequence` / `sequence_to_tree` recursive shape with an `anytree.Node`-substitute dataclass, the package uses a flat `OrderedDict[top_name, OrderedDict[lookup, filter_instance]]` grouping. This works for the Slice-2 surface because nested transforms (`created__date__year`-shaped multi-level lookups) are NOT in the Slice-2 contract — the spec's M5 example `galaxy__name` is a flat single-hop. The grouping shape pins the django source path on `_field_specs[(cls, python_attr)].django_source_path` so the runtime normalizer has the canonical lookup expression without needing tree-reconstruction.
- **Range-filter form-data positional patch — per-form-key prefix from caller.** `_normalize_range_value(filter_instance, raw_value, field_name=form_key)` accepts an optional override for the prefix because the form-key in `_normalize_input` is what `django-filter`'s form expects (`<form_key>_0` / `<form_key>_1`), not necessarily `filter_instance.field_name` (which could carry a relation path).
- **Skip expanded `RelatedFilter` entries at leaf-grouping time.** Self-referential filtersets (`SelfReferentialBranchFilter` in the cookbook fixtures) produce `self_link__self_link` after `get_filters()` expansion — a deep-copied `RelatedFilter` at depth 2. The grouping treats `self_link__self_link` as a top-level GraphQL field (not in `cls.related_filters`), and the leaf branch would hand the `RelatedFilter` to the converter, hitting the `ChoiceFilter` guard. Mitigation: `_build_input_fields` skips `filter_name` entries where `"__" in filter_name and isinstance(filter_instance, RelatedFilter)` — the top-level `RelatedFilter` forward-ref already covers the same target. This is a small mechanical drift from the cookbook (which handles the same case via the tree-form recursion); recorded here per the small-drift contract.
- **`RangeFilter` isinstance branch checks both `django_filters.RangeFilter` AND the package's `RangeFilter`.** The package's `RangeFilter` extends `TypedFilter` (NOT `django_filters.RangeFilter`), but the package's primitive provides the same start/end shape. The converter checks both classes to cover both shapes.
- **`type_name` for choice-enum derivation — falls back to `"Filter"`.** When `owner_definition` is `None` (the dynamic-factory path before owner binding), `_choice_enum_from_filter` calls `convert_choices_to_enum(model_field, type_name="Filter")`. The cache in `convert_choices_to_enum` is `(model, field_name)`-keyed so the type_name only affects the enum's GraphQL name on first generation; subsequent calls return the cached enum unchanged.
