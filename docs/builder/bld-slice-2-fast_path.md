## Plan (Worker 1)

This slice is the **consumer** of Slice 1's windowed prefetch. Slice 1 plans a `Prefetch(<accessor>, queryset=<windowed child qs>, to_attr="_dst_<field>_connection")` whose rows carry `_dst_row_number` / `_dst_total_count` window annotations and whose slice metadata is `(offset, limit, reverse)` from `walker._connection_window_slice`. Nothing reads that window yet — Slice 2 wires the synthesized relation-connection resolver to **hand off** the windowed rows, and the generated connection class to **build the Relay object** straight from the annotations, skipping the per-parent pipeline. Slices 1+2 are one mechanism in the joint `0.0.9` cut (build-plan "Joint-cut mechanism gate"); this slice is what makes the Slice-1 window non-wasted, so the two commits land together.

The split between resolver and connection class is structural and **non-negotiable** (Decision 5, lines 292-293, 318): Strawberry's `ConnectionExtension.resolve` calls the field resolver for a **node iterable**, then invokes the generated connection class's `resolve_connection` over that iterable. A resolver that returned a prebuilt connection would have that connection fed back through `resolve_connection` as if it were the node iterable. So the resolver's fast-path job is *only* to return an internal marker wrapping the windowed rows; the generated `<TypeName>Connection.resolve_connection` (both the bare `DjangoConnection` path and the `totalCount` variant) detects the marker and constructs the Relay object. This is exactly why the **through-schema test is mandatory** (spec line 68; a direct helper call bypasses the `ConnectionExtension` layer boundary).

### DRY analysis

**Existing patterns reused (cite `file::Symbol`):**

- **Slice-1 window constants are the single source of the annotation names.** `optimizer/plans.py` exports `WINDOW_ROW_NUMBER = "_dst_row_number"`, `WINDOW_TOTAL_COUNT = "_dst_total_count"`, `WINDOW_ROW_NUMBER_REVERSED = "_dst_row_number_reversed"` (plans.py #"WINDOW_ROW_NUMBER = "). The fast path's annotation-presence probe and the `_dst_row_number` / `_dst_total_count` reads MUST import these constants, not re-spell the `"_dst_*"` literals — otherwise a Slice-1 rename silently breaks the consumer. `connection.py` already imports from `optimizer.plans` (#"from .optimizer.plans import deterministic_order, ends_in_unique_column"); extend that import.
- **The `to_attr` derivation helper is already single-sited in the walker.** `optimizer/walker.py::_relation_connection_to_attr(relation_field_name)` returns `_dst_<field>_connection` and is the one site that builds the literal (Slice-1 DRY pin; Worker 3 confirmed no scattered f-string). The Slice-2 resolver probe MUST reuse this helper for its `getattr(root, <to_attr>, None)` probe rather than re-spelling `f"_dst_{field_name}_connection"`. The walker must not import from `connection.py` (the one-way dependency the optimizer package keeps), but `connection.py` importing `_relation_connection_to_attr` from `walker` is the **forced-correct** direction (connection.py already imports `apply_connection_optimization` from `optimizer.extension`, so the optimizer→connection one-way rule is about the optimizer not reaching into connection, which this respects). Reuse it; do not duplicate.
- **The edge-class resolution + cursor encoding is Strawberry's own.** The fast path resolves the edge class exactly as `strawberry.relay.types.ListConnection.resolve_connection` does (#"field_def = type_def.get_field(\"edges\")" → `resolve_type` → unwrap `StrawberryContainer` → `edge_class`), then builds each edge via `edge_class.resolve_edge(node, cursor=<int offset>)` (#"def resolve_edge" → `to_base64(cls.CURSOR_PREFIX, cursor)`). The cursor PREFIX and base64 encoding stay owned by Strawberry's edge type — the fast path passes only the **integer offset**, never a hand-encoded cursor string (Decision 5 "the cursor prefix stays owned by Strawberry's edge type"). This is the upstream `strawberry-django` reference shape verbatim (`/Users/riordenweber/projects/strawberry-django-main/strawberry_django/relay/list_connection.py::resolve_optimized_connection_by_prefetch` #"cursor=node._strawberry_row_number - 1"): forward cursor `_dst_row_number - 1`, `start_cursor`/`end_cursor` from `edges[0]`/`edges[-1]`, `PageInfo` from the row-number bounds.
- **`PageInfo` and the connection instance constructor are Strawberry's.** Build `relay.PageInfo(start_cursor=, end_cursor=, has_previous_page=, has_next_page=)` and `cls(edges=, page_info=)` exactly as `ListConnection.resolve_connection` does at its tail (#"return cls(" / #"page_info=PageInfo(") and as upstream's `resolve_optimized_connection_by_prefetch` does. No package-local PageInfo / Edge / Connection shape.
- **`_total_count_requested(info)` is the selection-gate, reused unchanged.** `connection.py::_total_count_requested` (line 100) already checks the connection's **direct** children (through fragment wrappers, NOT into `edges { node }`) — this is the outer-`totalCount`-predicate scope the spec re-affirms (Edge case line 456: a nested connection's `totalCount` must not fire the outer predicate). The `totalCount`-variant fast path reuses it verbatim to decide whether to read `_dst_total_count`; do NOT add a second selection-gate.
- **The synthesized resolver pipeline tail is the fallback.** `connection.py::_build_relation_connection_resolver::_resolve` already runs `_pipeline_sync(target_type, getattr(root, accessor_name).all(), info, filter_input=..., order_by_input=...)` (line ~728). The fast-path handoff is a **branch added before** that call; when the window is absent / unannotated / sidecar-kwargs-present, the existing pipeline call is the unchanged fallback. Do NOT re-implement the pipeline; only add the early handoff branch.
- **`_synthesized_signature(target_type)` stays the resolver's signature source.** The resolver builder already calls `_synthesized_signature` (line ~736) to attach the `filter` / `order_by` parameters; the field-name parameter is a builder argument, NOT a new resolver signature parameter — it must not change `__signature__` (the SDL arguments are unchanged this slice).

**New helpers justified (single responsibility each):**

1. `connection.py::_WindowedConnectionRows` — module-scope (not exported) marker class. Single responsibility: carry the prefetched windowed `rows` (the `to_attr` list) plus the resolved `offset` / `limit` / `reverse` slice metadata the planner used, so the generated connection class can build edges / cursors / `pageInfo` / `totalCount` without re-deriving the window. A plain class (or dataclass / `__slots__`) holding `rows`, `offset`, `limit`, `reverse`. It is NOT a connection instance and NOT iterable-as-nodes — it is a typed sentinel the `resolve_connection` paths `isinstance`-detect. (Decision 5 "wrap it in an internal `_WindowedConnectionRows` marker that stores the rows plus the resolved offset, limit, and reverse slice metadata used by the planner".)
2. `connection.py::_resolve_from_window(cls, window, *, info, **kwargs)` — module-scope helper (or a `DjangoConnection` classmethod). Single responsibility: given a `_WindowedConnectionRows`, build and return the Relay connection instance (`cls(edges=, page_info=)`) — edge cursors from `_dst_row_number` (forward `- 1`; reversed `_dst_total_count - _dst_row_number`), `pageInfo` from the row-number bounds vs `_dst_total_count`, and `totalCount` attached (when the `totalCount` variant is in play and selected) from `_dst_total_count`. Called by BOTH `resolve_connection` paths (the bare `DjangoConnection` path and the generated `totalCount` path) so the edge/cursor/pageInfo derivation is single-sited. The `totalCount` attach is the variant's one extra step — see Implementation step 4 for how the two paths share this helper.
3. `connection.py::_recompute_slice_metadata` derivation for the wrapper — NOT a new helper; the resolver receives `(offset, limit, reverse)` it must re-derive at resolve time. **Decision (not discretion):** the resolver does NOT have the walker's `_connection_window_slice` (that is walker vocabulary and would couple `connection.py` to the walker's slice helper). Instead the resolver re-derives `(offset, limit, reverse)` from its OWN resolved `before`/`after`/`first`/`last` kwargs the same way the walker did — via Strawberry's `SliceMetadata.from_arguments(info, ...)` — OR carries them as walker-free wrapper fields. See "Duplication risk avoided" and Implementation step 3 for the resolution: the resolver re-runs `SliceMetadata.from_arguments` against its own kwargs (the same engine, same `relay_max_results` config, so the result is identical to the walker's by construction — Edge case "`relay_max_results` agreement" line 447), keeping the slice arithmetic single-engine.

**Duplication risk avoided:**

- **`"_dst_*"` literals re-spelled in `connection.py`.** Risk: the fast path hardcodes `"_dst_row_number"` / `"_dst_total_count"` / the `to_attr` f-string, drifting from Slice 1. Avoided: import `WINDOW_ROW_NUMBER` / `WINDOW_TOTAL_COUNT` from `optimizer.plans` and `_relation_connection_to_attr` from `optimizer.walker`. The reversed-cursor branch reads `WINDOW_ROW_NUMBER` (the canonical `_dst_row_number`) — the rows carry `_dst_row_number` (the partition-ordered value) AND, in the reversed plan, also `_dst_row_number_reversed`; the **cursor math** uses `_dst_total_count - _dst_row_number` (Decision 5 line 301), reading `_dst_row_number` + `_dst_total_count`, NOT `_dst_row_number_reversed` (that annotation only existed to apply the `__lte` filter at plan time). Pin this in the last-only parity test.
- **A second slice-arithmetic implementation.** Risk: the resolver hand-computes `offset`/`limit`/`reverse` from `first`/`last`/`before`/`after` instead of using `SliceMetadata`. Avoided: re-run `SliceMetadata.from_arguments(info, ...)` (the same helper the walker uses, already a load-bearing dependency of the shipped connection field) so plan-time and resolve-time arithmetic cannot diverge. This is the cursor-parity invariant's resolve-time half (Decision 4 line 272 / Decision 5 line 313 "the window was computed from the *same selection's* resolved arguments `resolve_connection` receives").
- **A second selection-gate for `totalCount`.** Risk: the fast path adds its own "is totalCount selected" walk. Avoided: reuse `_total_count_requested(info)` (the existing direct-children-scoped gate) — which also keeps the outer-predicate scope correct (Edge case line 456).
- **A third `resolve_connection` edge-build path.** Risk: the bare `DjangoConnection.resolve_connection` and the generated `totalCount` `resolve_connection` each grow their own window-detection-and-build block. Avoided: both call `_resolve_from_window` for the edge/cursor/pageInfo build; the `totalCount` variant adds only the `_dst_total_count` attach on top (Implementation step 4). One edge-derivation implementation, two thin call sites.
- **Empty-window fallback re-spelled in two paths.** Risk: the ambiguous-empty rule (`limit == 0` or `offset > 0` → fall back; `offset == 0` and `limit > 0` → serve empty) is duplicated in both `resolve_connection` paths. Avoided: the fast-pathability decision (is this wrapper consumable, or must it fall back?) lives in `_resolve_from_window` (or a small `_window_is_fast_pathable(window)` predicate it calls), returning a sentinel that tells the caller to delegate to `super()` for the per-parent pipeline. One rule, both paths.
