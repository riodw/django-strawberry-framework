## Plan (Worker 1)

### DRY analysis

- **Existing patterns reused.**
  - The defensive `getattr(info, ...)` posture for an optional `info` is already the file's idiom — `runtime_path_from_info(info)` (`plans.py:264-274` short-circuits `info=None` to `()`), `_relay_max_results_from_info` (`walker.py:1119-1138` walks `getattr(info, "schema", None)`), and `_resolve_field_map`'s dual-contract `getattr(..., default)` reads. The `enable_only` derivation must follow the same shape: never raise on a `None`/partial `info`. (Cite `walker.py::plan_optimizations` doc + `walker.py::_relay_max_results_from_info`.)
  - `info.operation.operation` is **already read** in production at `extension.py::_build_cache_key #"operation = info.operation"` (extension.py:1062) — so the attribute chain G2 reads exists at the real call site, and the defensive arms only guard direct/test callers of `plan_optimizations`. `from graphql import OperationType` is the canonical enum (confirmed importable; `OperationType.QUERY` / `MUTATION` / `SUBSCRIPTION`).
  - The four projection writers and their `info`-threading are already wired: `plan_optimizations` → `_walk_selections` → (`_record_relation_access`, `_plan_select_relation`/`_plan_prefetch_relation`, `_build_prefetch_child_queryset`, `_project_scalar_only_window` via `_plan_connection_relation`). `info` is already passed down every recursion, so the gate rides the existing `info` thread — **no new parameter plumbing for `info` itself**.
  - `append_unique(plan.only_fields, ...)` (`plans.py:353-365`) is the single mutation entry for `only_fields`; the gate guards each call site rather than re-implementing the append.
  - Decision 5 reuses the deferred-column detection already in `resolvers.py::_will_lazy_load_single #"if field_name in getattr(root, \"__dict__\", {})"` (resolvers.py:90) — "a loaded scalar/FK column lives in `root.__dict__`; a deferred one is absent and triggers a query on access." The loaded-check is the same `attname in root.__dict__` test applied to the FK `attname` before `getattr(root, field_meta.attname)`. The loud-fallback path reuses the existing `_check_n1` strictness machinery already called by `forward_resolver` (resolvers.py:325-334).
  - Decision 5 tests reuse the B2 elision test scaffolding already in `tests/types/test_resolvers.py` (`_path`, `_isolate_registry`, the `_build_fk_id_stub` direct-unit pattern at test_resolvers.py:202-216, and the `caplog`/strictness-`raise` patterns at test_resolvers.py:304-389).

- **New helper justified.** One module-private derivation helper in `walker.py`:
  `def _enable_only_for_operation(info: Any | None) -> bool` — single responsibility: map an optional `info` to the operation-wide projection gate (the three arms: no/partial `info` → `True`; `QUERY` → `True`; non-`QUERY` → `False`). It is the ONE place the `OperationType` comparison and the defensive `getattr` chain live, so the decision is computed once in `plan_optimizations` and the resulting **bool** is threaded — never re-derived per writer. This avoids four scattered `info.operation` reads. Single call site (`plan_optimizations`); the bool it returns is the threaded value. No second helper is justified — the gate is a plain bool, not a new type.

- **Duplication risk avoided (the load-bearing DRY concern for this slice).** The naive implementation re-checks `info.operation is not OperationType.QUERY` inside each of the four writers — four copies of the same defensive comparison, four places to drift. The plan threads a **single derived bool** `enable_only` through the existing parameter chain instead:
  - `plan_optimizations` derives `enable_only` once via the new helper and passes it into the root `_walk_selections`.
  - `_walk_selections` receives `enable_only` and passes it to `_record_relation_access`, `_plan_select_relation`, `_plan_prefetch_relation`, `_build_prefetch_child_queryset`, and `_plan_connection_relation`; nested `_walk_selections` recursions forward the same bool (so root + every nested child plan share ONE operation decision — the plan-wide suppression Decision 4 requires).
  - Each of the four writers consults the threaded bool with a single `if enable_only:` (or `if not enable_only:` early-shape) guard around its `only_fields` append / direct `.only(...)` — never its own `info.operation` read.
  - The bool name `enable_only` is reused verbatim across signatures (matches the spec's and upstream's vocabulary) so the thread reads as one concept, not parallel flags.
