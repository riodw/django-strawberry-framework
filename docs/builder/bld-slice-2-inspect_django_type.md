### DRY analysis

- **Existing patterns reused.**
  - **`export_schema.py` `Command` shape** (`django_strawberry_framework/management/commands/export_schema.py:11-56`) is the verbatim structural template: `class Command(BaseCommand)` with `help`, `add_arguments(self, parser: CommandParser)`, `handle(self, *args, **options)`, and the `try: import_module_symbol(...) except (ImportError, AttributeError) as e: raise CommandError(str(e)) from e` discipline (`export_schema.py:34-40`). Slice 2's `--schema` loader is the **same call** — `import_module_symbol(options["schema"], default_symbol_name="schema")` — so the schema-import-and-finalize step is a literal reuse, not a near-copy.
  - **`registry.iter_types()`** (`django_strawberry_framework/registry.py:225-235`) yields `(model, type_cls)` once per registered type; the bare-name lookup iterates it collecting `type_cls.__name__ == arg` matches. **`registry.model_for_type(type_cls)`** (`registry.py:211-223`) gives the Django model for the ambiguity-candidate listing (`module.qualname` + model). No new registry surface needed.
  - **`DjangoTypeDefinition`** (`django_strawberry_framework/types/definition.py:73-106`): the command reads `definition.origin` (the `DjangoType` class), `definition.selected_fields` (Django field instances in selection order, `definition.py:79`), `definition.field_map` (`dict[str, FieldMeta]`, `definition.py:80`), and `definition.finalized` (`definition.py:95`). All already-public attributes.
  - **`FieldMeta`** (`django_strawberry_framework/optimizer/field_meta.py:50-189`): the command reads `name`, `is_relation`, `nullable`, `related_model`, the cardinality flags, plus `relation_kind` / `is_many_side` properties (`field_meta.py:107-115`) to classify the relation-converter row (forward FK / M2M / reverse FK). This is the Django-side metadata + converter-classification source.
  - **`scalar_for_field(field)` / `SCALAR_MAP`** (`django_strawberry_framework/types/converters.py:54-139`): the MRO walk that names *which* `SCALAR_MAP` row fired. The command re-runs this lookup **only to NAME the row** (Decision 4: "to NAME which row fired … not to determine nullability"). It MUST NOT re-run `convert_scalar(...)` for nullability — that would reproduce column-native `field.null` widening and miss a Slice-3 override or a consumer-authored annotation.
  - **`origin.__annotations__`** is the authoritative resolved-GraphQL-type + nullability record (written at `types/base.py:341` `cls.__annotations__ = {**synthesized, **consumer_annotations}` and rewritten per relation by the finalizer). The command reads nullability from here, NOT from a `convert_scalar` re-run.
  - **Test reload fixture**: `examples/fakeshop/test_query/test_library_api.py::_reload_library_project_schema` (lines 33-58) is the exact `registry.clear()` + `importlib.reload("apps.library.schema")` + `importlib.reload("config.schema")` (+ URLconf `clear_url_caches`) pattern the bare-name happy-path tests mirror, per `examples/fakeshop/test_query/README.md`.
  - **Failure-mode test scaffolding**: `tests/management/test_export_schema.py::_make_test_module` (lines 21-25) — the `types.ModuleType` + `monkeypatch.setitem(sys.modules, "test_module", module)` helper — is the template for the non-`DjangoType`-symbol and bad-dotted-path failure tests.

- **New helpers justified.** Within `inspect_django_type.py`, decompose `handle()` into small private methods/functions so each branch stays under the BUILD.md hotspot thresholds (40 lines / 8 branches) and each is independently legible. Suggested (names at Worker 2's discretion):
  - `_resolve_type(arg)` — the dot-dispatch resolver: dotted → `import_string` (raise-on-failure, original error), bare → unique-`__name__` registry scan.
  - `_row_for_field(definition, field)` — produce one printed row (Django field name, Django field type, resolved GraphQL type + nullability from `origin.__annotations__`, converter classification), including the Relay-suppressed-pk special case.
  These are command-internal, single-responsibility, and serve only this one command — they are NOT new package-level public helpers (the spec scopes Slice 2 to "no new public API"). Do NOT add a method to `DjangoTypeDefinition` or `registry` for the bare-name scan; iterating `registry.iter_types()` inline in the resolver is the right altitude (one call site).

- **Duplication risk avoided.** The naive implementation would call `convert_scalar(field, type_name)` to recompute the GraphQL type/nullability — duplicating the construction-time logic AND producing a wrong (column-native) answer for override/consumer-authored fields. The plan forbids that: nullability/type comes from `origin.__annotations__`; `scalar_for_field`/`SCALAR_MAP` is re-walked **only** for the converter-row *name*. Also avoid duplicating `export_schema`'s `import_module_symbol` try/except shape with a divergent message — reuse the same `except (ImportError, AttributeError) as e: raise CommandError(str(e)) from e` form for the `--schema` load so the two commands read identically.

### Implementation steps

Line numbers are pin-at-write-time hints; verify against current source before editing (Slice 1's accepted diff and the maintainer scaffold are already in the tree).

1. **`django_strawberry_framework/management/commands/inspect_django_type.py`** — replace the fail-loud `handle()` body (currently `inspect_django_type.py:25-30`) with the real command. Keep the existing `add_arguments` shape (`inspect_django_type.py:20-23`: positional `type`, optional `--schema`); update the module + class docstrings to describe shipped behavior (not "planned"). `handle()`:
   1. **Import `--schema` first when given.** `schema = options.get("schema")`; if truthy, `import_module_symbol(schema, default_symbol_name="schema")` inside `try/except (ImportError, AttributeError) as e: raise CommandError(str(e)) from e` (verbatim shape from `export_schema.py:34-40`). This registers + finalizes all types via the schema module's import-time side effects (class registration + the module's `finalize_django_types()` call). The resolved symbol itself is not used — the import is for its side effects (mirror `export_schema`'s import, but do not require it to be a `strawberry.Schema`).
   2. **Resolve the type argument by shape** (Decision 4; DoD item 5). `arg = options["type"]`. If `"." in arg`: dotted object path → `import_string(arg)` (from `django.utils.module_loading`), in a `try/except (ImportError, AttributeError) as e: raise CommandError(str(e)) from e` — **carry the original error; NEVER fall back to a registry lookup**. Else (no dot): bare name → scan `registry.iter_types()` collecting every `type_cls` whose `__name__ == arg`; **exactly one** → resolve it; **zero** → `CommandError("...not a registered DjangoType...")` (name `--schema` as the fix where appropriate); **two or more** → `CommandError` listing each candidate by `f"{type_cls.__module__}.{type_cls.__qualname__}"` + its `registry.model_for_type(type_cls)`, instructing the consumer to pass a dotted path. MUST NOT return the first match by iteration order.
   3. **Validate it is a `DjangoType` subclass.** `if not (isinstance(target, type) and issubclass(target, DjangoType)): raise CommandError("... is not a DjangoType subclass")`.
   4. **Two distinct finalized-state branches** (Decision 4 "Finalized-state contract"; DoD item 6). `definition = getattr(target, "__django_strawberry_definition__", None)`; if `definition is None` → `CommandError("... not a registered DjangoType")` (abstract / no-`Meta` base — distinct message). Else if `definition.finalized is False` → `CommandError("finalize_django_types() has not run — pass --schema <your project schema dotted path> so all types register and finalize")`. Keep the two messages separate.
   5. **Print the per-field table.** Iterate `definition.selected_fields` in order; for each field emit a row via `_row_for_field`. Print to `self.stdout` (so `call_command(..., stdout=out)` captures it). The exact column layout is an implementation detail (Decision 4: "every selected field, with its *resolved* GraphQL type and nullability, in selection order"); keep it human-readable, matching the illustrative table at spec lines 250-256.
   6. **`_row_for_field(definition, field)`** produces one row:
      - **Relation field** (`field_meta.is_relation`, from `definition.field_map[snake_case(field.name)]`): resolved GraphQL type read from `origin.__annotations__[field.name]` (the finalizer rewrote it to the concrete `list[T]` / `T` / `T | None`); converter classification from the cardinality — forward FK → "relation: forward FK", M2M / reverse FK → "relation: M2M" / "relation: reverse FK" (use `field_meta.relation_kind` / `is_many_side`). Nullability column from the annotation shape ("no (list)" for many-side, per spec table line 254-255).
      - **Relay-suppressed pk** (Decision 4; spec checklist line 109; spec table note line 258): when the type is Relay-Node-shaped and `field.name` is the suppressed pk (`source_model._meta.pk.name` on a `_is_relay_shaped` type — the field is **absent** from `origin.__annotations__`), report the interface-supplied `GlobalID!` and a "relay.Node id" converter row. **Do NOT index `origin.__annotations__[pk_name]`** — it would `KeyError`. (Detect Relay-shape via the type's interfaces / `relay.Node` membership the same way `types/base.py::_is_relay_shaped` does; reading `definition.interfaces` + an `issubclass(..., relay.Node)` check is the lightest in-command path. A non-Relay pk — e.g. `BookType.id` `BigAutoField` → `Int!` — is NOT suppressed and reads from `origin.__annotations__` like any other scalar.)
      - **Scalar field**: resolved GraphQL type + nullability from `origin.__annotations__[field.name]`; converter classification by re-walking `scalar_for_field(field)` / `SCALAR_MAP` MRO to NAME the row (`SCALAR_MAP[TextField]` etc.), and "choice enum" when `field.choices` is set (it resolves to the generated `<TypeName><FieldName>Enum`). Never re-run `convert_scalar` for nullability.
   - Imports to add: `from django.utils.module_loading import import_string`, `from strawberry.utils.importer import import_module_symbol`, `from ...registry import registry` (or the established relative depth — `management/commands/` is two packages below the root, so `from django_strawberry_framework.registry import registry` is cleanest and matches how a command module imports package internals), `from ...types.base import DjangoType`, and the converter/field-meta helpers as needed. Keep `import_string` (type arg) and `import_module_symbol` (`--schema`) as two **distinct** loaders — they are not interchangeable (Decision 4 P1: `import_string("config.schema")` fails on the empty `config` package; `import_module_symbol` reads the module's `schema` symbol).

### Implementation notes

- **GraphQL-type rendering.** The plan said "read the resolved GraphQL type from `origin.__annotations__`" but did not fix how to *stringify* it for the table. The raw annotations are Python objects (`int`, `str | None`, `list[GenreType]`, the generated enum class). I added `_render_annotation` (SDL-shaped: scalar → `Name!`, `T | None` → `Name`, `list[T]` → `[Inner!]!`) plus a `_GRAPHQL_SCALAR_NAMES` map (`int→Int`, `str→String`, `JSON`, `BigInt`, dates, etc.) mirroring Strawberry's SDL names, falling back to `__name__` for generated enums / resolved `DjangoType` relations. This is what produces `Int!` / `String!` / `[GenreType!]!` matching the spec's illustrative table — there was no shipped helper for "annotation → GraphQL string", so this is command-internal.
- **Nullability source.** Read from the annotation itself (`_annotation_is_optional` checks for a `T | None` union), NOT from `field.null` or a `convert_scalar` re-run — honors Decision 4 and is what makes the table reflect a future Slice-3 override.
- **Converter-row name for relations** uses `field_meta.relation_kind` (`forward_single` / `many` / `reverse_many_to_one` / `reverse_one_to_one`) prefixed `relation: `. The spec's illustrative wording was "relation: forward FK / M2M / reverse FK"; I used the package's own `relation_kind` classifier verbatim rather than re-deriving prose labels, so the row name stays grep-stable against the package vocabulary. Worker 1/3: flag if the spec wants the prose form instead.
- **Relay-shape detection** mirrors `types/base.py::_is_relay_shaped` exactly — `any(issubclass(i, relay.Node) for i in definition.interfaces) or issubclass(definition.origin, relay.Node)` — reading `definition.interfaces` (the lightest in-command path the plan named). The suppressed pk is detected as `field.name == definition.model._meta.pk.name`, matching `_build_annotations`'s `pk_name = source_model._meta.pk.name` guard.
- **`scalar_for_field(field)` is called for its side effect** in `_scalar_row` (result discarded): it NAMES nothing extra in the printed string (the row prints `SCALAR_MAP[<field-class>]`), but the call doubles as a guard that the field is actually `SCALAR_MAP`-resolvable, matching Decision 4's "re-walk the lookup to NAME which row fired" without re-running `convert_scalar`.
- **Cold-path mechanism (discretion item 1):** chose `sys.modules` eviction over subprocess — in-process, asserts on captured `call_command` stdout, no manage.py-path/subprocess-env fragility. - **`registry` import depth (discretion item 4):** used the absolute `from django_strawberry_framework.… import …` form for all four package-internal imports (matches how a `management/commands/` module two packages deep reads internals cleanly).
