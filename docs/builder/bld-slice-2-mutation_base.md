## Plan (Worker 1)

This slice ships **`django_strawberry_framework/mutations/sets.py`** (the `DjangoMutation` base + its metaclass + `Meta` validation + a mutation-declaration registry) and **`django_strawberry_framework/mutations/permissions.py`** (new ŌĆö `DjangoModelPermission`, the DRF-shaped default write-auth class + its exportable surface), wires the **phase-2.5 mutation bind** into `django_strawberry_framework/types/finalizer.py` (the live `# TODO(spec-036 Slice 2)` anchor at `finalizer.py #"bind ``DjangoMutation`` declarations"`), wires the **`registry.clear()` co-clear** into `django_strawberry_framework/registry.py` (the live `# TODO(spec-036 Slice 1-2)` anchor at `registry.py #"co-clear mutation generated-input globals"`), and adds `DjangoMutation` + `DjangoModelPermission` to the package-root public surface (`__init__.py` + `tests/base/test_init.py` `__all__` pin).

It builds directly on the Slice-1 substrate (`mutations/inputs.py`, accepted) and calls its dormant generators / lifecycle helpers from the bind for the first time. **No resolver, no `DjangoMutationField`, no permission *enforcement* lands here** ŌĆö those are Slice 3.

- **Duplication risk avoided.**
  - **Risk: routing the mutation bind through `_bind_sidecar_sets`** (`finalizer.py:1171-1247`). Avoided ŌĆö that driver is keyed off `registry.iter_definitions()` and `getattr(definition, spec.definition_attr)` (a `DjangoType`-definition sidecar attr like `orderset_class`). A `DjangoMutation` is NOT a `DjangoType` sidecar ŌĆö it is an independently-registered class with its own `Meta.model` (Decision 5: "the mutation's own `Meta`/method namespace, not a `DjangoType` `Meta` key"). The bind iterates the **mutation-declaration registry** (`iter_mutations()`), not type definitions. Forcing it into `_SidecarBindingSpec` would require faking a `definition_attr`/`helper_ledger`/`factory_cls`/`expand` for a non-sidecar ŌĆö a worse abstraction. Document `_bind_mutation` as a deliberate divergence (a sibling of `_bind_filtersets`/`_bind_ordersets` at the *placement* level, not a `_bind_sidecar_sets` consumer). Worker 3 must confirm this divergence is justified, not a missed reuse.
  - **Risk: re-implementing materialize / collision-raise inline in the bind.** Avoided ŌĆö reuse `materialize_mutation_input_class` (Slice 1, which wraps `utils/inputs.py::materialize_generated_input_class`) for the AR-M6 input-collision raise, and the same helper (or a one-ledger sibling) for the payload-collision raise. Zero new copies of the collision check.
  - **Risk: re-deriving the `<field>_id` / M2M python-attr scheme in `_validate_input_class`** separately from `relation_input_annotation`. Avoided ŌĆö drive the AR-M2 expected-attr set from the same `editable_input_fields` + `relation_input_annotation` the generator uses, so the validator's notion of "the generated scheme" cannot drift from the generator (one source of truth for the scheme).
  - **Risk: copying the `ALLOWED_META_KEYS` typo-guard set.** Avoided ŌĆö define a mutation-local allowed-key frozenset (`{"model", "operation", "input_class", "partial_input_class", "fields", "exclude", "permission_classes"}`) in `mutations/sets.py`; do NOT import or extend `types/base.py`'s sets (Decision 12: separate namespace; `DEFERRED_META_KEYS`/`ALLOWED_META_KEYS` byte-unchanged).
  - **Risk: a second request-resolution walk in `DjangoModelPermission`.** Avoided ŌĆö reuse `utils/permissions.py::request_from_info` (the read-side request resolver) rather than re-walking `info.context.request`.

### Implementation notes

- **Payload materialization reuses the input ledger (the plan's preferred one-ledger choice).** Payloads route through the SAME `materialize_mutation_input_class` (`mutations.inputs._materialized_names`) as inputs ŌĆö one ledger, one collision check, one `registry.clear()` co-clear. So the registry co-clear is the two blocks the plan named (input-namespace + declaration-registry); no third payload co-clear block.
- **Identical-shape dedupe needed a per-finalize build cache (`_shape_build_cache`), keyed `(model, operation_kind, fields, exclude)`.** Without it, two identical-shape create mutations each call `build_mutation_input` producing DISTINCT class objects with the same name ŌåÆ the materialize identity-check would (wrongly) raise. Decision 6 requires identical shapes to dedupe, so the bind caches the built class per shape and materializes the SAME object twice (idempotent no-op). This is a small mechanically-obvious refinement of the plan's "call `build_mutation_input` for the input shape" step ŌĆö flagged to Worker 1 below. AR-M6 distinct-shape collisions still raise (the cache key differs, so distinct shapes get distinct objects, and a custom `input_class` under a generated name is a distinct object ŌåÆ raise).
- **`check_permission` returns `bool`** (False on first denying class, True if all allow); the default delegates to `Meta.permission_classes`. Slice 3's resolver maps a `False` to a raised `GraphQLError`. Defined here, not invoked.
- **`_resolve_model` is a classmethod on `DjangoMutation`** (the most subclass-overridable shape, per the discretion item), so the `038`/`039` flavors override it. `DjangoModelPermission.has_permission` calls `mutation._resolve_model(mutation.Meta)` so the flavors authorize through the same default.
- **`_ValidatedMutationMeta` is a `__slots__` record (not a dataclass)** to stay dependency-light, mirroring `types/base.py::_ValidatedMeta`'s role (validate once at class creation, downstream readers trust the snapshot).
- **The metaclass skips a class with no nested `Meta`** (`attrs.get("Meta") is None`) ŌĆö the abstract `DjangoMutation` base and any intermediate base are not concrete mutations, the same in-flight-base guard the set metaclasses use.
- **`bind_mutations()` clears `_shape_build_cache` at pass start** so a stale class from a prior (failed or re-run) finalize never leaks; the cache is a build-pass scratch, not a persistent ledger.
