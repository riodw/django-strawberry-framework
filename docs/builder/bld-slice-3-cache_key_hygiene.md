## Plan (Worker 1)

Slice 1 bakes resolved nested-pagination values (`first` / `last` / `before` / `after`) into windowed prefetch querysets (`bld-slice-1-plan_foundation.md` step 8(e): `SliceMetadata.from_arguments(...)` over the converted selection's resolved arguments, applied via `apply_window_pagination`); Slice 2 consumes the window. ### DRY analysis

**Existing patterns reused (cite `path::Symbol` and shadow line):**

- **Reuse the directive-variable collection machinery, do NOT fork it.** `optimizer/extension.py::_collect_directive_var_names` (shadow lines 89-107) returns a `frozenset[str]` of variable names referenced in `@skip` / `@include`; `_build_cache_key` (shadow lines 878-971, #"directive_var_names = _collect_directive_var_names(") folds those names' VALUES into `relevant_vars` via #"relevant_vars = frozenset(" (`(k, variable_values[k]) for k in directive_var_names if k in variable_values`). The pagination collector is the **structural twin** of `_collect_directive_var_names`: same input (`operation`, `fragments`), same output (`frozenset[str]` of variable names). The key design rule (Decision 7 line 342): collect names referenced in `first` / `last` / `before` / `after` of **non-root** field nodes and **fold into the existing `relevant_vars` frozenset** — union the pagination-name set with `directive_var_names` BEFORE building `relevant_vars`, so the existing one-line `frozenset((k, variable_values[k]) …)` comprehension covers both families unchanged. One name→value resolution path, not two.
- **Reuse the AST-child + fragment-spread primitives.** `optimizer/extension.py::_child_selections` (shadow 149-163) centralizes the `getattr(node, "selection_set", None)` + `selections or ()` shape; `optimizer/extension.py::_unvisited_fragment_definition` (shadow 166-193) resolves a `FragmentSpreadNode` to its definition once, mutating a shared `visited_fragments` cycle guard. The pagination walker reuses BOTH — it is a third walker over the same three AST shapes (`_walk_directives` / `_walk_reachable_fragment_definitions` are the existing two; `_child_selections`'s docstring already names this as the shared "iterate children" seam). The new walker differs from `_walk_directives` in exactly one axis: it tracks **response-path depth** (root vs non-root) to gate collection, where `_walk_directives` collects unconditionally.
- **Reuse the AST node imports.** `DirectiveNode`, `FragmentSpreadNode`, `VariableNode` are already imported at `optimizer/extension.py` #"from graphql.language.ast import" (shadow lines 39-43). The pagination collector needs `VariableNode` (to detect a variable-valued argument) and `FieldNode` (to scope collection to field nodes / non-root). `FieldNode` and `ArgumentNode` are NOT yet imported — Worker 2 adds them to the same `from graphql.language.ast import (...)` block (the only net-new import this slice).
- **Reuse the per-execution memo shape.** `optimizer/extension.py` #"_printed_ast_cache: ContextVar[dict[int, str]" (shadow lines 354-357) is the existing per-execution memo for the rendered document key, installed in `on_execute` (#"ast_token = _printed_ast_cache.set({})", shadow 588) and reset in its `finally` (shadow 592), keyed by `id(operation)`. Decision 7 line 348 / 954-957 require the directive + pagination variable-NAME sets to be memoized per `id(operation)` in this same per-execution style, so nested fallback connection pipelines (which call `apply_connection_optimization` → `_get_or_build_plan` → `_build_cache_key` per parent row) don't re-walk the full AST per parent. The new memo is a sibling `ContextVar[dict[int, frozenset[str]] | None]` set in `on_execute` next to the printed-AST one and reset in the same `finally`.

**New helpers justified (single responsibility each):**

1. `optimizer/extension.py::_collect_nested_pagination_var_names(operation, fragments) -> frozenset[str]` — single responsibility: return the set of variable names referenced in `first` / `last` / `before` / `after` arguments on **non-root** field nodes (response-path depth ≥ 1), following fragment spreads with response-depth preserved at the spread site. Mirrors `_collect_directive_var_names`'s public shape (frozenset of names, takes `operation` + `fragments`). Sole call site: `_build_cache_key`, which unions its result with `directive_var_names`.
2. `optimizer/extension.py::_walk_pagination_vars(node, names, fragments, visited_fragments, depth)` — single responsibility: the recursive workhorse for #1, the depth-tracking twin of `_walk_directives`. Sole caller: `_collect_nested_pagination_var_names`. (Mirrors the `_collect_*` / `_walk_*` public/private split the directive and reachable-fragment families already use — see `_walk_directives` / `_walk_reachable_fragment_definitions`.)
3. `optimizer/extension.py::_pagination_var_names_cache: ContextVar[dict[int, frozenset[str]] | None]` (module-level) PLUS one combined-name-set memo lookup inside `_build_cache_key` — single responsibility: cache the **combined** directive+pagination variable-name frozenset per `id(operation)` so the per-parent nested-fallback path computes the union once per operation, not once per row. Spec line 348 phrases it as "derive directive and pagination variable name sets from one reachable-fragments traversal and store them on the extension's execution-local memo keyed by `id(operation)`." See "Implementation discretion items" for the one-memo-vs-two-memo shape — the observable contract (one full-AST walk per operation, not per parent) is fixed; the dict layout is Worker 2's call.

The directive collector, child/fragment primitives, AST imports, memo shape, and `on_execute` lifecycle are all **reuse / mirror**, not new infrastructure. The only genuinely new logic is the depth-gated pagination-name walk (#1/#2) and its memo (#3).

**Duplication risk avoided:**

- **A second full-AST traversal per parent row.** The naive shape calls `_collect_directive_var_names(operation)` AND a new `_collect_nested_pagination_var_names(operation)` on every `_build_cache_key` call — two full walks, and the nested fallback connection path calls `_build_cache_key` once per parent row. Decision 7 line 348/954-957 forbids this. Avoided by memoizing the COMBINED name frozenset per `id(operation)` (helper #3): the first call for an operation does the walk(s) and stores the union; every subsequent call (including per-parent fallbacks) reads the memo. The printed-AST memo (`_printed_ast_cache`) already proves this exact pattern in the same method, so the var-name memo is a direct sibling, not a new idea.
- **Two pagination-argument-name string sets.** The literals `"first"` / `"last"` / `"before"` / `"after"` must be a single module-level constant (e.g. `_PAGINATION_ARG_NAMES = frozenset({"first", "last", "before", "after"})`), referenced by the walker, NOT re-spelled inline (the shadow Repeated-string-literals section is the integration-pass DRY gate; a 4-name tuple spelled twice is a finding). The same constant is the obvious anchor a future `search:` extension (out of scope, `0.1.2`) would extend.
- **A third fragment-walk that re-implements cycle detection.** Risk: Worker 2 writes a bespoke fragment-spread descent instead of reusing `_unvisited_fragment_definition` + `_child_selections`. Avoided by reusing both primitives (their docstrings already name themselves as the shared seam for "the three AST walkers in this module"). The pagination walker becomes the third member of that family, not a fourth ad-hoc walk.
- **Forking the name→value fold.** Risk: building a separate `pagination_relevant_vars` frozenset alongside `relevant_vars` and concatenating the two tuples into the key. Avoided per Decision 7 line 342 ("fold into the existing `relevant_vars` frozenset"): union the NAME sets, then run the SINGLE existing `(k, variable_values[k]) …` comprehension over the combined names. The key tuple shape stays `(doc_key, relevant_vars, target_model, runtime_path, origin)` — one more contribution to one existing frozenset, no new key position (preserves the memoized printed-AST fast path and every existing key-shape test).

3. **`optimizer/extension.py::_collect_nested_pagination_var_names` + `_walk_pagination_vars` — the depth-preserving collector (new module-level functions, sited beside `_collect_directive_var_names` / `_walk_directives`).** Implement helper #1/#2:
   - `_collect_nested_pagination_var_names(node, fragments=None) -> frozenset[str]`: the public entry, mirroring `_collect_directive_var_names`'s body — `names: set[str] = set()`; `_walk_pagination_vars(node, names, fragments or {}, set(), depth=0)`; `return frozenset(names)`.
   - `_walk_pagination_vars(node, names, fragments, visited_fragments, depth)`: the recursive workhorse, modeled on `_walk_directives` (shadow 110-146) but **depth-gated**:
     - **Collect at this node only when it is a non-root field node** — i.e. `depth >= 1` (the operation node is depth 0; its direct selection-set children are the root fields, also conceptually root). The precise gate: a field node's pagination arguments are collected when the field sits at **response-path depth ≥ 1** (nested), NOT at the root field level. Worker 2 pins the exact depth bookkeeping against the fragment tests in step-4 / Test additions; the contract (Decision 7 line 346) is "track depth at the spread SITE, not raw fragment-definition nesting." See the depth-bookkeeping note below.
     - **Argument sweep** (only when collecting at this node): for each argument on the field node (`getattr(node, "arguments", ()) or ()`), when `arg.name.value in _PAGINATION_ARG_NAMES` AND `isinstance(arg.value, VariableNode)`, `names.add(arg.value.name.value)` — the structural twin of `_walk_directives`'s #"if isinstance(arg.value, VariableNode): names.add(arg.value.name.value)" directive sweep, swapping the directive filter for the pagination-arg-name filter.
     - **Recurse into children with incremented depth**, reusing `_child_selections(node)` (shadow 149-163) and `_unvisited_fragment_definition(child, fragments, visited_fragments)` (shadow 166-193) exactly as `_walk_directives` does (shadow 138-146). The recursion into a child field node passes `depth + 1`; the recursion into a resolved fragment definition passes the **spread-site depth** (the depth of the spread, not depth + 1 for the fragment-definition wrapper) so a fragment's field selections are scored at the depth the spread occupies — this is the "depth at the spread SITE, not raw fragment-definition nesting" rule (Decision 7 line 346).
   - **Depth-bookkeeping note (the load-bearing subtlety).** The operation node and an inline/named fragment on `Query` spread at the root are response-depth 0 — their direct field children are root fields (root pagination args, stay OUT). A field child of a root field is depth ≥ 1 (nested, collected). When a `FragmentSpreadNode` / `InlineFragmentNode` is encountered, the fragment's body inherits the spread site's depth: a fragment spread at the root contributes root-depth fields; a fragment spread inside a nested node contributes nested-depth fields. Concretely: increment depth when descending into a FIELD node's selection set; do NOT increment when descending into a fragment definition / inline fragment (a fragment is a transparent wrapper at its spread site). Worker 2 verifies this against `test_root_fragment_pagination_variable_shares_cache` (root connection through a `Query` fragment → name NOT collected) and `test_fragment_carried_nested_pagination_variable_collected` (nested connection through a parent-node fragment → name collected). **Hotspot caution:** `_walk_directives` is already a control-flow hotspot (37 lines / 10 branches — shadow line 79); `_walk_pagination_vars` is a NEW function (does not inflate the existing hotspot), but keep its branch count lean — the depth gate + arg sweep + child recursion is the whole function. Do not merge it INTO `_walk_directives` (two different collection axes: directives-everywhere vs pagination-non-root — merging would couple them and require a flag parameter).

### Implementation discretion items

These are assessed and decided to be Worker 2's choice (equivalent-shape / siting / dict-layout):

- **One combined memo vs two separate memos.** Decision 7 line 348/954-957 fixes the contract: ONE reachable-fragments traversal per operation, var-name sets stored on an execution-local memo keyed by `id(operation)`, no second full-AST walk on the per-parent path. Whether Worker 2 (a) stores the COMBINED directive+pagination frozenset in a single `_pagination_var_names_cache[id(operation)]` (cleanest — `_build_cache_key` reads one frozenset), or (b) keeps two memo dicts (directive names, pagination names) and unions on read, or (c) extends the existing `_printed_ast_cache` into a small per-operation record carrying both the doc key AND the combined name set — is Worker 2's call. Shape (a) is the plan's recommended default (one read, matches "store them on the … memo"); (c) risks coupling two concerns into one ContextVar. The observable contract (one walk per operation, memo tolerates `None` outside `on_execute`) is fixed.
- **`ArgumentNode` import necessity.** Whether the walker needs `ArgumentNode` as an explicit `isinstance` target or simply iterates `node.arguments` and reads `arg.name.value` / `arg.value` (graphql-core guarantees `ArgumentNode` shape on `FieldNode.arguments`) is Worker 2's call. The directive sweep at `_walk_directives` reads `arg.value` directly without an `ArgumentNode` isinstance, so the same untyped read is consistent — `ArgumentNode` may not need importing at all. `VariableNode` is definitely needed.
- **Depth representation.** Whether `depth` is an `int` counter incremented per field-node descent, or a boolean `is_root_level` flag flipped after the first field-node hop, or carried as the response-path tuple length — Worker 2 picks the leanest representation that satisfies "depth at the spread site, not raw fragment-definition nesting." An `int` depth (operation = 0, root fields = 0, their children = 1+) is the plan's recommended default; the fragment tests are the contract.
- **Constant siting and the pagination-arg-name set name.** `_PAGINATION_ARG_NAMES` vs another name; placement near `_MAX_PLAN_CACHE_SIZE` vs just above the collector. The value (`frozenset({"first", "last", "before", "after"})`) and single-source-of-truth requirement are fixed.
- **Test idiom per pin** — direct `_build_cache_key` identity vs schema-execute counters. Both are established in the file; Worker 2 picks whichever reads cleanest per assertion (the recommendations above are defaults, not mandates).
