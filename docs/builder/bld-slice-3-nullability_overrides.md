### DRY analysis

- **Existing patterns reused.**
  - **Validator-helper shape.** `_validate_nullability_override_targets(...)` mirrors `django_strawberry_framework/types/base.py::_validate_optimizer_hints` (the closest structural template — it also takes the selected `fields` tuple + `model`, derives a model-wide name set and a selected-subset name set, and raises `ConfigurationError` with the shared `_format_unknown_fields_error` for the unknown path). The `_validate_filterset_class`/`_validate_orderset_class` pair (base.py:72-123, cited by spec as the structural template) is the template for the *`_validate_meta`-resident shape check* — both return a normalized value (or `None`) and raise `ConfigurationError` for a bad shape; the new shape-check returns the normalized `frozenset` pair onto `_ValidatedMeta`.
  - **Shape-normalize helper.** `_normalize_sequence_spec` (base.py:403-409) already implements the exact "non-string sequence of field names, else `ConfigurationError`" guard the spec demands (Edge case lines 567, mirroring the `Meta.exclude` guard `base.py #"Meta.exclude must be a non-string sequence"`). Reuse it verbatim for both override keys — do NOT hand-roll a second sequence guard. (Note: its error message hardcodes "Meta.exclude"; see Implementation discretion item 1.)
  - **Unknown-field error formatting.** `_format_unknown_fields_error` (base.py:494-508) is the shared "names unknown fields … Available: …" shape used by `Meta.fields`/`Meta.exclude`/`Meta.optimizer_hints`. The *unknown* error path of the target-validator should route through it so the consumer-visible shape stays consistent.
  - **Consumer-authored / relation / selected-set inputs already computed.** `__init_subclass__` already computes `consumer_authored_fields` (the four-corner union, base.py:260-267), the selected `fields` tuple (`_select_fields`, used for the field_map), and `relay_shaped` (base.py:268). `field.is_relation` is the relation flag (used at `_build_annotations` line 947). The validator consumes these as inputs — it derives NONE of them itself, so no parallel computation.
  - **The Relay-pk suppression state.** `_build_annotations` computes `pk_name = source_model._meta.pk.name if suppress_pk_annotation else None` (base.py:945) and `suppress_pk_annotation = _is_relay_shaped(cls, interfaces)`. The validator's *Relay-pk reject* (rule e) uses the SAME `_is_relay_shaped(cls, validated.interfaces)` result (already bound as `relay_shaped` at base.py:268) plus `meta.model._meta.pk.name` — one source of truth for "is this the suppressed pk."
  - **Tri-state widening seam.** `convert_scalar`'s three existing `if field.null:` widening sites become the three `if effective_null:` sites — one computed value applied uniformly. No new conversion entry point (Decision 7 rejects a `convert_scalar_with_override` wrapper).
  - **Acceptance type / reload fixture / SDL introspection.** The live HTTP test reuses `examples/fakeshop/test_query/test_library_api.py`'s autouse `_reload_library_project_schema` fixture (lines 33-64), `_seed_library_graph` (line 67), `_post_graphql`/`_assert_graphql_data`/`_field_type` helpers (lines 90-107), and the exact `__type(name: ...) { fields { name type { kind name ofType {...} } } }` introspection shape already used by `test_library_choice_enum_and_nullable_subtitle_are_deliberate_http_contracts` (lines 368-393). The new acceptance type mirrors the scaffolded `NullabilityOverrideBookType` pseudo at `apps/library/schema.py #"TODO(spec-029 Slice 3): Add the nullability-override acceptance type"`.
  - **Converter test doubles.** `tests/types/test_converters.py` already ships `_FakeArrayField`/`_FakeHStoreField` doubles + a `monkeypatch.setattr(converters, "_ARRAY_FIELD_CLS"/"_HSTORE_FIELD_CLS", ...)` pattern, non-null/nullable `TextField` subclass fixtures, and choice fixtures — the `force_nullable` tri-state tests extend these, not new fixtures.

- **New helpers justified.**
  - **`_validate_nullability_override_targets(...)`** — single responsibility: given the normalized override frozensets + the selected fields + `consumer_authored_fields` + the model + the Relay-pk identity, raise `ConfigurationError` for every unknown / excluded / consumer-authored / relation / Relay-suppressed-pk target. Single call site (`__init_subclass__`, after `_select_fields`/`consumer_authored_fields`/Relay-shape). Justified because inlining into the 130-line `__init_subclass__` hotspot would push it past complexity thresholds (review_inspect finding) and because the five error paths read cleanly as one cohesive guard. It does NOT recompute the selected set, the consumer set, or the Relay state — those are passed in.
  - **No second new helper.** The `_validate_meta` shape/normalize/collision step and the `_build_annotations` `force_nullable` computation are small enough to live inline at their existing sites (3-5 lines each), reusing `_normalize_sequence_spec`. Extracting them would be premature; the condition that would justify a `_validate_override_shapes` helper later is a third override-key family appearing, at which point the normalize+collision logic could be hoisted.

- **Duplication risk avoided.**
  - **Risk: re-deriving the selected-field name set inside the validator.** Avoided — pass the already-computed `fields` tuple and derive the two name sets (`model._meta.get_fields()` for *unknown*, `{f.name for f in fields}` for *excluded*) locally as read-only sets, exactly as `_validate_optimizer_hints` does, not re-running `_select_fields`.
  - **Risk: collapsing *unknown* and *excluded* into one path** (which would weaken the `Meta.exclude` contract — spec Decision 8 explicitly forbids). Avoided — two distinct name sets, two distinct error messages, checked unknown-first then excluded.
  - **Risk: forwarding the outer `force_nullable` into the recursive `ArrayField.base_field` call** (`convert_scalar(field.base_field, type_name)` at converters.py:213). Avoided — the recursion stays `force_nullable`-unset (default `None`), so the inner element nullability follows `base_field.null` and only the OUTER `list[inner]` honors `effective_null` (spec Edge case lines 563 — "the inner element nullability follows base_field.null and is NOT affected by the outer override").
  - **Risk: a parallel nullability-stripping path at the `_build_annotations` call site** (unwrap `T | None`). Avoided by Decision 7 — the tri-state computes widening *before* it happens; nothing is unwrapped.
  - **Risk: persisting the override sets on `DjangoTypeDefinition`** (a new slot). Avoided per Non-goals (lines 167) / Decision 7 — the resolved annotation on `cls.__annotations__` IS the authoritative record; no definition slot is added.
