## Plan (Worker 1)

This is the **largest, highest-risk slice**. It ships the two reserved scaffolds:

- **`django_strawberry_framework/mutations/resolvers.py`** (replaces the TODO scaffold) — the sync + async create / update / delete write pipeline: decode → locate → authorize → validate → write → re-fetch/snapshot → payload, inside one `transaction.atomic()`, with the async path running the whole sync ORM pipeline in a single `sync_to_async(thread_sensitive=True)` call (AR-M4).
- **`django_strawberry_framework/mutations/fields.py`** (replaces the TODO scaffold) — `DjangoMutationField(MutationClass)`, the write-side sibling of `DjangoConnectionField`: synthesizes the per-operation resolver signature (`data:` + `id:` per operation), returns `strawberry.field(...)` typed via a `strawberry.lazy` forward-ref to the generated `<Name>Payload`, with **no class-attribute annotation** (Major-3 / Decision 7).

Plus the third net-new public export: `DjangoMutationField` added to `__init__.py` imports + `__all__` + the `tests/base/test_init.py` `__all__` pin.

It builds directly on the accepted Slice 1 substrate (`mutations/inputs.py` — generators, `FieldError`, `<Name>Payload`, `INPUTS_MODULE_PATH`, `NON_FIELD_ERROR_KEY`, `payload_object_slot`) and the accepted Slice 2 base (`mutations/sets.py` — `DjangoMutation`, the validated `_mutation_meta` snapshot, `check_permission`, the bind-stashed `_primary_type` / `_input_class` / `_payload_type_name`, `_VALID_OPERATIONS`; `mutations/permissions.py::DjangoModelPermission`). Slice 2 left the base **inert** (registered + bound, never resolved); this slice is what makes a `DjangoMutation` *resolve*.

### DRY analysis

- **Existing patterns reused (cite file:line).**
  - **The `is_async_callable` (construction-time) / `in_async_context()` (runtime) async-detection asymmetry** — `list_field.py::DjangoListField` (lines 161-197) and `connection.py::_build_connection_resolver` (lines 965-1033). `DjangoMutationField`'s default synthesized resolver has **no consumer `resolver=` seam** (a mutation pipeline is package-owned; unlike list/connection there is no consumer-supplied base resolver in 0.0.11), so the relevant precedent is the *default-resolver* branch: dispatch at **runtime via `in_async_context()`** so one factory output works under both `schema.execute_sync` and `await schema.execute` — mirroring `relay.py::DjangoNodesField._resolve` (lines 375-412), which is the closest analogue (a package-synthesized resolver with no consumer seam that picks sync-vs-async per call via `in_async_context()`). Reuse that runtime-dispatch shape verbatim; do **not** re-derive the predicate. (Note: the spec's "`is_async_callable` construction-time / `in_async_context()` runtime asymmetry" wording is the *general* `DjangoListField` contract; for `DjangoMutationField` only the runtime half applies because there is no consumer resolver to inspect at construction. Flag this to Worker 3 — it is the correct reading of Decision 8 line 368 "its default generated resolver dispatches at runtime via `in_async_context()`", not a divergence.)
  - **The relation `GlobalID` decode + `SyncMisuseError` discipline** — `relay.py` (the whole module) + `types/relay.py::decode_global_id` (lines 634-747). The mutation decode of a `<field>_id` that is a `relay.GlobalID` reuses **`decode_global_id`** to get `(target_type, node_id)`, then **type-checks `target_type` against the relation's Django target model** (AR-H4) — the same identity check `relay.py::_check_typed_match` (lines 139-154) does for the typed node field, but against the *relation target model* not a node-field target. A wrong-type id → `FieldError` on that relation field (NOT a `GraphQLError`, NOT a cross-model pk lookup, NOT a raw `DoesNotExist`). Reuse `decode_global_id`; do not re-parse GlobalIDs. For the raw-pk relation case (non-Relay target), the input field is already the raw pk scalar (Slice-1 `relation_input_annotation`), so no GlobalID decode runs — the value is the pk directly.
  - **The `get_queryset` visibility contract for the update/delete locate** — `utils/querysets.py::{initial_queryset, apply_type_visibility_sync, apply_type_visibility_async}` (lines 58-150) and the `_resolve_node_default` locate shape `types/relay.py::_resolve_node_default` (lines 787-820: `apply_type_visibility_sync(cls, initial_queryset(cls), info)` → `_apply_node_filter` → `.get()`/`.first()`). The mutation update/delete locate is `apply_type_visibility_sync(target_type, initial_queryset(target_type), info).get(pk=<decoded id>)` (Decision 10 line 407) — a miss (`DoesNotExist`) → not-found `FieldError` on `id`, **indistinguishable from a hidden row** (no existence leak). Reuse the visibility helpers; this is the *same sync-misuse site* (an `async def get_queryset` met on the sync path raises `SyncMisuseError`, coroutine closed first — `apply_type_visibility_sync` already does this). The async pipeline runs the whole thing inside one `sync_to_async`, so it calls the **sync** `apply_type_visibility_sync` inside the worker thread (AR-M4) — a sync mutation meeting an `async def get_queryset` raises `SyncMisuseError` regardless of operation context, exactly as the cascade helper does (`permissions.py` module docstring lines 30-36).
  - **The optimizer post-write re-fetch cooperation seam** — `optimizer/extension.py::apply_connection_optimization` (lines 1131-1179). This is the **exact** seam the mutation re-fetch reuses: it takes `(target_type, queryset, info)`, resolves the model, discovers the active optimizer via the `_active_optimizer` ContextVar, and delegates to `optimizer.apply_to(...)`. `apply_to` honors the G2 gate via the operation type (`walker.py::_enable_only_for_operation`), so under a `MUTATION` operation the re-fetch automatically keeps `select_related`/`prefetch_related` and applies **no** `.only(...)` deferral — **Decision 9 / G2 comes for free** by routing through this seam. It returns the queryset unchanged when no optimizer is installed (the consistent opt-in contract). The re-fetch queryset itself is `initial_queryset(target_type).filter(pk=<written pk>)` — **by pk WITHOUT the visibility `get_queryset` filter** (Medium-1, the deliberate GOAL crit-4 exception). Reuse `apply_connection_optimization`; do NOT call `_optimize` (that one infers the model from `info.return_type`, which is the payload type, not the node type — `apply_connection_optimization` takes the type explicitly, which is what we need, the same reason the connection field uses it).
  - **`request_from_info` for the write-auth user** — already single-sourced inside `DjangoModelPermission.has_permission` (`mutations/permissions.py` line 79). The resolver does **not** re-resolve the request; it calls `mutation_instance.check_permission(info, operation, data, instance)` (`sets.py` line 434), which delegates to `permission_classes` (already wired in Slice 2). The resolver only maps a `False` return → a raised `GraphQLError`. Reuse the Slice-2 `check_permission` method; do not re-walk `permission_classes` in the resolver.
  - **The synthesized-signature + `strawberry.field(resolver=...)` factory idiom** — `connection.py::_synthesized_signature` (lines 900-962) + `connection.py::DjangoConnectionField` (lines 1171-1221) and the simpler `relay.py::DjangoNodeField` (lines 260-312, which declares `id: strawberry.ID` as a raw string deliberately so malformed ids reach the package). `DjangoMutationField` synthesizes a signature with `root`/`info` (Strawberry-reserved, not GraphQL args) + the operation's GraphQL args: `data: <input> ` (`!` non-null) for create/update, `id: <id-type>` for update/delete. Reuse the `inspect.Signature` + `__signature__`/`__annotations__`-on-the-resolver shape from `_synthesized_signature`; the return annotation is the `strawberry.lazy("django_strawberry_framework.mutations.inputs")` forward-ref to `<Name>Payload` (Decision 7 line 354).
  - **The four-guard target validation** — `list_field.py::_validate_djangotype_target` is for *DjangoType* targets; `DjangoMutationField` takes a *DjangoMutation* subclass, so the guard is **different** (validate it is a concrete `DjangoMutation` subclass with a bound `_mutation_meta`, not a `DjangoType`). This is a small new guard, not a reuse — flag it (see "New helpers justified"). It mirrors the *spirit* of the connection/list guards (fail at the `DjangoMutationField(...)` line, not at finalize) but over a different target contract.
  - **`transaction.atomic` + `sync_to_async(thread_sensitive=True)`** — `permissions.py::aapply_cascade_permissions` (lines 284-306) and `filters/sets.py` (line 1690) are the package's `sync_to_async(thread_sensitive=True)` precedents (wrap one sync function, run it off the event loop on one thread). `django.db.transaction.atomic` is not yet used in the package (grep confirms zero prior use) — it is standard Django, imported `from django.db import transaction`. The async path is `await sync_to_async(_run_pipeline_sync, thread_sensitive=True)(...)` where `_run_pipeline_sync` wraps steps 3-6 in `with transaction.atomic():` — so the whole transaction runs on one thread (AR-M4), never interleaving ORM calls with `await`s.
  - **`_VALID_OPERATIONS` / the operation constants** — `mutations/sets.py::_VALID_OPERATIONS` (line 81), `_OPERATION_INPUT_KIND` (line 88). The resolver imports `_VALID_OPERATIONS` rather than re-spelling `{"create","update","delete"}` (Slice 2 line 80 explicitly reserves it for Slice 3's reuse). The resolver dispatches on `meta.operation` to the create/update/delete pipeline branch.
- **New helpers justified (single responsibility each).**
  - `mutations/resolvers.py::resolve_mutation_sync(mutation_cls, info, *, data=UNSET, id=UNSET) -> payload` — **single responsibility: the sync pipeline entry**, dispatching on `meta.operation` and running decode→locate→authorize→validate→write→re-fetch→payload inside one `transaction.atomic()`. One call site: the `DjangoMutationField` default resolver's sync branch.
  - `mutations/resolvers.py::resolve_mutation_async(mutation_cls, info, *, data=UNSET, id=UNSET) -> payload` — **single responsibility: the async pipeline entry**, `await sync_to_async(<the sync body>, thread_sensitive=True)(...)` (AR-M4). One call site: the field's async branch. It does NOT re-implement the pipeline — it wraps the same sync body so the transaction + ORM run on one thread.
  - `mutations/resolvers.py::_decode_relations(meta, model, data) -> (field_value_map, FieldError | None)` — **single responsibility: Decision-8 step 1 decode.** Walk the input dataclass fields; strip `UNSET`; for each `<field>_id` that is a `relay.GlobalID`, `decode_global_id(...)` then type-check the decoded `target_type._mutation`-... no — type-check the decoded target model against the relation's Django target model (`field.related_model`); a wrong-type id → return a `FieldError` on that relation field (AR-H4). Distinguish `UNSET`/`null`/value tri-state per field (Edge cases line 502). Returns the decoded `{model_attr: value}` map (FK as `<field>_id`→pk, M2M as a list of pks deferred to the write step) plus an optional decode `FieldError`. Single call site: the create/update build step.
  - `mutations/resolvers.py::_locate_instance(target_type, decoded_id, info) -> (instance | None)` — **single responsibility: Decision-10 visibility-scoped lookup** for update/delete. `apply_type_visibility_sync(target_type, initial_queryset(target_type), info).get(pk=...)`; `DoesNotExist` → `None` (caller maps to a not-found `FieldError` on `id`). Single call site: the update/delete branches. Reuses the `_resolve_node_default` locate shape.
  - `mutations/resolvers.py::_validation_error_to_field_errors(exc) -> list[FieldError]` — **single responsibility: map a Django `ValidationError` to the envelope** (Decision 7 / Decision 8 step 4). Uses `exc.error_dict` when present (per-field), keying the model's `NON_FIELD_ERRORS` bucket to the **`NON_FIELD_ERROR_KEY` sentinel** (`mutations/inputs.py::NON_FIELD_ERROR_KEY`, already pinned to `"__all__"` — AR-M3, single source); falls back to `exc.messages` under the sentinel for a non-dict `ValidationError`. Single source for both the `full_clean()` failure and the `IntegrityError`-race fallback mapping. Single call site each in the validate/write steps.
  - `mutations/resolvers.py::_unprovided_exclude(model, instance, provided_attrs) -> list[str]` — **single responsibility: AR-H2 partial-update `exclude` computation.** The set of model fields the `PartialInput` did **not** provide, **minus any unprovided field co-participating in a `UniqueConstraint` / `unique_together` / `unique` check with a *provided* field**. So a `name`-only update still validates `unique_item_per_category` (the unprovided `category` is co-constrained with the provided `name`, so it is NOT excluded). Reads `model._meta.constraints` (filter to `UniqueConstraint`), `model._meta.unique_together`, and per-field `field.unique`. Single call site: the update validate step. This is the highest-subtlety helper — pin it directly.
  - `mutations/resolvers.py::_assign_m2m(instance, m2m_values) -> None` — **single responsibility: AR-M1 M2M write semantics.** For each provided M2M field: `instance.<m2m>.set([...])` (replace-on-provide); a provided empty list clears it; an omitted (`UNSET`) field is never in the map so it is unchanged. Related objects resolve through the target model's default manager (the pks decoded in step 1). Runs inside the transaction, after `save()`. Single call site: the write step.
  - `mutations/resolvers.py::_refetch_optimized(target_type, pk, info) -> instance | None` — **single responsibility: Decision-9 by-pk re-fetch + optimizer plan.** `qs = initial_queryset(target_type).filter(pk=pk)` (NO visibility filter — Medium-1), `qs = apply_connection_optimization(target_type, qs, info)`, then `qs.first()` (create/update) — the G2 gate fires automatically because the operation is `MUTATION`. For **delete**, the snapshot is fully materialized (relations loaded) **before** `delete()` (AR-M5 / Medium-2) — see the delete branch.
  - `mutations/resolvers.py::_build_payload(payload_cls, slot, obj, errors) -> payload` — **single responsibility: instantiate the `<Name>Payload`** with the object in the uniform `node`/`result` slot + `errors`. Reads `payload_object_slot` indirectly via the stashed slot (or recompute via `payload_object_slot(primary_type)` — discretion item). Single source for the success and error-envelope returns.
  - `mutations/fields.py::DjangoMutationField(mutation_cls, *, description=None, deprecation_reason=None, directives=()) -> Any` — **single responsibility: the field factory.** Validates the target (concrete `DjangoMutation` subclass with `_mutation_meta`), synthesizes the per-operation signature, builds the runtime-dispatch resolver, returns `strawberry.field(resolver=..., ...)`.
  - `mutations/fields.py::_validate_mutation_target(mutation_cls)` — **single responsibility: the field-construction guard.** Reject a non-`DjangoMutation`, a non-class, or the abstract base (no `_mutation_meta`) with `ConfigurationError` naming `DjangoMutationField` (fail at the construction line). The mutation must be a *concrete, validated* subclass; the bind has NOT necessarily run yet at field-construction time (the field is assigned on `@strawberry.type Mutation` which evaluates at import, before `finalize_django_types()`), so the guard checks `_mutation_meta is not None` (set at class creation) — it does NOT require `_input_class` / `_payload_type_name` (those are bind outputs, populated at finalize). Single call site: the factory head.
  - `mutations/fields.py::_synthesized_mutation_signature(meta) -> (Signature, annotations)` — **single responsibility: per-operation argument synthesis.** `create`: `data: <Model>Input!`; `update`: `id: <id-type>`, `data: <Model>PartialInput!`; `delete`: `id: <id-type>` only (Decision 14). The `data:` annotation is a `strawberry.lazy(INPUTS_MODULE_PATH)` forward-ref to the input class name (the input is materialized at finalize, same timing hazard as the payload — so a lazy ref, NOT a direct class reference, because the field is constructed at import before the bind). The `id:` type is `relay.GlobalID` when the primary is Relay-Node-shaped else the raw pk scalar — but the primary type is a *bind output* (`_primary_type`), unknown at construction... see "Implementation discretion items" (the resolver-signature timing). The **return** annotation is `strawberry.lazy(INPUTS_MODULE_PATH)` forward-ref to `<Name>Payload` (Decision 7). Mirror `connection.py::_synthesized_signature` shape.
- **Duplication risk avoided.**
  - **Risk: re-deriving the GlobalID decode / wrong-type check.** Avoided by reusing `decode_global_id` + the identity-check shape from `relay.py::_check_typed_match`. The mutation's wrong-type check compares against the **relation target model** (not a node-field target), so it is a new *call shape* over the same decode primitive, not a new decoder.
  - **Risk: re-implementing the optimizer plan-and-apply for the re-fetch.** Avoided by routing through `apply_connection_optimization` (the connection field's own seam) — the G2 gate, the ContextVar optimizer discovery, and the no-optimizer-installed short-circuit all come for free. Zero new optimizer code.
  - **Risk: a second sync-misuse site.** Avoided — the locate reuses `apply_type_visibility_sync`, which is the package's ONE sync-`get_queryset` site that closes-then-raises `SyncMisuseError`. The resolver never calls `get_queryset` directly.
  - **Risk: re-walking `permission_classes` in the resolver.** Avoided — the resolver calls the Slice-2 `check_permission` method (which already iterates `permission_classes`); the resolver only maps `False`→`GraphQLError`.
  - **Risk: a third operation-string enumeration.** The resolver imports `_VALID_OPERATIONS` from `sets.py` rather than re-spelling the set (Worker 3 Slice-2 note flagged a shared-operations-enum as a possible *integration-pass* DRY idea **only if Slice 3 adds a third enumeration** — so Slice 3 must NOT add a fourth bare `{"create","update","delete"}` literal; reuse `_VALID_OPERATIONS` and `_OPERATION_INPUT_KIND`). If the resolver needs an operation→pipeline-branch dispatch, key it off the existing constants, not a new dict of the same keys, unless the dispatch genuinely differs (a method-per-operation dispatch table is fine; a re-enumeration of the *valid set* is not). Flag for Worker 3's DRY check and the integration pass.
  - **Risk: duplicating the `_payload_object_slot` Relay check.** Avoided — reuse `mutations/inputs.py::payload_object_slot(primary_type)` (Slice 1) for the success/error payload slot, rather than re-deriving `implements_relay_node`.

### Implementation discretion items

Worker 1 has assessed these and decided they belong to Worker 2 (equivalent-shape / naming-detail, no architectural content):

- **The `id:` argument id-type at signature-synthesis time.** The `id:` type is `relay.GlobalID` for a Relay-Node primary, else the raw pk scalar — but the primary type is a *bind output* (`_primary_type`), unknown when `DjangoMutationField(...)` runs at import (before `finalize_django_types()`). Two equally-valid resolutions: **(a)** declare `id: strawberry.ID` (a raw string, the `relay.py::DjangoNodeField` precedent — line 287 — which deliberately uses the raw string so the package decodes server-side and malformed ids reach it), and decode it inside the resolver via `decode_global_id` / coerce-to-pk exactly as the node field does; **or (b)** use a `strawberry.lazy` forward-ref to the primary's GlobalID type. **Prefer (a)** — it matches the shipped node-field server-side-decode discipline, sidesteps the construction-time primary-type-unknown problem entirely, and gives the package the malformed-id handling for free. The spec headline schema (lines 191-220) shows `id: GlobalID!`, but the node field already renders `ID!`-as-decoded-server-side and the SDL difference (`ID!` vs `GlobalID!`) is cosmetic for the wire contract — Worker 2 picks; if (a) yields `ID!` in the SDL where the headline shows `GlobalID!`, that is acceptable (the node field shipped the same way) — note it for Worker 3 so the SDL difference is not read as a defect. If Worker 2 finds a clean way to render `GlobalID!` via a lazy ref to the primary's id type without a construction-time primary lookup, that is also fine.
- **Whether `resolve_mutation_sync` and `_run_pipeline_sync` collapse into one function.** If the sync entry IS the transaction-wrapped body, one function suffices; if the async path needs to wrap a bare callable, a thin split is cleaner. Worker 2 picks; the contract is "one sync body, wrapped once by the async path in `sync_to_async(thread_sensitive=True)`".
- **`save(update_fields=...)` vs full `save()` on update.** A targeted `update_fields` is more precise (only writes provided columns) but interacts with `auto_now`; a full `save()` is simpler and `auto_now` re-stamps. Either is acceptable in 0.0.11; Worker 2 picks. Pin that an unprovided field is unchanged in the DB regardless of which is chosen.
- **The payload slot source — recompute `payload_object_slot(primary_type)` vs read a stashed slot.** Slice 2 stashed `_primary_type` and `_payload_type_name` but not the slot name. Recomputing `payload_object_slot(mutation_cls._primary_type)` (Slice-1 helper) is single-sourced and cheap; a stashed slot would be a Slice-2 change (out of scope). Prefer recompute.
- **Delete-snapshot materialization mechanism.** AR-M5 requires relations loaded into the instance before `delete()`. The optimizer-planned re-fetch queryset (`apply_connection_optimization` → `select_related`/`prefetch_related`) plus a `.first()` evaluation loads `select_related` joins eagerly; `prefetch_related` children load on first access (so the snapshot must force-access them, or the queryset's `_prefetched_objects_cache` must be populated by evaluation). Worker 2 picks the exact force-load mechanism (evaluate the queryset to a list, or access the prefetched accessors) so nested/connection-child relations survive the row's deletion. Pin a delete payload selecting a relation returns it populated.

### Medium:

**M3-1 — `_provided_attr_names` naive `_id` suffix-strip mangles a scalar model field whose name ends in `_id`, silently skipping it from partial-update `full_clean` validation.**
- Source: `django_strawberry_framework/mutations/resolvers.py::_provided_attr_names` (lines 262-277, the `attr[:-3] if attr.endswith("_id") else attr` strip), consumed by `_run_update` (line 532-533) via `_unprovided_exclude`.
- Why it matters: `_decode_relations` stores FK attrs under a genuine `<field>_id` key AND scalar attrs under their real `python_name`. A scalar model field named, e.g., `object_id` (a `PositiveIntegerField`) lands in `scalar_and_fk_attrs` under `object_id`; `_provided_attr_names` then strips the suffix and records it as `object`. The real `object_id` is therefore treated as **unprovided**, `_unprovided_exclude` adds it to the `full_clean(exclude=...)` set, and a provided-but-invalid value **skips validation entirely**. This is a live, reachable defect, not theoretical: the example `apps.library.models.TaggedItem` has exactly such a field, and `editable_input_fields(TaggedItem)` emits `object_id` as a generated scalar input field (verified — `['tag', 'content_type', 'object_id']`). Reproduced end-to-end: a partial update `{ objectId: -5 }` on `TaggedItem` did **not** produce a clean `full_clean` `FieldError` on `object_id`; the invalid value slipped to the DB and surfaced as a mis-labeled `IntegrityError` → the generic `"A uniqueness constraint was violated."` `"__all__"` envelope (a non-uniqueness failure wrongly reported as a uniqueness race — the `_integrity_error_field_errors` hardcoded message, line 372, is part of the same blast radius). Create is unaffected (it validates with `exclude=None`); the defect is update-only.
- Recommended change (root-cause, not test-only): `_decode_relations` already knows which keys are FK-derived `<field>_id` (the `fk_by_attr` index). Make the decode return the provided **model-field-name** set directly (map FK keys back to `field.name` at decode time, where the relation field is in hand), or have `_provided_attr_names` strip `_id` only for keys it confirms are in the model's FK index — never by blind string suffix. The string-suffix heuristic is the wrong abstraction; the relation index is the source of truth.
- Test expectation: add a package test in `tests/mutations/test_resolvers.py` over a fixture model with a scalar field ending in `_id` (TaggedItem fits) pinning that (a) a provided such field IS validated by `full_clean` on partial update (an invalid value yields a `FieldError` keyed to that field, not a swallowed IntegrityError), and (b) `_provided_attr_names({"object_id": v}, [])` contains `"object_id"`, not `"object"`. Promote the disposed probe (see Temp test verification).

### FV-1 (High / blocking) — the async mutation pipeline poisons the next read-side optimizer execution via the optimizer's per-execution ContextVar lifecycle

- **Symptom (deterministic, reproduced).** `uv run pytest tests/mutations/ tests/optimizer/ tests/base/test_init.py --no-cov` → **1 failed, 515 passed**: `tests/optimizer/test_extension.py::test_cascading_target_downgrades_join_to_prefetch` fails with `apps.products.models.Item.category.RelatedObjectDoesNotExist: Item has no category` raised inside `django_strawberry_framework/types/resolvers.py::forward_resolver` — i.e. the `category` relation was **not planned** for that read query, so the lazy access raised. `test_extension.py` passes **alone** (135 passed); the failure appears only when a mutation test runs before it.
- **Bisected polluter:** `tests/mutations/test_resolvers.py::test_async_pipeline_create_happy_path` — the **only** test in `test_resolvers.py` that pollutes the target (verified one-by-one; the sync `test_create_happy_path` is a clean control). The poison is **one-shot**: `async-test, target, target` → the *first* target run fails and the *second* passes, so a single subsequent `schema.execute_sync` consumes/overwrites the leaked state.
- **Root cause (introduced by THIS slice).** On HEAD `mutations/resolvers.py` is the TODO scaffold — the async pipeline `resolve_mutation_async` → `await sync_to_async(_run_pipeline_sync, thread_sensitive=True)(...)` is **net-new this slice**, so this is the first time an `await schema.execute` of a mutation detours the ORM pipeline (which calls `apply_connection_optimization`, reading the `_active_optimizer` ContextVar) through `sync_to_async`. The optimizer's `DjangoOptimizerExtension.on_execute` (`optimizer/extension.py::DjangoOptimizerExtension.on_execute` #"active_token = _optimizer_active.set") is a generator hook that `.set()`s four ContextVars (`_optimizer_active`, `_active_optimizer`, `_printed_ast_cache`, `_pagination_var_names_cache`) before `yield` and `.reset(token)`s them in `finally`. On the async execution path Strawberry drives that sync generator's `__enter__`/`__exit__` through its async extension-context machinery; the `finally` `.reset(token)` is not guaranteed to run in the **same `contextvars.Context`** the `.set()` ran in. `ContextVar.reset(token)` raises `ValueError` (or no-ops) across a context boundary, leaving a **stale per-execution ContextVar** (most plausibly `_printed_ast_cache` / `_pagination_var_names_cache`, the `id(operation)`-keyed memo dicts that `_build_cache_key` reads — a stale memo surviving into the next execution yields a wrong/empty plan via an `id(operation)` reuse-after-GC collision; `_active_optimizer` pointing at the prior test's extension instance is a second candidate). The one-shot symptom matches a leaked memo dict consumed by the next execution.
- **This is a read-side optimizer regression, not a mutation-test-only issue.** The corrupted execution is a *read* query (`{ allItems { name category { name } } }`) over a *fresh* `DjangoOptimizerExtension()` — the slice's new async write path is reaching across the test boundary and breaking the shipped read-side planner. It is exactly the B1–B8 / DoD-item-6 surface the spec (Decision 9) and the build plan flag as must-not-regress.
- **Recommended fix (root-cause, not a band-aid).** Make the optimizer's per-execution ContextVar teardown **context-safe across the async-driven generator hook** so a token set on entry is always cleared regardless of which context the `finally` resumes in — e.g. reset each var defensively (guard the `ValueError` and fall back to an explicit `.set(default)`/sentinel restore so the var cannot stay stale), OR ensure the mutation `sync_to_async` detour does not run inside a context whose teardown diverges from the hook's set-context. Do **not** ship a "clear the memo dict in `finally`" surface patch that leaves the underlying reset-in-wrong-context defect: the correct contract is "each execution starts with the optimizer ContextVars at their documented defaults," and the fix must restore that invariant at the source (the hook lifecycle), per `AGENTS.md` "always recommend the root-cause fix over the surface patch." Worker 1 has NOT prescribed the exact patch (it is Worker 2's design call, Worker 3-reviewed); the contract is: the focused full scope `tests/mutations/ tests/optimizer/ tests/base/test_init.py --no-cov` passes with **zero** failures AND a permanent regression test pins the invariant (an async mutation execution followed by a read optimizer execution in the same session must not corrupt the read plan — promote the bisected `async-then-read` repro into a durable pin, package-tier per AGENTS.md since it exercises the ContextVar lifecycle directly).

The alternatives Worker 1 floated (a defensive `finally`-clear of the optimizer ContextVars, or routing the mutation `sync_to_async` differently) would have been band-aids for a non-existent ContextVar bug AND would have left the actual row leak intact — they were correctly avoided.
