### DRY analysis

- **Existing patterns reused.**
  - **Phase-2.5 loop body shape.** The new filter-binding subpasses reuse the phase-2.5 loop scaffolding at `django_strawberry_framework/types/finalizer.py::finalize_django_types #"for type_cls, definition in registry.iter_definitions()"` (the second occurrence at lines 230-244 â€” the Relay-Node loop). The four ordered subpasses are FOUR iterations of `registry.iter_definitions()`, each with the same `if definition.finalized: continue` guard at the head; this mirrors the Relay-Node loop's existing shape rather than inventing a parallel iteration helper. The Slice-1 owner-aware path in `django_strawberry_framework/filters/sets.py::FilterSet._resolve_relation_target_type` already consumes `_owner_definition.related_target_for(field_name)` behind a `callable(...)` guard (defensive against pre-Slice-3 absence) â€” the binding subpass just sets `_owner_definition`; the runtime accessor is already wired.
  - **`ConfigurationError` raise-text shape.** Reuses the canonical formatter pattern at `django_strawberry_framework/types/finalizer.py::_format_unresolved_targets_error` (the unresolved-relation-target error). The new filter-binding raise sites use the same lead-in shape ("Cannot finalize Django types: â€¦") and the same indent-the-offenders style so consumer error matching stays grep-stable across both Layer-3 sidecars.
  - **`registry.clear()` TODO anchor.** The `try / except ImportError` cycle-safe-clear pattern is already pseudocoded at `django_strawberry_framework/registry.py #"# TODO(spec-021-filters-0_0_8 Slice 3): Co-clear filter input"`; the implementation matches that anchor and the spec's snippet at Decision 9 verbatim (per M4 of rev8 â€” inline `_types.clear()` shape, NOT a `_reset_state` call).
  - **`_apply_get_queryset_*` precedent.** Decision 8 / `relay.py::_apply_get_queryset_sync` / `_apply_get_queryset_async` are already the sync/async dispatch precedent the apply pipeline reuses; Slice 3 inherits that contract by binding `_owner_definition` and does NOT add new dispatch surface.
  - **`Meta.interfaces` promotion gate as architectural twin.** The promotion gate for `filterset_class` follows the exact same shape used for `Meta.interfaces` in DONE-011-0.0.5 â€” a deferred Meta key promoted only when the subsystem applies it end-to-end. Reuses the `ALLOWED_META_KEYS | DEFERRED_META_KEYS` typo-guard at `types/base.py::_validate_meta #"declared - ALLOWED_META_KEYS - DEFERRED_META_KEYS"` without modification; only the membership of the two frozensets changes.

- **New helpers justified.** Three new helpers, each with a single responsibility:
  1. **`DjangoTypeDefinition.related_target_for(self, field_name)`** in `types/definition.py` per H1 of rev5. Single responsibility: given a model field name on the owning type, return `(target_definition, model_field)` for a relation field, else `None`. This is the seam Slice 1 already pseudo-consumes via `getattr(owner, "related_target_for", None)`; Slice 3 lands the real method.
  2. **`_bind_filterset_owner(filterset_cls, definition)`** in `types/finalizer.py` (module-private). Single responsibility: validate the `_owner_definition` slot is unset OR strictly compatible per H2 of rev8 (every owner-sensitive `related_target_for(field_name)` resolves to the EXACT same `DjangoTypeDefinition` AND `graphql_type_name`); set it; raise `ConfigurationError` on mismatch. Centralizing avoids duplicating the H2-rev8 comparison logic between subpass 1 and any future caller (e.g., `_dynamic_filterset_cache` Layer-6 callers in 0.0.9).
  3. **`_validate_filterset_class(meta_value)`** in `types/base.py` (module-private, local-import shape). Single responsibility: validate `meta.filterset_class` is a `FilterSet` subclass via a local import (per the TODO anchor at line 589-596). Lives next to `_validate_meta` because it shares the `ConfigurationError` raise convention; the local import keeps `types/base.py` from growing a static import cycle into `filters/`.

  No new module is justified â€” the four new editing surfaces (`base.py` validator extension, `definition.py` slot+method, `finalizer.py` subpasses, `inputs.py` materializers) live in their existing modules.

- **Duplication risk avoided.**
  - **`f"{Name}InputType"` literal.** This literal exists at 4 sites already (`__init__.py::filter_input_type`, `factories.py::FilterArgumentsFactory.__init__`, `factories.py::_ensure_built`, `factories.py::_build_class_type`). Slice 3 must NOT introduce a 5th raw occurrence in the finalizer â€” the binding pass reads the name from the factory's own `filter_input_type_name` attribute (already on the instance) and from the `_type_filterset_registry` mapping the factory exposes, so no new `f"{Name}InputType"` literal is needed in `finalizer.py`. The cross-slice integration pass will fold this literal into a shared helper.
  - **`graphql_type_name` derivation.** The H2-rev8 comparison needs the GraphQL type name for both owners. Strawberry derives the type's GraphQL name as `definition.name if definition.name is not None else definition.origin.__name__`. The plan computes this in ONE helper inside `_bind_filterset_owner` (the new helper above), NOT inline at each comparison site, so the fallback never drifts.
  - **`registry.clear()` co-clear.** Risks a near-copy if the helper-referenced-set clear and the input-namespace clear are written as two separate `try / except ImportError` blocks that drift over time. The spec snippet at Decision 9 (lines 793-820) already pins the TWO-block shape (one per local import) â€” the plan adopts that snippet verbatim, with both clears guarded individually so the failure mode of one block does not silently suppress the other. The early `return` in the second block (per the spec snippet line 818) is intentional â€” if `_helper_referenced_filtersets` is unimportable, the first clear has already run and the second block has nothing to do.
  - **Phase-2.5 subpass iteration.** Four subpasses each calling `registry.iter_definitions()` looks like duplication but is correct per H1 of rev8: each subpass MUST complete across ALL types before the next starts. Collapsing them into a single loop would re-introduce the rev4â€“rev7 order-dependency bug. The plan keeps four separate loops with comments naming each subpass; the loops are not folded.
  - **`_helper_referenced_filtersets` orphan check.** Risks duplicating the set-difference logic. The plan computes `orphans = _helper_referenced_filtersets - wired_set` once in subpass 4, sorts the offending classes by `__qualname__` for deterministic error output, and emits one `ConfigurationError` naming all orphans (NOT one per orphan). Single source of truth for the orphan-format string.

### Implementation notes

- **Medium #2 test shape â€” two `DjangoType`s on the same model (one primary, one secondary).** This was the simplest way to force two distinct `DjangoTypeDefinition` instances that both consult `Book._meta` and therefore resolve `related_target_for("shelf")` to the same target. The alternative shape (two different models whose `shelf` field resolves to the same `Shelf` target type â€” e.g., `BookType.shelf` and `MagazineType.shelf` both pointing at `ShelfType`) would require a second model with an FK to `Shelf`, which the fakeshop library models don't already provide; the same-model-two-owners shape uses only existing fixtures and exercises the same strict-equality walk code path.
- **Medium #2 first-binding assertion â€” `iter_definitions()` order is registration order.** The test asserts `BookFilter._owner_definition is primary_definition` (the first-declared owner). This works because `registry.iter_definitions()` yields in registration order and `DjangoType.__init_subclass__` registers in source-declaration order; `PrimaryBookType` (declared first) is therefore bound first by subpass 1 of `_bind_filtersets()`. If a future change re-orders `iter_definitions()` (e.g., to alphabetical order for deterministic output across consumer module-import orders), this assertion would need to update to match.
- **Medium #3 wrap shape â€” preserve `__cause__` AND keep the `ImportError`'s message in the wrapped string.** The wrap concatenates `{exc}` (which evaluates to `str(exc)` â€” the original Layer-2 `ImportError` message naming the unresolved string) into the new `ConfigurationError` message. The `from exc` also preserves the full traceback chain. Both signals are needed: the inline message keeps the unresolved name greppable by consumer error matching; the `__cause__` chain keeps the debugger traceback intact for maintainers.
- **`test_phase_2_5_unresolved_related_filter_raises_at_finalize` assertion shape.** Asserts three message substrings (`"BookFilter"`, `"unresolved"`, `"UnknownFilter"`) rather than the exact full message string, so a future tightening of the message wording (e.g., adding spec-line references, swapping the leading verb) does not break the test. The `__cause__` assertion uses `isinstance(..., ImportError)` rather than identity so subclasses of `ImportError` (e.g., `ModuleNotFoundError`) also pass â€” Layer-2 resolution can raise either form depending on the failure path inside `import_string`.
