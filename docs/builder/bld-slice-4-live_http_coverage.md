  - **None at the test-helper level.** The 14 tests are coverage-anchored and each pins its own contract; introducing a new per-test seed helper would dilute the test-as-documentation shape AGENTS.md #"Library acceptance tests use inline Model.objects.create" already pins.

  - **GlobalID-string construction risk.** Two tests construct GlobalIDs (the Relay-M2M test L1047 and the wrong-`type_name` test L1057). Don't collapse.
  - **Test-file `import` near-copy risk.** The existing file imports `base64`, `importlib`, `sys`, `pytest`, `apps.library.models`, `django.db.connection`, `django.test.Client`, `CaptureQueriesContext`, `clear_url_caches`, `registry`. Slice 4 adds two imports: `from strawberry import relay` (for the wrong-`type_name` GlobalID construction; not yet imported in the test file but present in the schema file — keep the import at module-top per PEP 8) and possibly nothing else. Do NOT import `filters` from `apps.library` at the module top — the autouse reload fixture means classes from `apps.library.schema` are stale across tests; the same applies if we module-import `filters`. The two tests that need to reach into a filter class do so by `importlib.import_module("apps.library.filters")` inline (matching the existing fixture's pattern at L29-32), OR — and this is the preferred shape — by issuing the GraphQL query and asserting through the response, which is the canonical "system-under-test is the live `/graphql/` endpoint" stance per spec L1042. The 14 tests should NOT need to import the filter classes directly.

### Implementation notes

- **`clear_filter_input_namespace()` orchestration shape.** The helper now owns four cache clears: (1) module-global `delattr` for materialized input classes; (2) `_materialized_names` + `_field_specs` reset; (3) `FilterArgumentsFactory.input_object_types` + `_type_filterset_registry` clear (via local import to keep `inputs.py` a leaf); (4) per-subclass `delattr` of binding state on every `FilterSet.__subclasses__()` entry. Each clear is in its own try/except-style block (the factory and `FilterSet` imports are local + tolerant of `ImportError`) so a partial-load environment (e.g., a process that imported `registry` but not the filter subsystem) still gets whatever clears ARE reachable. The contract is "after `registry.clear()` returns, the filter subsystem is fully primed for a fresh build" — the prior shape only honored part of that contract and the failure manifested only when the consumer's autouse reload exercised the materialization path (Slice 4's `filter_input_type(...)` wiring).
- **`_iter_filterset_subclasses(root)` is a depth-first walk with identity dedup.** Python's `cls.__subclasses__()` returns immediate subclasses only; for `FilterSet` -> `BranchFilter` -> (no further subclasses) the depth is 1, but consumer code could in principle declare a `class CustomBranchFilter(BranchFilter)`. The walk handles that future shape without adding a recursion knob. Dedup via a `set[type]` because subclass diamond inheritance is uncommon but the identity check is cheap insurance.
- **`delattr` over `cls.__dict__.pop`.** First-pass attempt used `subclass.__dict__.pop(attr, None)` which raises `AttributeError: 'mappingproxy' object has no attribute 'pop'` — `cls.__dict__` is a read-only `mappingproxy`. The public `delattr(subclass, attr)` form (guarded by `attr in subclass.__dict__`) is the canonical write path; pinned in the helper's inline comment so a future refactor does not try to switch shapes again.
- **Optimizer test assertion change is a contract-correctness fix.** The pre-Slice-4 `test_library_optimizer_selects_book_shelf_in_http_query` asserted "1 query with JOIN" against a schema where `ShelfType` had no `get_queryset`. The test was passing pre-revision only because the broken isolation prevented the schema reload from actually picking up `ShelfType.get_queryset` when this single optimizer test ran. Fixing the isolation surfaced the correct optimizer behavior; updating the assertion reflects what the optimizer should do, not a regression.
