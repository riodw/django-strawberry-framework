### `django_db` test-mode decision (the write-then-read isolation pin)

**All live mutation tests that drive a write use `@pytest.mark.django_db(transaction=True)`.** Rationale, verified against the codebase:

- The fakeshop project is **WSGI / sync** (`config/settings.py #"WSGI_APPLICATION"`); `strawberry.django.views.GraphQLView` is the sync Django view, so a `/graphql/` mutation over `django.test.Client` runs the **sync** pipeline (`resolvers.py::resolve_mutation_sync`), which commits inside `transaction.atomic()` on the **main thread**. A plain `@pytest.mark.django_db` would nest that atomic inside pytest's outer transaction and roll it back at teardown, so write-then-read *within one request* works — but a test that **independently verifies committed DB state after the request** (the create persisted, the delete removed the row) and the `CaptureQueriesContext` count are cleanest under a real transaction: `transaction=True` flushes between tests (no row leak across tests) and avoids `SAVEPOINT` / nested-savepoint SQL polluting the captured-query count.
- This is the **suite-wide async-ORM-commit convention** carried forward from Slice 3 (`tests/mutations/test_resolvers.py::test_async_pipeline_create_happy_path` at `:501` uses `@pytest.mark.django_db(transaction=True)` and its docstring documents *why* — an async write commits on asgiref's executor-thread connection that escapes plain-`django_db` main-thread rollback; package mutation suite + `test_list_field.py` ×8 + `test_relay_connection.py` ×4 all follow it for committing writes). The live products surface is sync today, so the async-connection hazard does not bite here, but `transaction=True` is the correct, leak-free mode for committing-write tests and matches the established convention — pinning it now also future-proofs the live suite against a later ASGI switch.
- **Read-only tests stay on plain `@pytest.mark.django_db`** (the existing products read tests are unchanged). Only the new write tests carry `(transaction=True)`. Each new write test still starts with `seed_data(N)` / `create_users(N)` per `AGENTS.md` (the `transaction=True` mode does not change the fixture-first rule).

Worker 2 must verify, when adding each write test, that the write actually persists/rolls back as the test asserts (run the focused test) — a plain-`django_db` write test that asserts post-request committed state can silently pass on the in-request transaction and then leak into a sibling under a different mode mix.
