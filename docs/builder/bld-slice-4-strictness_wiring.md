### DRY analysis

The DRY headline of this slice (Decision 8, build-plan "DRY-first rule (short copy)", hard constraint): **parameterize `types/resolvers.py::_check_n1` and call it from `connection.py` — never write a second strictness checker.** Decision 8's rejected alternative ("A separate `connection.py`-local checker") is the trap; the parameterization is mechanical because `_check_n1` already carries the exact probe/flag/raise shape (the sentinel read, the `resolver_key` lookup, the lazy-load probe, the `"raise"` / `"warn"` dispatch).

**Existing patterns reused (cite `file::Symbol`):**

- **`types/resolvers.py::_check_n1` IS the contract — parameterize, do not fork.** Today (lines 123-185) it: reads `DST_OPTIMIZER_PLANNED` from context (`return` if absent — the no-optimizer / strictness-off no-op); builds `key = resolver_key(parent_type, field_name, runtime_path_from_info(info))` and `return`s if planned; probes the instance cache for a lazy load (`_will_lazy_load_many` for many-side `kind`, `_will_lazy_load_single` otherwise) and `return`s if not lazy; then reads `DST_OPTIMIZER_STRICTNESS` and raises `OptimizerError(f"Unplanned N+1: {field_name}")` under `"raise"` / logs `_resolver_logger.warning("Potential N+1 on %s", field_name)` under `"warn"`. Slice 4 adds a NEW lazy-probe branch keyed on `kind == "connection_to_attr"`: the access queries iff the fast-path `to_attr` is **absent** on `root` (`getattr(root, to_attr, None) is None`) — when present, Slice 1's window already served the rows so it is **not** lazy → silent. This is the three-condition guard (sentinel stashed AND key absent AND `to_attr` absent) reproducing the list-relation semantics exactly (Decision 8 line 369; spec line 75). The `reason` enriches the message. **The list-relation three branches stay byte-identical** (`many` / `single` dispatch unchanged) — DoD item 7 "the list-relation strictness suite passes unmodified".
- **`resolver_key(parent_type, field_name, runtime_path)` — the shared key vocabulary.** `optimizer/plans.py::resolver_key` (lines 256-261) produces `{parent_type.__name__}.{field_name}@{path}` and is already imported into `types/resolvers.py` (line 48) AND used by `_plan_connection_relation` (`walker.py` line 1191-1193, `resolver_key(type_cls, relation_field_name, runtime_path)`). For the resolver-side check to MATCH the walker's planned key, `_check_n1` must be called with `parent_type` = the **declaring** `DjangoType` (`type_cls` at the nested walk level — `registry.get(model)`, the primary type) and `field_name` = `relation_field_name` (the underlying relation field name `"books"`, NOT the generated `"books_connection"`). The resolver-time `runtime_path_from_info(info)` reads `info.path` at the connection field's response key, which is exactly the runtime path the walker emitted (`runtime_paths` from the same response key — `walker.py` lines 1186-1190). This match is the load-bearing correctness pin (`test_strictness_silent_when_planned`). Reuse `resolver_key` + `runtime_path_from_info`; build no parallel key shape.
- **`optimizer/_context.py` sentinels + `get_context_value`.** `_check_n1` already reads `DST_OPTIMIZER_PLANNED` / `DST_OPTIMIZER_STRICTNESS` via the imported `_get_context_value` (lines 39-46, 167-185). The connection consultation rides the SAME read path through `_check_n1` — `connection.py` does **not** read sentinels directly (that would be a second consultation site to keep in sync). `connection.py` only supplies `root`, `info`, the key inputs, and the `to_attr`; `_check_n1` owns every sentinel read. (Slice 1's `_publish_plan_to_context` union is what guarantees a nested fallback pipeline's own publish does not shrink the parent's `DST_OPTIMIZER_PLANNED` set before the parent resolver's `_check_n1` reads it — Decision 8 line 370; consumed, not re-implemented here.)
- **`_relation_connection_to_attr(relation_field_name)`** — `optimizer/walker.py` (single-sited per Slice 1's DRY pin), already imported into `connection.py` (line 70) and already used to build the fast-path probe `to_attr` (line 998). The Slice-4 consultation reuses that SAME `to_attr` local — the strictness `to_attr` probe and the fast-path probe are one value. No re-spelled `f"_dst_{...}_connection"`.
- **`OptimizerError` + `_resolver_logger`** — the exception class and logger `_check_n1` already raises/logs (`types/resolvers.py` lines 30, 38, 183-185) are the connection contract too (spec line 215 "the same exception class and vocabulary list relations use"). No new exception, no new logger; `connection.py` never raises `OptimizerError` itself (it flows out of `_check_n1`).
- **`instance_accessor(field)`** — already used at the resolver builder call-site (`finalizer.py` line 438) to compute `accessor_name`. The `to_attr` for the connection is keyed on the FIELD NAME (`relation_field_name`), already threaded in Slice 2; no new threading of the accessor needed for strictness (the connection's lazy probe is the `to_attr`, not Django's per-relation cache, so `_check_n1`'s `accessor_name` parameter is unused on the connection branch — see Implementation step 1).

**New helpers justified (single responsibility each):**

- **None.** This slice adds NO new helper, NO new module, NO new constant, NO new exception. It widens `_check_n1`'s keyword-only signature (`to_attr: str | None = None`, `reason: str | None = None`, and accepts the new `kind` value `"connection_to_attr"`) and adds one call site in `connection.py`. Decision 8 mandates the single-implementation shape and forbids a second checker; introducing a helper here would re-fragment the contract the parameterization exists to keep single. (If Worker 2 finds the connection-branch lazy probe reads cleanest as a one-line private predicate inside `resolvers.py`, that is a within-module micro-helper at Worker 2's discretion — but it must live in `resolvers.py` beside `_will_lazy_load_*`, never in `connection.py`.)

**Duplication risk avoided:**

- **A second strictness checker in `connection.py`** (Decision 8 rejected alternative). The trap: `connection.py::_resolve` grows its own `DST_OPTIMIZER_PLANNED` read + `resolver_key` build + `DST_OPTIMIZER_STRICTNESS` dispatch. Avoided: `connection.py` calls the parameterized `_check_n1` and supplies only inputs; every sentinel read and the raise/warn dispatch stay single-sited in `_check_n1`.
- **Re-spelled `_dst_<field>_connection` `to_attr`.** The consultation reuses the `to_attr` local already computed at `_build_relation_connection_resolver` line 998 via `_relation_connection_to_attr` — passed to `_check_n1` as the `to_attr` argument. No fourth f-string.
- **A divergent message format.** `_check_n1`'s raise is `OptimizerError(f"Unplanned N+1: {field_name}")`. The connection branch must keep `Unplanned N+1: {field_name}` as the stem and append the reason in parens when present (`f"Unplanned N+1: {field_name} ({reason})"`) — Decision 8 line 371 / Error shapes line 215. The list-relation message (no reason) is unchanged; the connection message is the same stem + optional suffix, one f-string family, not a second message vocabulary.
- **A mismatched resolver key.** If `connection.py` passed `target_type` (the related type) or the generated `"books_connection"` name instead of the declaring `type_cls` + `relation_field_name`, the resolver-time key would never match the walker's planned key and `_check_n1` would false-positive on EVERY planned connection. Avoided by threading the **declaring type** into the resolver builder (Implementation step 2) and passing `relation_field_name` (already a builder parameter from Slice 2) as `field_name`. Pinned by `test_strictness_silent_when_planned` and `..._when_window_served`.
