## Plan (Worker 1)

### DRY analysis

- **Existing patterns reused.**
  - `_normalize_input` already emits the logical-operator keys (`and` / `or` / `not`) at the top level of the form-data dict via the `_LOGIC_KEYS` mapping. The override reads them off `self.data`, not off `form.cleaned_data` (django-filter's form has no `and`/`or`/`not` fields, so those keys are dropped at form-validation time). Cite `django_strawberry_framework/filters/sets.py:438-444` (the `_LOGIC_KEYS` branch of `_normalize_input`).
  - `django_filters.filterset.BaseFilterSet.filter_queryset(self, queryset)` (the inherited instance method) already iterates `self.form.cleaned_data` and calls each registered `Filter.filter(queryset, value)` for the FLAT leaf clauses. Cite `~/projects/django-graphene-filters/.venv/lib/python*/site-packages/django_filters/filterset.py:224-241`. Slice 4a's override calls `super().filter_queryset(queryset)` first so the leaf-clause path stays unchanged; the override only adds the tree-form composition step on top.
  - `django.db.models.Q` — the standard Django `Q` primitive — is the cookbook's chosen composition tool. Cite `~/projects/django-graphene-filters/django_graphene_filters/filterset.py::AdvancedFilterSet.get_queryset_proxy_for_form` lines 622-645 (`q & and_q`, `or_q | new_q`, `~new_q`). The package borrows the same Q-composition shape but expresses it through sibling-filterset PK intersection rather than a parallel `QuerySetProxy` because our `_normalize_input` already produces nested Strawberry input dataclasses that a sibling `cls(data=..., queryset=...)` instantiation can consume directly. `models` is already imported in `sets.py` at line 22 — `Q` is reached via `models.Q`, no new import.
  - Sibling-instance instantiation pattern: the existing `_apply_related_constraints` step at `sets.py:738-763` and `_derive_related_visibility_querysets_sync` at `sets.py:590-621` already construct child `FilterSet` instances against a constrained queryset and consult `.qs`. Slice 4a's tree walker reuses the same instantiation shape (`cls(data=child_normalized_data, queryset=queryset).qs`) so the sync misuse sentinel, the form-validation raise, and the related-constraints baking all carry through transparently.
  - `_normalize_input` is already a classmethod and the input parameter is the Strawberry input dataclass (or a dict) — both shapes are what an `and` / `or` / `not` branch carries. The walker calls `cls._normalize_input(branch)` to convert each nested input into a form-data dict and re-uses the existing `cls(data=..., queryset=queryset)` instantiation.
  - The `_apply_related_constraints` step in the apply pipeline runs BEFORE filterset instantiation (`sets.py::FilterSet.apply_sync` line 783, per H3-of-rev8). The visibility-scoped and related-constraint-baked queryset is what `filterset_instance = cls(data=data, queryset=constrained, ...)` carries through; `self.queryset` in the override is therefore already the parent's visibility-scoped + RelatedFilter-constrained queryset. The tree walker composes ON TOP of that, never against the unscoped manager — visibility-before-filter ordering is preserved by construction.
- **New helpers justified.** One new private classmethod, `_evaluate_logic_tree(cls, queryset, tree_data) -> Q`, on `FilterSet` at `sets.py`. Single responsibility: given the parent's `self.data` (or a recursive sub-branch's normalized dict), return a `Q` expression composed from `and` / `or` / `not` sub-branches. The override `filter_queryset(self, queryset)` is a thin two-step composition — `qs = super().filter_queryset(queryset); return qs.filter(self._evaluate_logic_tree(qs, self.data))` — so the recursive Q-building lives in one helper rather than inline in the override. Justification for extracting: the walker is recursive (an `and` branch may itself carry `or` / `not` sub-branches per the Strawberry self-reference depth), and keeping recursion logic in a named helper makes the three new tests (intersect / union / negate) self-targeting at one symbol. The helper's input is the already-normalized form-data dict on `self.data` for the top call and the result of `cls._normalize_input(child)` for each nested branch.
- **Duplication risk avoided.**
  - **`_LOGIC_KEYS` literal duplication is OUT OF SCOPE for Slice 4a.** The tuple already lives twice — `sets.py:53-57` and `inputs.py:108-112` — and the override consumes the `"and"` / `"or"` / `"not"` string keys directly off `self.data`. Adding a third inlined copy would be the easy mistake; the override reuses the existing `sets.py` `_LOGIC_KEYS` constant by re-deriving the wire keys from `dict(_LOGIC_KEYS).values()` (matching how `_normalize_input` line 438 already builds its `logic_lookup`). The cross-module consolidation between `sets._LOGIC_KEYS` and `inputs._LOGIC_KEYS` is a Worker-1 integration-pass carry-forward documented in `worker-1.md` 2026-05-28 Slice-4 final-verification entry — do not fold it in Slice 4a.
  - **Do not re-implement leaf-clause filtering.** `BaseFilterSet.filter_queryset(self, queryset)` already handles every `form.cleaned_data` leaf clause via `self.filters[name].filter(queryset, value)`. The override calls `super().filter_queryset(queryset)` first and only composes tree-form Q's on top. Re-iterating `self.filters` for the flat leaves would duplicate the inherited path and risk drift if `django-filter` upgrades.
  - **Sibling-filterset instantiation must reuse the parent's `queryset` argument** so the visibility-scoped + related-constrained scope from the apply pipeline carries through. Passing the bare `cls._meta.model._default_manager.all()` to a recursive `cls(data=..., queryset=...)` would re-introduce the visibility-before-filter ordering bug the apply pipeline already prevents. Recurse against the queryset the override receives (the parent's post-`super().filter_queryset` queryset).
  - **Do not write a parallel `QuerySetProxy` / `TreeFormMixin` shape.** The cookbook's `AdvancedFilterSet.get_queryset_proxy_for_form` (cited above) walks a tree-form Django form via a custom `QuerySetProxy` because the cookbook supports per-leaf Q-mutation alongside queryset-mutation. Our `FilterSet` lookup primitives are `django_filters.Filter` subclasses whose `.filter(queryset, value)` mutates the queryset directly — there is no Q-only contract on the leaves to compose against. The Slice-4a override leans on PK-intersection (`pk__in=<sibling_qs>`) for tree composition instead, which is one Q expression per branch and avoids reimplementing the cookbook's parallel form/proxy machinery. Tradeoff: PK-intersection issues one subquery per branch; acceptable for the substrate-shipping goal and matches the spec L143 contract (`Q(...) & Q(...)` / `Q(...) | Q(...)` / `~Q(...)`).

### Implementation steps

1. **Add `filter_queryset(self, queryset)` instance-method override to `FilterSet` in `django_strawberry_framework/filters/sets.py`.** Place it adjacent to the other instance-bound methods (after `check_permissions` at `sets.py::FilterSet.check_permissions` and before `_validate_form_or_raise`, OR at the natural location between the apply-pipeline named helpers and the `apply_sync` / `apply_async` classmethods — Worker 2's call on exact insertion point). Body:
   - Call `qs = super().filter_queryset(queryset)` first — inherits the FLAT leaf-clause path verbatim from `django_filters.filterset.BaseFilterSet.filter_queryset` so existing behavior for non-tree filters is preserved bit-for-bit.
   - Compute `q = type(self)._evaluate_logic_tree(qs, self.data or {})`.
   - Return `qs.filter(q)` (a default empty `Q()` is a no-op; `qs.filter(Q())` returns `qs` unchanged).

2. **Add classmethod `_evaluate_logic_tree(cls, queryset, tree_data) -> models.Q` to `FilterSet`.** Single recursive helper. Body:
   - If `tree_data` is empty or carries no logical keys, return `models.Q()` (no-op).
   - For `and` branch: read `tree_data.get("and", []) or []`; for each `child_input` in the list, compute `child_q = cls._q_for_branch(queryset, child_input)`; AND them via `q &= child_q`.
   - For `or` branch: read `tree_data.get("or", []) or []`; for each `child_input`, compute `child_q = cls._q_for_branch(queryset, child_input)`; OR them via `q |= child_q` after starting from an explicit `Q()` accumulator (note: an empty `or` list returns a no-op `Q()`, NOT a forced-false predicate).
   - For `not` branch: read `tree_data.get("not")`; if present, compute `child_q = cls._q_for_branch(queryset, not_input)`; AND in `~child_q`.
   - Return the composed `Q`. Use `models.Q` from the already-imported `from django.db import models` at `sets.py:22`.

3. **Add classmethod `_q_for_branch(cls, queryset, child_input) -> models.Q` to `FilterSet`.** Single responsibility: given the parent queryset and one Strawberry input dataclass (or dict) representing one branch of the logical tree, compute the Q expression that matches the same set of rows the branch's nested filterset would. Body:
   - Normalize the child input via `child_data = cls._normalize_input(child_input)`.
   - Construct a sibling filterset instance against the same `queryset`: `child_set = cls(data=child_data, queryset=queryset, request=None)`. (Permission checks and form validation already ran against the top-level `apply_*` instance; nested branches reuse the parent's gating outcome because the tree-form-substrate operates only on data the top-level form already accepted at `apply_sync`/`apply_async` time. The override at runtime under `apply_*` is reached only after `_validate_form_or_raise` succeeded.)
   - Trigger Django-filter's leaf-clause path on the child instance against an unfiltered manager — `child_qs = child_set.qs` — which itself recursively invokes our override against any deeper `and` / `or` / `not` keys carried in `child_data`. Recursion terminates at the absence of nested logical keys (spec L143 contract: "no separate guard beyond the recursion's natural termination").
   - Return `models.Q(pk__in=child_qs.values("pk"))`. PK-intersection is the spec L143 composition primitive expressed against the parent's already-visibility-scoped `queryset`. Composition: `parent_qs.filter(q1 & q2)` reduces to `parent_qs.filter(pk__in=sibling_a_qs).filter(pk__in=sibling_b_qs)` (intersection); `q1 | q2` reduces to a single OR'd `pk__in` subquery; `~q` flips to `WHERE NOT pk__in(...)`.

### Implementation notes

- **Insertion point chosen: apply-pipeline clustering.** Inserted the override + two helpers immediately above `_apply_related_constraints` rather than near `check_permissions`. The override is part of the apply pipeline's runtime steps (it composes on top of the inherited leaf-clause path), so co-locating with `_apply_related_constraints` keeps the runtime composition logic visually grouped. Plan's discretion item explicitly licensed either placement.
- **Empty-tree handling: unconditional `dict.get` with `or` fallback.** `_evaluate_logic_tree` checks `isinstance(tree_data, dict) and tree_data` first (handles None / empty / non-dict in one branch), then unconditionally walks each of the three logical keys with `tree_data.get(key) or default`. Empty `and: []` and `or: []` are no-ops (an empty `and` list never enters the loop; the explicit `if or_branches:` guard prevents an `OR'd Q()` from forcing `WHERE False`). An absent `"not"` key uses `is not None` so a future explicit `not: None` (if any consumer ever sets it) is also a no-op.
- **`or` composition uses a fresh `Q()` accumulator AND'd into the running `q`** rather than OR-ing directly into `q`. Without the nested accumulator, an empty `or` list combined with a non-empty `and` list would still work, but a non-empty `or` combined with a non-empty `and` would compose as `(and_q | or_branches)` instead of `(and_q) & (or_branches)`. The nested accumulator preserves the cookbook's `q & and_q`, `q & or_aggregate` semantics — every top-level branch ANDs into the running `q`, and the OR fan-out lives inside the `or_q` sub-accumulator.
- **`_q_for_branch` returns `Q(pk__in=child_set.qs.values("pk"))`** rather than calling `child_set.qs` directly into a Q-and-merge. PK-intersection is the spec L143 composition primitive; it keeps the override's contract identical regardless of how many leaf filters the child branch carries.
- **Sibling instantiation passes `request=None`** because permission checks and form validation already ran against the top-level `apply_*` filterset instance. Per the plan's step-3 rationale: the override is reached only after `_validate_form_or_raise` and `_run_permission_checks` succeeded at the top level, and the tree-form substrate operates only on data the top-level form already accepted.
- **`self.data or {}` defensive default in `filter_queryset`** — `BaseFilterSet`'s `__init__` accepts `data=None` (unbound filterset) and sets `self.data` to an empty `QueryDict`; the `or {}` is belt-and-braces for any future code path that bypasses `__init__`.
