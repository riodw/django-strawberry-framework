## Plan (Worker 1)

Slice 6 is the **products connections-only conversion** — the cookbook mirror and the card's strongest live proof. It is a **two-file change, both outside `django_strawberry_framework/`** (build plan line 21; Decision 11 line 405 lists the package modules — none change in Slice 6):

1. `examples/fakeshop/apps/products/schema.py` — the four `@strawberry.field` list resolvers (`all_categories` / `all_items` / `all_properties` / `all_entries`) become four `DjangoConnectionField` class attributes; their hand-written `filter:` / `order_by:` parameter declarations and the now-unused `filter_input_type` / `order_input_type` imports disappear; the `Query` docstring/TODO block is updated to describe the shipped shape.
2. `examples/fakeshop/test_query/test_products_api.py` — the whole live suite re-pins from list-shape `{ … }` to connection-shape `{ edges { node { … } } }`; the three `test_products_optimizer_*` SQL-shape tests re-pin through the connection wrapper with query counts intact; the `check_name_permission` filter/order denial gates re-pin against the synthesized arguments; one NEW reverse-FK nested-connection fixed-query-count pin is added.

**Scope discipline (Decision 10, lines 395 / 82-84):** ONLY the four list resolvers → four `DjangoConnectionField` attributes. NO `Meta.connection` opt-in (products ship without `totalCount`, like the cookbook); NO root `DjangoNodeField` / `DjangoNodesField` (those stay `TODO-BETA-051-0.1.5`); NO model or sidecar changes. The four types' relation-connection siblings (`itemsConnection`, `entriesConnection`, `propertiesConnection`) already exist live via the `DONE-032-0.0.9` implicit `"both"` default and now plan through Slices 1-2 (final-accepted).

**Carry-forward from Slice 5's final verification (load-bearing — the cap/order + fallback semantics):**
- A list resolver returns *every* row; a `relay.connection` with no `first`/`last` returns at most `relay_max_results` (default 100) plus `pageInfo`, and `connection.py::_finalize_queryset` appends the deterministic `ORDER BY pk` via `optimizer/plans.py::deterministic_order` (the cursor-parity invariant). The re-pin is therefore **NON-mechanical in one respect**: any assertion that expected the full unbounded set (or a count past the default page) must stay under the cap or pass an explicit `first:` (Decision 10 line 396; Test plan line 537). The failure mode is a silently-truncated-but-still-parsing result — call it out, even though current fixtures stay well under 100.
- The appended `ORDER BY pk` is a **no-op on the SQL `ORDER BY` shape** for the products roots as currently spelled, because the prior resolvers passed `Model.objects.order_by("id")` and `id` IS each model's auto pk (no `Meta.ordering` on `Category` / `Item` / `Property` / `Entry`). `deterministic_order` appends pk only when the effective ordering does not already end in a unique column; `id`-terminal already qualifies, so it appends nothing. Net: the connection's pre-slice queryset orders by pk just as the old resolver did — **wire order and query-count shape are preserved**. (The connection still *re-derives* the order from scratch rather than inheriting the resolver's `.order_by("id")`; the plan must not assume the resolver's explicit `.order_by` survives — it is replaced by the synthesized pipeline's own deterministic-order step.)
- **A sidecar argument on a nested connection silently diverts it to the per-parent FALLBACK path, not the window.** This is the single sharpest Slice-5 lesson (final-verification: a visibility/window test that passes `filter:`/`orderBy:` on a *nested* connection pins the wrong path). The NEW reverse-FK nested-connection fixed-count pin must therefore select `itemsConnection(first: N)` with **only `first:`** (no `filter:` / `orderBy:`), or it measures the fallback's per-parent query count instead of the windowed fixed count.

## DRY analysis

- **Existing patterns reused (cite file:line).**
  - **The `DjangoConnectionField` declaration shape is already live in the example project** — `examples/fakeshop/apps/library/schema.py:334`: `all_library_genres_connection: DjangoConnection[GenreType] = DjangoConnectionField(GenreType)`, with the imports at `examples/fakeshop/apps/library/schema.py:11-12` (`DjangoConnection`, `DjangoConnectionField` from `django_strawberry_framework`). Slice 6 copies that exact shape four times for the products types. The spec's User-facing API sample (lines 199-207) is byte-identical to this shape. This is the canonical cite — products does not invent a new declaration form.
  - **The synthesized `filter:` / `orderBy:` arguments replace the hand-written ones from the SAME sidecars.** `DjangoConnectionField` derives its `filter:` / `orderBy:` arguments from the wrapped type's `Meta.filterset_class` / `Meta.orderset_class` via the synthesized resolver signature (`connection.py::_annotations_for` #"annotations[\"order_by\"]" lines ~805-858; GLOSSARY `DjangoConnectionField` entry). The products types already declare `filterset_class = filters.{Category,Item,Property,Entry}Filter` and `orderset_class = orders.{…}Order` (`products/schema.py:57-58, 91-92, 125-126, 159-160`). So removing the hand-written `filter: filter_input_type(filters.CategoryFilter) | None = None` / `order_by: list[order_input_type(orders.CategoryOrder)] | None = None` parameter declarations (`products/schema.py:229-230, 243-244, 257-258, 271-272`) loses NO capability — the connection regenerates the identical argument surface from the identical sidecars, with the identical `visibility → filter → orderBy → deterministic-order → optimizer` composition the resolver bodies spelled by hand (`products/schema.py:232-237` etc.). The `filter_input_type` / `order_input_type` imports (`products/schema.py:37-38`) become dead and are removed in the same change (sweep for orphan imports, AGENTS.md).
  - **The shared schema-reload fixture is inherited free** — `test_products_api.py:73-77` `_reload_project_schema_for_acceptance_tests` (autouse) + `_reload_products_project_schema` (`:42-70`). Reloads `apps.products.schema` → `config.schema` → `config.urls`. The conversion changes WHAT the reloaded `products.schema` module defines (connection fields instead of resolvers) but NOT the fixture mechanism; every existing and new test rides it unchanged. The `_post_graphql` / `_assert_graphql_data` / `_staff_client` / `_global_id` helpers (`:79-106`) are reused verbatim.
  - **`CaptureQueriesContext(connection)` + `len(captured)` is THE query-count idiom in this file** (`test_products_api.py:35` import; `:191`, `:216`, `:265` existing uses), matching the library suite (`test_library_api.py` `:262`/`:298`/`:1060`). The re-pinned `test_products_optimizer_*` tests and the new nested-connection pin reuse it verbatim — do NOT introduce `django_assert_num_queries` (mixing idioms is a DRY/consistency smell; same rule Slice 5 held).
  - **The `edges { node }` rewrite pattern is uniform** — every list-shape selection `{ name }` becomes `{ edges { node { name } } }`, and every list-shape expected literal `{"allItems": [ {…}, … ]}` becomes `{"allItems": {"edges": [ {"node": {…}}, … ]}}`. One mechanical transform applied across the suite; the only place it is NOT mechanical is default cardinality (cap) and the query-count accounting (below).
- **New helpers justified.** **None required.** The new reverse-FK nested-connection pin is one `@pytest.mark.django_db` function built from the established helpers (`seed_data`, `_post_graphql`, `CaptureQueriesContext`). No module-level helper crosses tests. (Condition that would justify extraction later: a second nested-connection products pin needing the same two-level query builder — not the case in this slice.) A per-test local closure/constant for a repeated query string is Worker 2's discretion (see Implementation discretion items).
- **Duplication risk avoided.** The naive risk is **re-spelling the `edges { node }` wrapper inconsistently** across ~18 re-pinned assertions (some with `node { id name }`, some `node { name }`), or hand-recomputing each expected literal. The plan prevents drift by: (a) one uniform rewrite rule (list selection → `edges { node { … } }`; expected `[…]` → `{"edges": [{"node": …}]}`); (b) keeping every `expected` derived from the SAME ORM query the pre-conversion test used (the file's data-driven convention, docstring lines 18-22) — the ORM `expected` list is simply wrapped in the `edges`/`node` envelope, never re-typed by hand. The denial-gate tests assert on `payload["errors"]` (unchanged shape) so they need only the query-string rewrite, not an expected-data rewrite.
