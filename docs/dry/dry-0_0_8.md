- [ ] **Existing patterns reused.**

The walker calls `cls._normalize_input(branch)` to convert each nested input into a form-data dict and re-uses the existing `cls(data=..., queryset=queryset)` instantiation.

The visibility-scoped and related-constraint-baked queryset is what `filterset_instance = cls(data=data, queryset=constrained, ...)` carries through; `self.queryset` in the override is therefore already the parent's visibility-scoped + RelatedFilter-constrained queryset. - [ ] **New helpers justified.** One new private classmethod, `_evaluate_logic_tree(cls, queryset, tree_data) -> Q`, on `FilterSet` at `sets.py`. Justification for extracting: the walker is recursive (an `and` branch may itself carry `or` / `not` sub-branches per the Strawberry self-reference depth), and keeping recursion logic in a named helper makes the three new tests (intersect / union / negate) self-targeting at one symbol. The helper's input is the already-normalized form-data dict on `self.data` for the top call and the result of `cls._normalize_input(child)` for each nested branch.
- [ ] **Duplication risk avoided.**
  - **`_LOGIC_KEYS` literal duplication is OUT OF SCOPE for Slice 4a.** The tuple already lives twice — `sets.py:53-57` and `inputs.py:108-112` — and the override consumes the `"and"` / `"or"` / `"not"` string keys directly off `self.data`. Adding a third inlined copy would be the easy mistake; the override reuses the existing `sets.py` `_LOGIC_KEYS` constant by re-deriving the wire keys from `dict(_LOGIC_KEYS).values()` (matching how `_normalize_input` line 438 already builds its `logic_lookup`). The cross-module consolidation between `sets._LOGIC_KEYS` and `inputs._LOGIC_KEYS` is a Worker-1 integration-pass carry-forward documented in `worker-1.md` 2026-05-28 Slice-4 final-verification entry — do not fold it in Slice 4a.
  - **Do not re-implement leaf-clause filtering.** `BaseFilterSet.filter_queryset(self, queryset)` already handles every `form.cleaned_data` leaf clause via `self.filters[name].filter(queryset, value)`. The override calls `super().filter_queryset(queryset)` first and only composes tree-form Q's on top. Passing the bare `cls._meta.model._default_manager.all()` to a recursive `cls(data=..., queryset=...)` would re-introduce the visibility-before-filter ordering bug the apply pipeline already prevents. Recurse against the queryset the override receives (the parent's post-`super().filter_queryset` queryset).

The Slice-4a override leans on PK-intersection (`pk__in=<sibling_qs>`) for tree composition instead, which is one Q expression per branch and avoids reimplementing the cookbook's parallel form/proxy machinery. Tradeoff: PK-intersection issues one subquery per branch; acceptable for the substrate-shipping goal and matches the spec L143 contract (`Q(...) & Q(...)` / `Q(...) | Q(...)` / `~Q(...)`).
