## 1. [HIGH — missing edge case / correctness] The ported `apply_window_pagination` drops upstream's DISTINCT guard

The one I'd block Slice 1 on. The spec ports `apply_window_pagination` "at the mechanism level" and is explicit about what it deliberately does *not* borrow (the `_clone` monkeypatch, the queryset-config flag, the `_strawberry_*` names). But upstream's `pagination.py` carries a guard the spec never mentions: when the windowed queryset has `DISTINCT` enabled, the `_strawberry_total_count` annotation is **wrong**, because SQL evaluates window functions *before* `DISTINCT` dedup — so `Count(1) OVER (PARTITION BY …)` counts pre-dedup rows. Upstream handles it (`get_total_count`: `if queryset.query.distinct: … remove_window_pagination(); return queryset.count()`); spec-033 has zero mentions of `distinct`.

Why it bites this package specifically: the windowed child queryset bakes the target's `get_queryset` (Decision 4), and a visibility/permission `get_queryset` that joins and de-duplicates with `.distinct()` is a normal shape — `.distinct()` already appears across the fakeshop apps. The moment a windowed target's `get_queryset` (or its default manager, or a model `Meta`) carries `.distinct()`, nested `totalCount` silently over-counts and `hasNextPage` derived from `_dst_total_count` goes wrong — a *silent wrong-data* outcome, not a fallback, which violates the spec's load-bearing "every fast-path result is byte-identical to the pipeline" promise.

**Proposed spec change:** add a Decision 4 bullet (and a Decision 5 fast-path bullet) porting the guard — when the windowed child queryset's `query.distinct` is set, either (a) skip window planning for that relation and fall back per-parent, or (b) serve `totalCount` from a `remove_window`-style `.count()` rather than the annotation. Add `test_window_total_count_correct_under_distinct_target_get_queryset` to the Slice 1 plan and a live pin to Slice 5 if any library/products target distincts. Note the limitation in the GLOSSARY entry alongside the existing window-backend caveat.

## 2. [MED-HIGH — under-specification / contradiction] The synthesized resolver can't build its own `to_attr` probe

Decision 5 has the resolver probe `getattr(root, "_dst_<field>_connection", None)`. But the resolver is constructed by `_synthesize_relation_connections` as `_build_relation_connection_resolver(target_type, instance_accessor(field))` — it receives only the **accessor**, never the relation field name. The `to_attr` (and the generated `<field>_connection` attribute, and the walker's `relation_connections` value) are all keyed on `field.name`, which **diverges from the accessor** for a reverse relation without `related_name` (`field.name == "book"`, accessor `"book_set"`, generated `book_connection`, `to_attr` `_dst_book_connection`). With only the accessor in hand, the resolver computes the wrong probe attribute and the fast path silently never fires for exactly the reverse-no-`related_name` shape the package already special-cases everywhere else (Round-4 S3).

Decision 11's file list and the Slice 2 checklist mention "`connection.py` gains the fast path" but not the **signature change** to `_build_relation_connection_resolver` nor the **finalizer call-site change** that must pass `field.name` (or the precomputed `to_attr`) alongside the accessor.

**Proposed spec change:** add to Slice 2 (Decision 5) an explicit item — `_build_relation_connection_resolver` gains a third parameter (the relation field name, or the precomputed `_dst_<field>_connection` string), and `_synthesize_relation_connections` passes it. Add a reverse-FK-without-`related_name` parity test so the divergence is pinned, not assumed. (Anchored in the TODOs left in `finalizer.py` and `connection.py`.)

## 3. [MED — inconsistency with `registry.py`] Nested recognition reads the PRIMARY type's `relation_connections`, but the slot is per-declaring-type

Decision 3 records `relation_connections` on "the declaring type's `DjangoTypeDefinition`." But the walker resolves the definition at **nested** levels through `_resolve_field_map(model, source_type=None)`, which routes to `registry.get(model)` — the **primary** type (the documented spec-018 nested contract). So when the walker descends into a relation and looks for nested connection siblings, it reads the *primary's* `relation_connections`, regardless of which `DjangoType` actually shapes that node in the response. If a model has a secondary `DjangoType` whose `Meta.relation_shapes` differs from the primary's — secondary synthesizes `booksConnection` while the primary narrowed it to `"list"`, or vice versa — nested recognition uses the wrong slot: it either misses a real connection (no window planned, silent per-parent N+1 that strictness then flags) or tries to recognize one this type didn't synthesize.

Connection recognition inherits the same primary-resolution rule as `field_map`, which is *correct by construction* for field maps (a model's relations are the model's relations) but *not* obviously correct for `relation_shapes`, which is a per-type opt-in. The spec's Decision 3 normalization-sharing note is the right instinct but addresses name conversion, not type identity.

**Proposed spec change:** add an Edge-case / Decision-3 sentence pinning the behavior — either "nested connection recognition follows the primary-type contract; secondary types with divergent `relation_shapes` are out of scope for windowed planning in `0.0.9`" (cheap, honest, matches the shipped nested contract) or, if that's unacceptable, route nested recognition through the resolved target definition rather than the primary. A `test_secondary_type_relation_shapes_nested_recognition` pin either way.

## 4. [MED — performance expectation / test gap] The flagship cookbook example produces an uncacheable plan, so Slice 3 never helps it

`BookType.get_queryset` filters `circulation_status="repair"` for non-staff. Baking it into the windowed child queryset flips `plan.cacheable = False` (the shipped request-scope rule, which Decision 4 correctly inherits). That means the headline two-level shape the card exists to optimize — `allLibraryGenresConnection { … booksConnection { … } }`, where `Book` has visibility — produces an **uncacheable plan**, re-walked every request. Slice 3's whole point (nested pagination variables must key the cache) only matters for plans that *are* cacheable, i.e. nested connections whose target has **no** custom `get_queryset`. Not a contradiction, but the spec frames Slice 3 as protecting "the hot path this card exists to optimize," and for the visibility-bearing flagship example B1 is already disabled for an orthogonal reason.

**Proposed spec change:** one line in Decision 7 (and the "Visibility-filtered targets" edge case) making the interaction explicit — "windowed plans cache only when no windowed target overrides `get_queryset`; the visibility-bearing library example is uncacheable for the shipped request-scope reason, so Slice 3's correctness rule is exercised by non-visibility synthetic targets, not the cookbook shape." It also means the Slice 5 fixed-query-count pin and the Slice 3 cache pins must use *different* fixtures; the spec should say so rather than letting someone assume the library example covers both.

## 5. [MED — test setup gap] products list→connection silently changes default cardinality

The hard part of the products re-pin is not shape-rewriting (`{ name }` → `{ edges { node { name } } }` is mechanical) — it's **cardinality**. A list resolver returns *every* row; a `relay.connection` field with no `first`/`last` returns at most `relay_max_results` (default 100) plus `hasNextPage`. Slice 6 calls the re-pin "mechanical shape rewrites, not semantic changes," but any `test_products_api` assertion that expects the *full* unbounded result set (or a count > the default page) becomes wrong after conversion, and the `test_products_optimizer_*` query-count pins now run through a connection whose pre-slice queryset also gets the deterministic-order `ORDER BY pk` appended by `_finalize_queryset`. The current products fixtures look small (no bulk creation in `test_products_api.py`), so this likely doesn't bite *today* — but the spec should not call the rewrite purely mechanical, because the failure mode is a silently truncated result that still parses.

**Proposed spec change:** a Slice 6 note — "list→connection caps default cardinality at `relay_max_results`; re-pinned assertions must either stay under the cap or pass an explicit `first:`. The `test_products_optimizer_*` query-count pins must account for the appended `ORDER BY pk` and the connection's slice query, and must confirm the optimizer (installed in `config/schema.py` with strictness `off`) plans the root connection's `edges { node }` so the counts match the pre-conversion list shape." Reassuringly: Slice 4's strictness tests genuinely cannot be live (fakeshop runs strictness `off`, confirmed in `config/schema.py`), so they require synthetic package types — the spec already plans exactly that, so *no* test-harness rewrite is needed for strictness; the only real harness pressure is the products re-pin, which the spec already sizes at `+260/-240`.

## 6. [LOW — configuration & performance] The only config touch is a safe read; no lazy-validation overhead

There is no new setting, no validation step, and no per-request config mutation in this card. The only configuration touch is reading `relay_max_results` off `info.schema.config` inside the walker via `SliceMetadata.from_arguments` (Decision 4 / the "`relay_max_results` agreement" edge case). That value is fixed at schema construction and read-only at request time, so there is no lazy-validation overhead, no redundant per-query validation, and no thread-safety hazard — the schema config is immutable shared state, read not written. The genuine per-request cost in this card is the AST walk in `_build_cache_key`, and the spec already addresses it (Decision 7: memoize the directive- and pagination-variable name sets per `id(operation)` from one reachable-fragments traversal). The one thing worth holding the line on: Slice 3 must not regress the memoization into a second per-parent full-AST walk — already a Slice 3 DoD item.

## 7. [LOW — re-affirming] Cursor portability is a correctness prerequisite, not polish

The entire fast-path↔fallback wire-parity promise (and `test_fast_path_cursor_round_trips_to_fallback_after`) depends on the plan-time window's `ORDER BY` being **byte-identical** to the resolve-time pipeline's order. The spec files this under Decision 11 ("module locations") as hoisting `_ends_in_unique_column` to `plans.py`. That hoist is not a tidiness refactor — it is the *only* thing guaranteeing the window's row numbers and the pipeline's offset cursors agree. If the two ever diverge (someone "improves" one order rule and not the other), cursors stop round-tripping between cached and uncached executions and the failure is data-dependent and hard to spot. Recommend promoting the hoist's rationale from Decision 11 into Decision 4/5 as a stated invariant ("the window order MUST be produced by the same helper the pipeline uses") with the parity test named as the guard.

## Net assessment

The spec is in good shape — earlier review rounds did real work and the spec absorbed them. Of the seven above, only **finding 1 (DISTINCT)** is a true correctness hole that can ship silently-wrong data, and **finding 2 (resolver `to_attr` identity)** is a concrete under-specification that will produce a non-obvious "fast path never fires" bug on a common relation shape; both should be written into the spec before Slice 1. Findings 3–5 are honest scope/expectation pins that prevent a reviewer or a future maintainer from being surprised; 6–7 are clarifications. None argue against the architecture — they tighten the edges the borrowed mechanism is known to have sharp corners on.
