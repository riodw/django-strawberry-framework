# Review: `django_strawberry_framework/management/commands/inspect_django_type.py`

Status: verified

## DRY analysis

- **Defer-with-trigger: collapse the two GraphQL-rendering twins (`_render_annotation` vs `_render_strawberry_type`) once a third renderer lands.** `inspect_django_type.py::_render_annotation` (lines 392-408) and `inspect_django_type.py::_render_strawberry_type` (lines 314-327) are deliberate siblings: one walks Python `typing` annotations (`typing.get_origin`/`get_args`), the other walks Strawberry wrapper objects (`StrawberryOptional`/`StrawberryList`). They share the same three output shapes (`Name!`, `Name`, `[Inner!]!`) but consume structurally different inputs, so a single function would need an input-kind branch on every recursion. This is the same intentional-twin calibration recorded for sync/async resolvers across the package â€” do NOT extract now. Defer until a third GraphQL-rendering surface needs the same `Name!`/`Name`/`[Inner!]!` shaping; then lift the shaping rule into a shared `_graphql_shape(is_optional, is_list, leaf_name)` and let each walker feed it.
- **Defer-with-trigger: the relay-shaped predicate is duplicated against `types/base.py::_is_relay_shaped`.** `inspect_django_type.py::_is_suppressed_relay_pk` (lines 209-212) re-spells the exact body of `django_strawberry_framework/types/base.py::_is_relay_shaped` (base.py:454): `any(issubclass(i, relay.Node) for i in definition.interfaces) or issubclass(definition.origin, relay.Node)`. This is the load-bearing parity that keeps the command's pk-suppression detection aligned with the synthesizer's. `_is_relay_shaped` is module-private to `types/base.py`. Defer until a second consumer outside `types/base.py` needs the predicate; then promote `_is_relay_shaped` to a shared `types`-package helper (e.g. `types/relay.py` or `utils/relations.py`) and have both the synthesizer and this command import it, so the parity is enforced by a single source rather than by review vigilance. (Flagged for the folder/project pass as a cross-file placement question, not a local defect.)

## High:

### `_relation_row` KeyErrors on a `"connection"`-shaped many-side relation

`_relation_row` reads `definition.origin.__annotations__[field.name]` (`inspect_django_type.py::_relation_row`, line 224) for every auto-synthesized relation field. That key is **absent** whenever the relation was reshaped to `relation_shapes = {"<rel>": "connection"}`: the Phase-2.5 synthesizer calls `_suppress_relation_list_form(type_cls, name)` which does `type_cls.__annotations__.pop(name, None)` (`types/finalizer.py::_suppress_relation_list_form`, line 329; invoked from both the first-attach path at finalizer.py:517-521 and the partial-finalize rerun at finalizer.py:457-458). The Django relation field stays in `definition.selected_fields` (the synthesizer iterates `selected_fields` at finalizer.py:413 and never removes the entry), and the synthesized sibling lives under a DIFFERENT name (`<field>_connection`, finalizer.py:447), so when `_print_table` walks `selected_fields` (line 167) the underlying relation field still reaches `_resolve_row`:

- it is not a suppressed relay pk (the pk branch, line 193, only fires for `model._meta.pk.name`);
- it is not consumer-authored â€” the synthesizer explicitly SKIPS consumer-authored relations (finalizer.py:419) and only reshapes auto-synthesized ones, so a `"connection"`-shaped relation is never in `consumer_authored_fields`;
- `field_meta.is_relation` is True â†’ `_relation_row` (line 198) â†’ `origin.__annotations__[field.name]` â†’ **`KeyError`** (an unhandled exception / traceback, not a `CommandError`).

This crashes the diagnostic on a legitimately finalized type whose schema builds correctly. It is reachable by any consumer declaring a connection-only relation shape (the whole point of `inspect_django_type` is to inspect such types). The `"both"` default keeps the list annotation, so the bug is specific to the `"connection"` shape; that is also why the existing tests miss it â€” no test and no example DjangoType currently declares `"connection"` (verified: `grep relation_shapes` over `examples/fakeshop/apps/*/schema.py` is empty; `tests/management/test_inspect_django_type.py` has no connection-shape case; the relay-shaped `GenreType` test exercises only the `"both"` default).

Why it matters: "errors that crash normal consumer usage" is the High rubric. A connection-only relation is a supported, shipped (spec-032/033) configuration, and the inspection command is exactly the tool a consumer reaches for when their connection shape behaves unexpectedly.

Recommended change (root-cause, not a `try/except KeyError` patch): `_resolve_row` should detect the connection-only-reshaped relation and report it honestly rather than indexing an annotation that the synthesizer deliberately removed. The synthesizer already records the mapping the command needs â€” `definition.relation_connections` maps `"<rel>_connection" -> "<rel>"` (set by `types/finalizer.py::_record_relation_connection`, lines 334-349). The command can:

1. detect `field.name in (definition.relation_connections or {}).values()` AND `field.name not in origin.__annotations__` â†’ the list form was suppressed; render the row from `field_meta` alone (cardinality + nullability are still available from `FieldMeta`) with a converter label naming the connection shape (e.g. `relation: <kind> (connection-only)`), OR
2. iterate the synthesized connection annotation/Strawberry field instead.

Either way the row should describe the connection that actually shipped for that field, never `KeyError`. A guard that simply `.get(field.name)` and falls back to a generic string would silence the crash but print a misleading row â€” prefer reading the real connection metadata.

```django_strawberry_framework/management/commands/inspect_django_type.py:224
graphql_type = _render_annotation(definition.origin.__annotations__[field.name])
```

Test to add (package tree, `tests/management/test_inspect_django_type.py`): build a Relay-Node-shaped type with a many-side relation to another Node type and `Meta.relation_shapes = {"<rel>": "connection"}`, finalize, then `call_command("inspect_django_type", ...)` and assert the row renders (no `KeyError`) and the converter column names the connection shape. A live `examples/fakeshop` variant (a connection-only relation on an existing relay-shaped example type, inspected via `call_command` in `examples/fakeshop/tests/test_inspect_django_type.py`) is the preferred earn-it-through-real-usage placement per `AGENTS.md`; fall back to the package test only if no example type can carry the shape without disturbing other coverage.

## Medium:

### GLOSSARY drift: type/nullability are NOT always read from `origin.__annotations__`

`docs/GLOSSARY.md` `#schema-introspection-management-command` (GLOSSARY.md:1181) states: *"the resolved GraphQL type and nullability are read from `origin.__annotations__` (the authoritative post-finalize record that already reflects `Meta.nullable_overrides` / `Meta.required_overrides` and consumer-authored annotations)"*. This is a public-contract description and is now stale: the source reads `origin.__annotations__` ONLY for the auto-synthesized scalar (`_scalar_row`, line 233) and auto-synthesized relation (`_relation_row`, line 224) branches. For **consumer-authored fields** the command deliberately does NOT read `origin.__annotations__` â€” it reads the finalized Strawberry field metadata `origin.__strawberry_definition__.fields[â€¦].type` (`_consumer_authored_row`, lines 264-265), precisely because `origin.__annotations__` holds a `StrawberryAnnotation` (assigned) or an unresolved forward-ref string (annotated relation) for those names, "neither renderable here" (module docstring lines 16-21, and the method docstring lines 245-263). The GLOSSARY's parenthetical "and consumer-authored annotations" is exactly the case where `origin.__annotations__` is the WRONG source. This is documented-public-contract drift â†’ Medium.

Verbatim replacement for the relevant sentence in GLOSSARY.md:1181 (replace the clause starting "the resolved GraphQL type and nullability are read from"):

> the resolved GraphQL type and nullability are read from the authoritative post-finalize record â€” `origin.__annotations__` for auto-synthesized scalar and relation fields (already reflecting `Meta.nullable_overrides` / `Meta.required_overrides`), and the finalized Strawberry field metadata `origin.__strawberry_definition__` for consumer-authored fields (annotation or `strawberry.field` overrides), whose `origin.__annotations__` entry is an unrenderable `StrawberryAnnotation` or unresolved forward-ref string; an unresolved forward reference surfaces as Strawberry's `UNRESOLVED` sentinel and raises `CommandError` rather than printing a bogus type â€” while [`SCALAR_MAP`](#scalar-field-conversion) is re-walked only to NAME the converter row, never to re-derive nullability via `convert_scalar`.

`_resolve_bare_name` (lines 143-152) builds the ambiguity candidate list with `getattr(registry.model_for_type(type_cls), '__name__', '?')`. `registry.model_for_type` returns `None` only for an unregistered `type_cls` (registry.py:242-254), but every `type_cls` here came from `registry.iter_types()` so it is registered by construction â€” the `'?'` fallback is therefore unreachable for a finalized registry. This is a harmless defensive default (same calibration as worker-1 memory's `conf.py` unreachable-guard-set note: unreachable-but-harmless = Low, not dead-code Medium, no behavior at stake). No change needed; recording for audit completeness.

## What looks solid

### DRY recap

### Other positives

- **Resolution dispatch order is correct and well-justified.** `_resolve_row` (lines 193-199) checks most-specific-first: suppressed relay pk â†’ consumer-authored â†’ auto relation â†’ auto scalar. The pk-first ordering is load-bearing â€” a relation pk on a Relay type (`OneToOneField(primary_key=True)`) would otherwise reach `_relation_row` and `KeyError`; the suppressed-pk branch correctly intercepts it (docstring lines 180-184 names this exact trap, verified against `types/base.py:1603-1609` where the pk `continue`s past synthesis).

Tested at `tests/management/test_inspect_django_type.py::test_inspect_unresolved_forward_ref_relation_raises_command_error`.
- **Error/exit-code handling is consistent.** Every operator-facing failure raises `CommandError` (Django converts to a clean stderr message + non-zero exit, no traceback): bad dotted path (line 129, preserves the ORIGINAL error via `from e`, never masked by a registry fallback), bad `--schema` (lines 105-106), unregistered/ambiguous bare name (lines 139-152), non-`DjangoType` symbol (line 110), missing definition (lines 114-117), unfinalized definition (line 119). The unregistered-vs-unfinalized split is two distinct branches as the GLOSSARY documents (GLOSSARY.md:1185). All exercised by `tests/management/test_inspect_django_type.py`.
- **Reflective access is safe.** Every `getattr`/`issubclass`/`isinstance` is guarded or operates on a verified shape: the `issubclass(target, DjangoType)` guard (line 109) gates all subsequent `target`-attribute reads; `getattr(target, "__django_strawberry_definition__", None)` (line 112) defaults to `None` and is explicitly checked (line 113); `_scalar_name`'s `getattr(scalar, "__name__", None) or str(scalar)` (line 389) cannot raise. `_meta.pk.name` (line 215) is reached only after `_is_relay_shaped` returns True, and pk always exists on a concrete model.
- **`--schema` cold-process register+finalize is correct.** `import_module_symbol(schema, default_symbol_name="schema")` (line 104) imports the schema module symbol, which transitively triggers `DjangoType.__init_subclass__` registration and `finalize_django_types()` (the `strawberry.Schema(...)` construction in the consumer's schema module) â€” so a cold CLI process has a fully finalized registry before `_resolve_type` runs. The `definition.finalized is False` guard (line 118) is the backstop when `--schema` was omitted on a cold process. `import_module_symbol` failures are narrowed to `(ImportError, AttributeError)` â†’ `CommandError`.
- **Nullability rows match the resolution contract.** Auto-scalar reads optionality from the resolved union annotation (`_annotation_is_optional`, lines 286-290, handles both `typing.Union` and `types.UnionType` PEP 604 spellings); auto-relation gates on `is_many_side` â†’ `"no (list)"` else `FieldMeta.nullable` (line 227), matching `FieldMeta.nullable`'s own cardinality gate (field_meta.py:60-74); consumer rows read optionality from the Strawberry wrapper (`_consumer_nullable`, lines 330-341). All three paths agree with how the type was actually synthesized.
