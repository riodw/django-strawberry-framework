### Anonymous inline fragment under an optimized field crashes the resolver (`'NoneType' object has no attribute 'name'`)

A valid GraphQL query placing an **anonymous inline fragment** (`... { ... }` - an inline fragment with NO `on TypeName` type condition) under any optimized `DjangoType` field (a `DjangoConnectionField` OR a plain `DjangoListField`/list resolver) crashes the field resolver with `AttributeError: 'NoneType' object has no attribute 'name'`, surfaced to the consumer as a `GraphQLError` on the field. It reproduces with NO directive and NO `totalCount` in the fragment - the bare anonymous-inline-fragment shape is sufficient. This is a crash on a spec-valid query against the package's headline optimizer surface -> **High** ("errors that crash normal consumer usage", REVIEW.md severity definitions).

**Root cause (where the unguarded `.name` actually lives).** The crash is NOT in `walker.py`'s `sel.name` accesses (those are reached only after fragments are inlined, see below). It is upstream of the walker, on the optimizer's selection-conversion entry: `optimizer/extension.py::DjangoOptimizerExtension.apply_to #"convert_selections(info, info.field_nodes)"` (src line 783) calls Strawberry's `strawberry.types.nodes.convert_selections`, which dispatches an anonymous `InlineFragmentNode` to `InlineFragment.from_node` and reads `type_condition=node.type_condition.name.value` (`.venv/.../strawberry/types/nodes.py:126`). For an anonymous inline fragment `node.type_condition is None`, so `.name` raises. The package never gets a chance to see the converted selection - the converter crashes inside the package's own call. `apply_to` is the shared entry for BOTH the connection path (`connection.py::_finalize_queryset` -> `apply_connection_optimization` -> `apply_to`, the path the original review's `_plan_connection_relation` focus traverses) AND the middleware list path (`extension.py::DjangoOptimizerExtension.resolve` -> `_get_or_build_plan` -> `apply_to`), so the bug is NOT connection-specific.

**Trigger shape (anonymous-only; typed inline fragments are SAFE).** Reproduced live via in-process `schema.execute_sync` against fakeshop (`docs/review/temp-tests/walker_recheck/test_inline_fragment_crash.py`):

- `{ allItems { edges { node { ... { name } } } } }` (anonymous inline fragment at node level, connection field) -> **CRASH** (`'NoneType' object has no attribute 'name'`, path `['allItems']`).
- `{ allItems { ... { edges { node { name } } } } }` (anonymous inline fragment directly under the connection field) -> **CRASH** (path `['allItems']`).
- `{ allGlossaryTerms { ... { title } } }` (anonymous inline fragment under a plain LIST field) -> **CRASH** (path `['allGlossaryTerms']`) - confirms the bug spans the middleware list path too, not just connections.
- `{ allItems { edges { node { ... on ItemType { name } } } } }` (TYPED inline fragment `... on ItemType`) -> **NO error** (resolves cleanly). The typed form carries a `type_condition` AST node, so `InlineFragment.from_node` reads `.name.value` successfully.
- `{ allItems { edges { node { id name } } } }` (baseline, no fragment) -> **NO error**.

So the crash is specific to **anonymous (type-condition-less) inline fragments**; typed inline fragments and named fragment spreads are unaffected. The full traceback terminates at `strawberry/types/nodes.py:126` via the `apply_to` -> `convert_selections` frame, confirming the package-owned call site is the trigger.

**Why `walker.py`'s `sel.name` reads do not fire (and stay latent).** `walker.py::_walk_selections #"django_name = snake_case(sel.name)"` (src line 247) and `walker.py::_selected_scalar_names #"django_name = snake_case(sel.name)"` (src line 696) both iterate `_merge_aliased_selections(_included_field_selections(selections))`. `selections.py::included_field_selections` (src lines 173-191) INLINES fragment bodies and never returns a fragment shell, and `_merge_aliased_selections` (`walker.py:889`) `continue`s on `_is_fragment(sel)` before reading `.name`. So even if a `name=None` anonymous-fragment shell reached the walker, those two reads would be downstream-guarded for the field-merge path - they are a LATENT hazard, not the live crash. The live crash is strictly the converter call at `extension.py:783`. (Upstream `strawberry-django` sidesteps this entirely by walking the raw graphql-core AST `field_nodes` via `graphql.collect_fields`/its `gql_compat` shim instead of `convert_selections` - see `~/projects/strawberry-django-main/strawberry_django/utils/gql_compat.py` + `optimizer.py:726`.)

**Recommended root-cause fix (no bare try/except).** The defect is that the package routes selection conversion through Strawberry's `convert_selections`, whose `InlineFragment.from_node` cannot represent the anonymous (`type_condition=None`) shape. `optimizer/selections.py` already owns the package's fragment-aware traversal substrate and already duck-types fragments on `hasattr(selection, "type_condition")` (`selections.py::is_fragment`, src line 142) - a `type_condition=None` shell would satisfy that predicate and flow correctly through `included_field_selections` / `named_children` / `with_runtime_prefix` (all of which read `.selections`, never `type_condition.name`). The fix is to stop depending on Strawberry's crashing converter for the inline-fragment shape and route the `info.field_nodes` -> converted-selection conversion through a fragment-aware adapter the package owns in `optimizer/selections.py`, so an anonymous inline-fragment AST node becomes a `SimpleNamespace(type_condition=None, selections=[...], directives=...)` shell (matching the `InlineFragment` duck shape the converted-selection adapter already consumes) rather than crashing. Concretely: add an `ast_to_converted_selections(info, field_nodes)` adapter to `optimizer/selections.py` (the existing single home for both AST and converted-selection traversal) that mirrors Strawberry's `convert_selections` but builds the inline-fragment shell with `type_condition = node.type_condition.name.value if node.type_condition is not None else None`, and have `extension.py::apply_to` call that adapter instead of `strawberry.types.nodes.convert_selections`. This keeps directive/fragment policy single-sourced in `selections.py` (the `docs/feedback.md` Major 2 consolidation contract) and removes the package's dependency on a Strawberry internal that mishandles a valid query shape. Do NOT wrap the `convert_selections` call in try/except - that would swallow the whole selection tree and silently disable optimization for any query containing an anonymous inline fragment (and AGENTS.md forbids try/except as a workaround for an abstraction-level bug). While implementing, also harden the latent `walker.py:247`/`walker.py:696` reads' assumption by confirming `included_field_selections` keeps inlining the now-non-crashing anonymous-fragment shells (it will - `is_fragment` already matches `type_condition=None`).

```django_strawberry_framework/optimizer/extension.py:781:783
        from strawberry.types.nodes import convert_selections

        selections = convert_selections(info, info.field_nodes)
        # ^ convert_selections -> InlineFragment.from_node reads
        #   `node.type_condition.name.value`; anonymous `... { f }` has
        #   type_condition=None -> AttributeError. Route through a
        #   fragment-aware optimizer/selections.py adapter instead.
```

**Test expectation (High requires it).** Pin the fix with a LIVE fakeshop `/graphql` query carrying an anonymous inline fragment under a connection field (preferred per AGENTS.md "Test through real usage and prefer the example project") - add to `examples/fakeshop/test_query/test_products_api.py`: a `django.test.Client` POST of `{ allItems { edges { node { ... { name } } } } }` asserting `response.json()` has NO `errors` key and returns the seeded item names. Add a sibling assertion for the typed form `... on ItemType { name }` (regression guard that the typed path stays working) and, since the bug also spans the list path, a list-field anonymous-inline-fragment query in `test_glossary_api.py` (e.g. `{ allGlossaryTerms { ... { title } } }`). If a live query cannot reach a specific converter branch, fall back to package-internal `tests/optimizer/test_walker.py` (or a new `tests/optimizer/test_selections.py` for the new adapter) exercising the adapter directly on a synthesized anonymous-`InlineFragmentNode`; the live query is reachable here, so it MUST be earned that way per AGENTS.md.

## Low:

### `_connection_window_slice` `try` block is wider than its documented intent

`_connection_window_slice` (`walker.py::_connection_window_slice #"except (ValueError, TypeError)"`, src lines 1118-1128) wraps the call to `derive_connection_window_bounds` and returns `None` on `ValueError`/`TypeError`. The intent (per the docstring) is to catch only Strawberry's pagination-validation errors (negative/over-max `first`/`last`, malformed cursor) so the field falls back per-parent and raises its own error at resolve time. But the `try` body also covers `_relay_max_results_from_info(info)` and the `arguments.get(...)` reads — a `TypeError` raised by an unexpected `info`/`arguments` shape (not a genuine pagination error) would be silently swallowed and mis-routed to the "malformed slice -> record resolver key, emit no window" path (src `#"append_unique_many(plan.planned_resolver_keys, resolver_identities)"`), masking a real bug behind a per-parent fallback. In practice only `derive_connection_window_bounds` raises here (`_coerce_pagination_int` catches its own int-cast failures; `_relay_max_results_from_info` is pure `getattr`), so this is a maintainability/locality concern, not a live correctness bug. **Defer with trigger:** narrow the `try` to wrap only the `derive_connection_window_bounds(...)` call (lift `first`/`last`/`max_results` resolution above the `try`) when a future argument-shape change adds a second raise site inside the block; today the blast radius is one call and the behavior is correct. Recorded-intent Low — no act-now source edit.

### `_can_elide_fk_id` composite-PK guard `# pragma: no cover` rationale will rot

`_can_elide_fk_id` (`walker.py::_can_elide_fk_id #"Composite primary key"`, src lines 726-730) returns `False` for composite PKs (`len(pk_fields) > 1`) under `# pragma: no cover`, justified by "Test fixtures do not define one." Per AGENTS.md, `pragma: no cover` is only for branches genuinely unreachable under the test runner; this branch IS reachable (a consumer adopting a Django 5.2+ composite PK hits it), it is merely un-exercised because the example project has no composite-PK model. The fail-closed `False` has a sound correctness rationale (eliding would compare a single-column source attname against a tuple target pk and surface wrong data), so leaving it uncovered is defensible at 0.0.9, but the "fixtures do not define one" rationale is the kind that rots once a composite-PK fixture lands elsewhere. **Defer with trigger:** when the example project (or `tests/`) gains any `CompositePrimaryKey` model, drop the pragma and pin the fail-closed `False` with a real test; until then the pragma + rationale is acceptable. Recorded-intent Low — no act-now source edit.
